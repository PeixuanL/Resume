## 2. JD 核心需求拆解

### 2.1 业务场景

岗位属于西门子 Smart Infrastructure 智能建筑方向，重点围绕楼宇、园区、能源系统和智能基础设施。产品场景集中在楼宇运维、设备设施管理、能源管理、照明、暖通等方向。

### 2.2 产品职责

- 负责楼宇运维相关产品的业务洞察、用户痛点挖掘、产品战略规划与 Roadmap。
- 主导软件 / SaaS 产品从需求调研、PRD、原型、研发测试到上线运营的全生命周期。
- 设计面向楼宇运维市场的软件或平台型产品，典型模块包括工单系统、设备资产管理、能耗监测。
- 推动数据驱动和智能算法在运维场景落地，例如预测性维护、能效优化、智能派单。
- 与研发、算法、设计、市场销售等团队协同，推动产品落地和商业化成功。

### 2.3 任职要求

- 本科及以上学历，5 年以上软件 / SaaS 产品经理经验。
- 有楼宇自动化、智慧园区、运维管理等行业背景，熟悉业务流程与核心痛点。
- 具备需求分析、文档撰写、数据分析能力。
- 具备商业化思维，理解软件产品商业模式与定价策略。
- 有 AI 产品经理经验，或主导过数据智能 / 算法类功能落地优先。
- 了解机器学习、数据分析基本原理，能与算法工程师顺畅沟通。
- 有 IoT 平台或 B 端企业服务产品经验优先。

## 4. 修改优先级

### P0 必须改

#### 4.1 求职意向

当前表达：

```text
B 端产品经理 / AI 产品经理
— 工业互联网 · 大模型应用平台 · Agent 产品化方向
```

建议改为：

```text
B 端产品经理 / AI 产品经理
— 智能楼宇运维 SaaS · IoT 数据平台 · AI 智能化应用方向
```

目的：把岗位最关注的“智能楼宇 / 运维 SaaS / IoT / AI”放到第一屏。

#### 4.2 个人总结首句

当前重心是“B 端 Agent 应用产品化”，对西门子来说略泛。建议改成：

```text
聚焦智能楼宇 / 园区 / 工厂等复杂设施运维场景，擅长将设备资产、告警、工单、SOP / 知识库与 IoT 数据抽象为可复用的 SaaS / AI 产品能力。
```

目的：把原有设施运维 Agent 经历映射到 JD 的楼宇运维、设备资产、工单、能耗 / 数据场景。

#### 4.3 核心能力

建议将当前三条核心能力调整为四条：

1. **智能楼宇 / 设施运维 SaaS 产品规划**：从楼宇、园区、工厂等设施运维流程中抽象设备资产、告警、工单、能耗与知识库能力，定义 MVP 范围、Roadmap、优先级与可复用模块。
2. **AI 与数据智能能力落地**：围绕告警诊断、预测性维护、智能派单与工单草稿生成，定义 Agent / RAG / Workflow 能力边界、低置信兜底与人工确认机制。
3. **IoT 数据接入与业务对象建模**：熟悉 IoT / 设备数据、API 接入、业务对象、状态枚举与字段映射，支撑设备、告警、工单、能耗等数据闭环。
4. **POC 验证与商业化复制**：定义 POC 范围、数据前置条件、成功标准与验收清单，推动试点结果沉淀为产品包、交付基线与规模化复制路径。

目的：同时命中产品规划、AI、IoT、商业化四个 JD 高权重维度。

#### 4.4 DataMesh 工作经历开头

当前表达：

```text
数字孪生 SaaS/私有化产品，服务制造、通信、物流等行业大客户；产品范围覆盖设施运维 Agent、工业仿真编辑、MR 培训等方向，主线聚焦设施运维 Agent 底座建设与大客 POC/交付复制。
```

建议改为：

```text
数字孪生 SaaS / 私有化产品，服务制造、通信、物流及园区 / 工厂类大客户；产品范围覆盖设施运维 Agent、设备数据接入、智能告警与工单流转、工业仿真编辑等方向，主线聚焦设施运维 Agent 底座建设与大客 POC / 交付复制。
```

目的：补上“园区 / 工厂 / 设备数据 / 告警 / 工单”，承接智能楼宇运维。

#### 4.5 设施运维项目标题

当前标题：

```text
企业级设施运维 Agent 产品（0→1）
```

建议改为：

```text
智能楼宇 / 园区设施运维 Agent SaaS 产品（0→1）
```

目的：直接命中智能建筑、园区、运维 SaaS。

### P1 建议改

#### 4.6 设施运维项目的问题描述

建议加入楼宇运维关键词，但不夸大为已有暖通 / 照明专项经验：

```text
面向大型园区 / 工厂 / 楼宇类客户，设备、空间、告警、工单、SOP / 运维知识与能耗数据分散；暖通、照明、设备资产等运维场景依赖人工经验判断故障原因与处置路径，响应效率与复盘沉淀不足，难以形成可复制的智能运维闭环。
```

#### 4.7 Agent / RAG 方案条

建议从“告警诊断”扩成 JD 更容易识别的“预测性维护 / 智能派单”相邻表述：

```text
围绕告警诊断、预测性维护、智能派单与工单草稿生成，定义 Agent 任务边界、Workflow 编排、RAG 知识源、可调用功能范围、输入输出约束与低置信兜底策略，形成场景脚本和验收标准。
```

#### 4.8 数据接入方案条

建议加入能耗、设备状态：

```text
协同研发梳理 IoT 设备数据、REST API、视频流与第三方系统接入边界，定义设备状态、告警、工单、能耗等数据前置条件、字段映射、状态枚举与检查项，保障运维上下文可被 Agent 调用。
```

#### 4.9 结果表达

建议把“实施周期压缩”与“商业化复制”绑定：

```text
MVP 覆盖海外标杆大客 80%+ 高优先级智能运维诉求，沉淀为可复制的设施运维 SaaS 产品包、POC 验收清单与交付基线；吸引 5 家以上潜在商机，头部通信运营商多期实施周期由 5-6 个月压缩至 1-2 个月，并支撑新加坡数百万级项目方案交付与落地。
```

### P2 可选改

#### 4.10 压缩工业仿真项目

该项目保留“数字孪生、数据分析、AI 辅助分析”即可，不建议占用太多篇幅。可以弱化传送带、机械臂等细节，把版面让给设施运维项目。

#### 4.11 教育背景补一句

建筑学 / 建筑设计背景对智能建筑岗位有帮助。可在 UCL 相关方向中微调为：

```text
相关方向：空间系统与交互设计、AI 平台策略、数字孪生 / Unity 原型开发
```

## 6. 不建议改或不要硬改

- 不要硬改工作年限。JD 要 5 年以上，但简历不能伪造；用项目深度和结果抵消。
- 不要把经历写成明确做过暖通 / 照明专项产品，除非有真实素材支撑。可以写“覆盖 / 可扩展到楼宇、园区、设备资产、能耗等运维场景”。
- 不要把工业仿真项目完全删掉。它能证明数字孪生、数据分析、工业软件理解，但应作为辅助项目。
- 不要过度技术化写机器学习算法。岗位要的是能和算法工程师沟通，产品侧表达即可。

