## 二、按模块调整建议

### 1. 页眉：求职意向

**反馈**：缺少明确求职意向。

**现状**：HTML 仅有姓名与联系方式。

**建议**：
- 姓名下方单独一行：**主投岗位 + 行业方向**（如 B 端产品经理 / AI 产品经理 — 数字孪生 · 工业互联网 · 智能运维）。
- 可按投递 JD **只保留 1 个主意向**，避免三个岗位并列显得分散。
- 如需，可补充城市 / 远程 / 海外项目制偏好（参考 docx）。

**参考（docx 已有）**：
```
求职意向：B端产品经理 / AI产品经理 — 数字孪生 · 工业互联网 · 智能运维方向
接受北京 / 远程 / 海外项目制
```

---

### 2. 个人总结

**反馈要求的两块内容**：

1. **建筑学阶段**：提炼与产品相关的能力（跨专业协作、需求分析、设计评审等），最好有具体场景而非空词。
2. **转型理解**：说明为何转产品；**空间思维如何帮助 3D 可视化 / 孪生产品的需求抽象**。

**现状（HTML）**：仅「以建筑学背景做系统化需求抽象」，未写建筑实践、未写转型逻辑、未点明空间思维与岗位的连接。

**优化版 docx 已改进**：用「需求解构→方案推演→落地验证」类比 B 端；「空间复杂度→可交付系统架构」；DataMesh 成果与方向。

**仍建议在下一版加强**：
- 点名 2～3 个建筑阶段能力（跨专业协作 / 需求分析 / 设计评审等）。
- 结构建议：**建筑训练的方法论（上半段）→ 在孪生/运维/仿真上的落地（下半段）→ 1 句差异化（空间语义 / 3D 场景）**。
- 避免只写「有建筑背景」一句带过。

---

### 3. 核心能力

**反馈**：加简单对比，例如——将大客户共性需求抽象为通用模块，**而非逐个项目做加法**。

**现状（HTML）**：有「抽象为通用模块与实施基线」，但缺少「不是什么」的对照。

**建议**：在「平台化与路线取舍」一条中并入对比句（docx 第二条可直接迁移）。

**示例句式**：
> 从大客户定制中提炼通用模块与实施基线，**而非逐个项目做加法**——实现约 70% 定制向标品转化……

---

### 4. 工作经历：70% 标品化

**反馈（面试必问）**：70% 的判断标准是什么？剩余 30% 为什么不能标品化？

**现状（HTML）**：只有「约 70% 定制需求向标品转化」，无口径、无边界。

**优化版 docx 已补**：剩余 30% 因数据接口差异与合规要求保留定制通道。

**建议简历再补 1～2 句（与内部统计一致）**：

| 要素 | 建议写法方向 |
|------|----------------|
| 统计口径 | 按需求条数 / 人天 / 模块数（选一种写清） |
| 进标品条件 | 高频 × 跨客户 × 研发可复用 × 商业优先级（见 `docs/面试/面试问答沉淀.md` Q6） |
| 30% 典型 | 客户专属 UI、非标接口、合规审计、一次性集成等 |

**原则**：简历上的数字必须能在面试中用案例辩护，避免「漂亮数字」。

---

### 5. 项目经历：问题 → 方案 → 结果

**反馈**：讲故事要 123 闭环；Inspector 需能说清 0→1 时长与核心决策点。

**现状（HTML）**：`项目背景` + 能力型 bullet，结果散落在各条。

**优化版 docx**：Inspector / Designer 已用 **问题 / 方案 / 结果** 分段。

**建议**：
- HTML **对齐 docx 的项目结构**。
- **结果**集中 2～3 条量化（覆盖率、周期压缩、商机/合同等），不要埋在方案 bullet 里。
- **时间口径统一**（投递前必查）：
  - docx 个人总结：约 5 个月 Demo→商用；项目时间 `2024.10`。
  - HTML：项目 `2025.10`；工作经历「1 个月 Demo、次月 MVP」。
  - 需统一为一套说法，避免 HR/面试官对不上。
- **Inspector 关键决策**（简历各半句，面试展开）：为何做通用底座 vs 继续项目制；MVP 范围如何砍；Agent 闭环与可视化底座的先后顺序等。
- **Designer**：docx 结果仍为「具体数据待补充」——建议补 1 个可验证指标（验证周期、搭建工时、效率提升等）。

---

### 6. 教育背景

**反馈**：建筑学期间实践也可体现产品能力。

**现状**：两版均只有学校 / 学位 / 时间。

**建议**（篇幅紧则二选一）：
- 教育下增加 **1 条 bullet**：studio / 毕业设计中的多专业协同、方案评审、需求与约束梳理；
- 或把上述内容压缩进个人总结，不重复占行。

UCL 若涉及 AI / 交互 / Unity 等，可一句带过，支撑「转型有连续准备」。

---

### 7. 三条产品线：简历 vs 面试

**反馈**：3 条产品线如何分配精力？Inspector 0→1 多久、核心决策是什么？

**简历建议（半句即可，需与真实时间分配一致）**：
- 例：Inspector 战略 0→1 为主；Designer 迭代 + 大客交付支撑；Director 维护与售前资产。

**面试必准备**（详见 `docs/面试/面试问答沉淀.md`）：
- 问题 → 你的取舍 → 结果 → 沉淀；
- 避免只说「都很重要」。

## 四、不必全写进简历、但面试必须闭环的话题

| 面试题 | 简历可留的「钩子」 | 口述要补全的内容 |
|--------|-------------------|------------------|
| 为何从建筑转产品？ | 个人总结 1 句「可复用产品能力」 | 决策逻辑、UCL/实践、跨行适应案例 |
| 建筑给产品什么独特优势？ | 空间思维 / 结构化需求 1 句 | 与孪生、3D、跨团队对齐的具体例子 |
| 三条产品线如何分配精力？ | 半句优先级原则 | 时间占比、典型一周、冲突如何裁 |
| Inspector 0→1 多久？核心决策？ | 统一后的月数 + 1 个决策关键词 | 问题定义、scope、立项、商用标准 |
| 70% 标品如何判断？30% 是什么？ | 口径 + 30% 原因各 1 句 | 佛吉亚 / 运营商等案例与前后数据 |

**注意**：简历改完后，口述应与纸面一致；可对照 `docs/面试/面试问答沉淀.md`、`docs/项目与学习/ai-agent-alert-diagnosis-story.md` 演练。
