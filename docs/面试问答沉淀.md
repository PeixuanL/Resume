## 三、核心能力深挖

### Q5. 你怎么理解「AI 与产品融合」？举一个最代表性的例子。

**答：**

我理解的融合不是「接一个大模型 API」，而是回答三个问题：**解决什么业务闭环、输入什么上下文、如何可控地失败**。

在 Inspector 运维场景，一线人员要的不是聊天，而是告警来了之后能快速判断原因、影响范围和下一步动作。我的做法是：

1. 梳理主流程：告警接入 → 上下文检索 → 诊断辅助 → 方案/工单 → 执行 → 知识回流。  
2. 按环节拆 Agent（告警理解、RAG、诊断推理、方案生成、工单草稿等），并规定每个环节的输入输出边界。  
3. 产品机制上坚持：**可解释输出、低置信兜底、人工确认、反馈回流**；消防/电梯/强电等高风险场景必须人工复核或升级。

这样 AI 是嵌在运维闭环里的生产力，而不是独立功能 demo。

### Q6. 你说的「平台化」具体怎么做？怎么判断什么该进标品？70% 怎么来的？

**答（建议按 1→2→3→4 讲，约 1～1.5 分钟）：**

#### 1）口径（先定调，避免「拍脑袋」）

简历里的 **约 70%** 不是单项目体感，而是：对多个**大客项目的需求做盘点、归类、对比**后的估算——看有多少条目适合沉进产品基盘，而不是继续项目制堆功能。

#### 2）我怎么拆需求（方法论）

- **横向对比**：拉通**相似业务场景**（如设施运维孪生交付、灯塔工厂/运营商多期项目）的历史需求清单，看哪些反复出现。  
- **外部校准**：结合**竞品与现有产品能力边界**做调研（面试时可据记忆补充 1 个对标对象或能力差距，不必展开成报告）。  
- **三维拆分**（识别后再决定放哪一层）：  
  | 维度 | 含义 | 典型去向 |
  |------|------|----------|
  | **基盘** | 平台底座能力，跨行业可复用 | 进 Roadmap / Inspector 底座 |
  | **行业取向** | 同一行业多客户的高频共性 | 行业包 / 实施基线 |
  | **客户定制** | 单客户绑定、难复用 | 配置层、单独报价或项目增量 |

#### 3）什么才「转化进标品」（判断标准）

只在三维拆分后，对**基盘 + 行业取向**里同时满足以下条件的条目做标品化：

**基础能力 · 行业高频 · 跨客户 · 可复用**（可叠加商业优先级：窗口紧的先做）。

**不进标品**的典型项（简历可不写，追问 30% 时用一句带过）：客户非标数据接口、合规审计、一次性集成、强绑定客户原有系统的深度耦合。

#### 4）落地三步（与简历「盘点归类」对齐）

1. **盘点归类**：需求按场景/模块归类（沙盘、资产追踪、告警可视化、工单流转等）。  
2. **抽象**：沉淀「业务对象 + 状态机 + 数据接口」，而不是某客户的 UI 皮肤。  
3. **基线化**：写入底座能力与实施基线；定制留在**客户定制**维，用配置或明确报价承接。

#### 5）案例 + 结果（收口）

围绕佛吉亚、头部运营商等大客：把告警可视化、复杂工单状态机、3D 沙盘基线等沉进 Inspector 底座，实施侧重配置与空间语义对齐。团队整体约 **70%** 定制诉求标品化；该运营商多期交付由 **5–6 个月** 压到 **1–2 个月**；Inspector MVP 覆盖海外标杆 **80%+** 高优标准诉求——是「项目驱动 → 产品沉淀」的结果。

> 更完整的单客户故事见 **Q12**；与简历工作经历「业务梳理与标品转化」一条口径保持一致。

### Q7. 资源有限时怎么做优先级取舍？

**答：**

我会对齐四个维度：**商业窗口（合同/POC 节点）> 底座复用度 > 技术风险 > 团队产能**，然后给售前和研发一张「做 / 不做 / 替代方案」清单。

举例：多项目并行时，若某定制会伤害底座架构，我会推动用配置或已有模块组合替代，并把增量需求记入下一版本；若某 POC 关系签约窗口，则集中产研做最小可验证闭环，而不是追求完美功能。

我在团队里的角色常是**产研枢纽**：不是简单传话，而是把业务压力翻译成可执行的版本范围和验收标准。

### Q8. 你怎么做商业化与交付协同？

**答：**

B 端 AI 产品的商业化关键是**对齐验证路径和交付边界**，避免售前承诺与产品现实脱节。

我的做法：  
- **售前阶段**：输出可演示的 Demo、方案 PPT 和口径一致的 value story（丰田、Goodman 等 10+ 份深度方案）。  
- **POC 阶段**：提前定义范围、成功标准、数据前置条件和试点节奏；跨境项目配合英文沟通与本地化预期管理。  
- **交付阶段**：用 SOP、发布流程、内部交付清单降低一线 variance——例如 DLC 发布从 4 周缩到 1–2 周，本质是商业化交付效率。

结果上，我配合售前推动过日本头部建工 **POC → 正式合同**，也用底座支撑过新加坡数百万元级项目交付。

### Q10. Inspector 的 AI Agent 你是怎么设计的？为什么不用一个「大聊天框」？

**答：**

因为运维场景要的是**可追责的决策辅助**，不是开放域对话。

设计上我按业务链路拆 Agent，并配套 RAG 知识源（台账、BrickSchema 关系、时序数据、历史工单、SOP/知识库）。核心原则：

1. **必须可解释**：输出结构含结论、依据、置信度、建议动作、缺失信息。  
2. **必须可兜底**：低置信不给确定结论，只给检查项；高风险设备强制人工/专家升级。  
3. **必须可回流**：人工修正和工单结果沉淀回知识库，避免错误案例污染。

MVP 阶段与业务方约定场景验收脚本，用走查 + bad case 归类迭代提示词和检索策略——这是 B 端 AI 产品化，不是一次性算法 demo。

## 五、AI 产品专项

### Q14. 如何评估 B 端 AI 功能是否「做得好」？

**答：**

我不用单一模型指标，而用**业务 + 产品**双层指标：

| 类型 | 示例指标 |
|------|----------|
| 业务效果 | 平均诊断时长、工单一次解决率、重复告警率、人工升级比例 |
| 产品体验 | 建议采纳率、人工修改率、低置信触发率、bad case 收敛速度 |
| 安全合规 | 高风险场景误触发率、必须人工复核的覆盖率 |

MVP 阶段重点是**场景脚本验收 + bad case 分类**，先保证「不敢乱说」再追求「说得更好」。

### Q15. RAG 在产品里你会怎么设计？常见坑是什么？

**答：**

**设计要点**：  
- 知识源分层：设备台账 / 空间关系 / 时序与告警 / 历史工单 / SOP（步骤化而不只是整 PDF）。  
- 检索要与告警对象绑定：先结构化「问题是什么」，再检索，避免泛化问答。  
- 生成层要求**引用依据**，没有依据就不给确定结论。

**常见坑**：  
1. 把未审结工单直接灌库 → 错误经验污染；应人工审结后沉淀。  
2. 整文档切块导致检索噪音 → SOP 要拆成可执行步骤。  
3. 忽略缺失数据提示 → 模型幻觉；产品上要显式列出「还需要哪些数据」。

价值在于：**缩短「想法 → 可讨论原型」的周期**，让研发和业务更快对齐，而不是替代正式研发流程。对 B 端 PM，这等于多了一种「可执行的沟通语言」，尤其适合复杂交互和 3D 相关需求的早期验证。

## 六、协作与软技能

### Q17. 你作为「产研枢纽」，和销售/研发意见不一致怎么办？

**答：**

先对齐**共同目标**（签约、按期交付、底座不被击穿），再把争议拆成「事实」和「判断」。

- 与销售：用客户成功标准和 POC 范围说话，提供「能做 / 延期做 / 需额外成本」三档方案，而不是直接拒绝。  
- 与研发：把需求翻译成验收条件和优先级依据，承认技术债和产能现实。

若仍无法一致，升级到负责人决策，但我会上会前准备好**影响分析**（对合同、对架构、对工期），提高决策效率。我经历过资源受限下协调前后端、美术、测试多团队并行，核心是**透明优先级 + 明确验收**。

---

### Q18. 你同时负责三条产品线，如何管理精力？

**答：**

1. **共用底座思维**：Inspector、Designer、Director 在场景、数据、交付资产上找协同，避免三条线三套说法。  
2. **版本节奏化**：11 个版本迭代靠 Roadmap 和固定评审点，而不是随时插入。  
3. **资产复用**：售前 Demo、SOP、培训清单跨产品线沉淀，减少重复劳动。  
4. **明确主战场**：每个阶段有主线（如 Inspector 0 到 1 期），其余线维持交付与关键迭代。

**优势**：

### Q21. 压力最大的一段时间你怎么度过的？

**答：**

Inspector 从 0 到 1 且同时要保障 Designer/Director 交付时，压力来自**时间、客户窗口和跨团队依赖**。

我的应对：  
1. 把里程碑拆到周（Demo → MVP PRD → 试点验收）；  
2. 每日只追 **3 件阻塞项**（数据接入、场景基座、Agent 边界）；  
3. 对非关键定制坚决进 backlog；  
4. 用文档和 SOP 减少重复解释（50+ PRD/验收文档、4 份发布 SOP）。

这段经历强化了我在资源约束下**用优先级换确定性**的习惯。
