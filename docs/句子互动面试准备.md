## 三、RPA、产品架构、交付模式

### 3.1 RPA 在他们语境里是什么？

**不是** 传统 UiPath 式「点 SAP、填 Excel」的桌面 RPA。

**是** **对话 / IM RPA**（Wechaty：*Conversational RPA SDK*）：软件机器人 **模拟真人在微信 / 企业微信上的操作**——发消息、加好友、拉群、读群消息等（很多能力无官方全自动开放 API，需 RPA 层接入）。

**一句话**：RPA = 替运营完成大量重复、有固定节奏的 IM 操作；**SOP** = 这些操作按业务剧本编排好的 **时间 + 规则**。

### 3.2 RPA + SOP 具体例子

| 场景 | 人工 | RPA + SOP |
|------|------|-----------|
| 新客加好友 | 通过后手动欢迎、发资料 | 触发：通过好友 → 自动欢迎语 + 链接 + 打标「渠道-抖音」 |
| 培育跟进 | Excel 记第 3/7 天手动聊 | T+0 / T+3 / T+7 自动不同话术 |
| 活动群发 | 复制 500 人，易漏、易封 | 按标签筛选、定时、分批发送 |
| 社群 | 每天固定早报/福利 | 群 SOP：入群欢迎 → 第 3 天案例 → 第 7 天券 |
| 多账号 | 各登各的企微 | **聚合工作台**：一屏多号、分配、协作 |
| 简单咨询 | 反复答发货/退换货 | 关键词 / 知识库 / AI → 复杂转人工 |
| 系统联动 | 不知谁已下单 | API：订单变化 → 打标「已购」→ 换 SOP |

### 3.3 RPA 与 AI / Agent 的分工

| 类型 | 特点 | 目标 |
|------|------|------|
| **RPA + SOP** | 流程确定、步骤可预期 | **稳定、规模化**（什么时候发、发给谁、哪条模板） |
| **Agent / 大模型** | 问法不确定、需理解与生成 | **体验与转化**，但必须 **可控**（合规、不乱承诺） |

**与本人经历类比**：Inspector「告警 → 检索台账/SOP → 处置建议 → 人工确认 → 知识回流」≈ 私域「消息 → 检索商品/政策 → 回复草稿 → 客服确认 → 话术沉淀」。

### 3.4 产品逻辑分层（复习用）

```
┌─────────────────────────────────────────────────────────┐
│  开放与交付：API/SDK · CRM/订单对接 · SaaS/私有化/定制   │
├─────────────────────────────────────────────────────────┤
│  平台：企业控制台 · 多账号聚合 · IAM 权限 · 标签/报表    │
├─────────────────────────────────────────────────────────┤
│  智能：知识库/话术库 · LLM/Agent · 人机协同/转人工      │
├─────────────────────────────────────────────────────────┤
│  自动化：SOP 引擎（定时/触发/标签）· 规则/关键词分流     │
├─────────────────────────────────────────────────────────┤
│  RPA：Wechaty · 稳定登录 · 消息/好友/群/朋友圈等操作     │
├─────────────────────────────────────────────────────────┤
│  渠道：企业微信 · 个人微信 · 飞书 / WhatsApp 等          │
└─────────────────────────────────────────────────────────┘
```

### 3.5 产品形态（怎么卖）

| 形态 | 说明 |
|------|------|
| **SaaS 云产品** | 如「句子秒回」类，开箱订阅 |
| **SCRM / 私域套件** | 获客 → 标签 → SOP → 社群 → 看板 |
| **PaaS** | API/SDK/Webhook，客户自研系统调用发消息、管客户 |
| **私有化** | 金融、大品牌数据不出域 |
| **定制开发** | 行业话术、大促、深度对接 |

公开信息另有 **企业控制台**（如 `miaohui.juzibot.com`）用于企业侧配置。

### 3.6 客户侧运营怎么用（一天简化流）

1. 公域投流 / 门店扫码 → 加企微（渠道码打标）
2. RPA 执行欢迎 SOP → 培育 or 拉群
3. 活动期定时群发 / 群 SOP
4. 客户提问 → AI 或关键词 → 转人工
5. 下单后标签变化 → 复购 / 沉默唤醒 SOP
6. 主管看板：触达、转化、人效、封号与话术优化

### 3.7 句子互动侧交付模式（推断，面试可验证）

| 阶段 | 内容 | 本人可对标经历 |
|------|------|----------------|
| 售前 | ROI（获客成本、人效、转化）；Demo 一条完整 SOP | 10+ 售前方案、Demo 视频 |
| POC/试点 | 选 1 品牌 / 1 链路，定成功标准 | 日本建工 POC→合同 |
| 实施上线 | 配企微、话术、标签、账号、培训 | Inspector 实施基线、发布 SOP |
| 定制/集成 | API 接 CRM、订单；私有化 | 矩阵集成 IoT/DFS、大客边界 |
| 持续运营 | 改 SOP、控风控、迭代知识库 | 版本迭代、bad case、标品化 |

宣传语 **「不改变运营路径也能 10 倍提效」** ≈ 不推翻客户原有私域玩法，把现有 SOP **配置进系统自动跑**（类似「定制转可配置基线」）。

## 四、IAM 是什么？

### 4.1 定义

**IAM = Identity and Access Management（身份与访问管理）**

回答三类问题：

1. **你是谁**（登录身份）
2. **你能干什么**（角色权限）
3. **你能看到哪些数据**（数据范围 + 审计）

### 4.2 在私域 SCRM / RPA 产品中的例子

| 对象 | 例子 |
|------|------|
| 组织 | 集团 → 事业部 → 门店 → 班组 |
| 角色 | 超管 / 运营主管 / 一线客服 / 只读分析 |
| 资源 | 能操作哪几个 **企微账号**、哪些 **客户群**、哪套 **话术库** |
| 动作 | 能否导出手机号、改 SOP、开 AI 自动回复 |
| 审计 | 谁改了标签、谁发了哪条群发（合规、客诉） |

企微自有组织架构；SCRM 再叠一层：**一人登录控制台管多个销售名下企微号** → 即 IAM + 多账号权限。JD 写 IAM 偏好，多指 **B 端多角色、多租户、数据权限、审计**，不一定是「只做 SSO 的技术 PM」。

### 4.3 面试话术（未做过 IAM 产品线）

> 大客交付里做过 **多角色协作边界**（谁能改场景、谁能批工单、试点数据范围），本质是 **角色 × 资源 × 动作 × 审计**。未主导独立 IAM 产品，但理解千人企业采购会问权限与合规，愿在企微多账号、导出、金融场景快速补齐。

## 五、「短期 vs 长期」竞争重心

### 5.1 短期（0–12 个月，客户为何先买单）

| 关键词 | 含义 | 产品要交付 |
|--------|------|------------|
| **稳定触达** | 消息真发到客户手机；少封号、少掉线；群发不全军覆没 | RPA 稳定、分批、频控、通道备份 |
| **运营 SOP** | 「第几天发什么、什么标签发什么」配好就能跑 | SOP 编辑器、触发器、标签、营销日历、人效报表 |

**商业逻辑**：先为 **省人、规模化触达** 买单（ROI 直观：5 人管 100 号）。  
**PM 工作**：配置体验、稳定性、看板、实施速度 → 对标本人「发布 4 周→1–2 周」。

### 5.2 长期（差异化与续费）

| 关键词 | 含义 |
|--------|------|
| **知识库** | 商品、政策、FAQ、合规话术、审结后的优秀会话 |
| **可控 Agent** | 意图 → 检索 → 生成 → **引用依据** → 低置信转人工 → 禁止乱承诺 |
| **CRM/订单打通** | 已购 VIP、客单价、沉默 30 天 → 驱动不同 SOP 与话术 |

**商业逻辑**：短期 SCRM 易同质化；长期靠 **懂客户的数据 + 懂业务的 AI + 与业务系统一体** 提高转化与续费。  
**PM 工作**：知识库运营、Agent 评测/bad case、集成与数据模型 → 对标本人 **RAG + Agent 验收 + 多系统矩阵**。

## 八、面试口述材料

### 8.1 自我介绍（约 90 秒）

您好，我是李沛宣，目前在 DataMesh 担任 B 端产品经理，专注 **B 端 AI 产品**与复杂行业场景的 **0→1 平台化**。

过去一年我主导 **Inspector 设施运维底座**从定制-heavy 走向标品：接手后 1 个月交付可面客 Demo，次月推动 MVP 商用；约 **70%** 定制诉求沉淀为通用模块，头部运营商交付从 **5–6 个月压到 1–2 个月**。在 AI 上，我负责「告警—诊断—处置」链路的 **Agent 与 RAG 设计**，强调可解释输出、低置信兜底和人工确认，让能力可验收、可迭代，而不是聊天 demo。

同时我长期做产研枢纽：配合 **10+** 售前方案与 POC，推动日本头部建工等项目 **POC 转正式合同**，并用 SOP 把发布周期从 4 周缩到 1–2 周。

我关注句子互动，是因为你们在做 **企业级 Agentic 数字员工**，把 RPA 与大模型落在企微私域的真实运营闭环里——这和我「**AI 进业务、项目经验变标品、对交付结果负责**」的路径高度一致。希望能把我在复杂 B 端场景里的抽象与验证方法，用到对话式营销与私域运营产品上。谢谢。

### 8.2 为什么句子互动（完整版）

1. **方向**：从「工业/设施运维数字员工」到「IM 私域数字员工」，都是一线降本增效、专家经验产品化。  
2. **可迁移**：Agent 分工、RAG 分层、人机协同、POC 验收 — 与客服/运营 SOP 自动化同一套产品纪律。  
3. **想贡献**：复杂 B 端 0→1、标品化、交付边界；客户多、场景碎，需要既能下钻运营动作、又能守平台架构的 PM。  
4. **学习点**：快速补企微生态与私域指标（加粉成本、触达、转化漏斗）；入职前看官网案例与 RPA 边界。

### 8.3 四个 STAR（各 2–3 分钟）

#### STAR-1：B 端方案 + 体验 — Inspector AI 告警诊断

- **S**：告警只显示表象，一线在 BMS、台账、工单、微信记录间切换。  
- **T**：可验收的「诊断辅助」，不是大聊天框。  
- **A**：拆 Agent；输出依据、置信度、缺失信息；消防/电梯/强电强制人工。  
- **R**：MVP 覆盖标杆 **80%+** 高优诉求；bad case 迭代机制。  
- **桥接**：企微客服同样要定义「什么自动回、什么转人工、什么进知识库」。

#### STAR-2：一线调研 → 验证迭代 — Inspector 0→1

- **S**：定制重、难复制。  
- **A**：盘点项目 → Roadmap → 1 个月 Demo、次月 MVP；场景验收脚本。  
- **R**：5+ 潜在客户；运营商交付大幅缩短。  
- **桥接**：从「一个品牌玩法」沉淀为可配置 SOP/模板。

#### STAR-3：竞品与策略 — 面试前自备

- 维度：企微官方、微盛/尘锋等 SCRM、通用 RPA、客服机器人。  
- 句子差异：Wechaty/RPA、Agentic、头部案例、PaaS/API。  
- 观点：短期拼触达+SOP；长期拼知识库+可控 Agent+CRM（见 §五）。  
- 谦虚：孪生赛道做过竞品取舍，私域竞品入职后系统补全。

#### STAR-4：跨职能对结果 — POC 转合同 + 发布 SOP

- 日本建工：收窄 POC、双语、成功标准 → 合同。  
- 发布：4 周→1–2 周，清单复用。  
- 体现：沟通、同理、取舍。

### 8.4 RPA 与 Agent 边界（约 1 分钟）

- **RPA**：流程确定、在 IM 上稳定执行（发、拉群、打标、定时）。  
- **Agent**：非结构化理解 + 生成，必须在规则与知识库内。  
- **边界**：价格/疗效/合规承诺 → 规则或人工；低置信 → 转人；审结会话才入库。

### 8.5 没做过私域，凭什么能做？（约 1 分钟）

- 承认场景差异。  
- 强调 **B 端方法论**：闭环、验收、人机协同、标品化。  
- 举例：Inspector 1 个月 Demo、次月 MVP。  
- 企微规则：**2 周可补齐**，愿从试点客户 SOP 学起。

## 九、通用面试题改口（对应《面试问答沉淀》Q1–Q8）

> 与 [面试问答沉淀.md](./面试问答沉淀.md) **Q1–Q8** 一一对应，按句子互动语境改口与延展。模拟面试可直接按本节口述；面试前一晚通读 1 遍即可。

### 9.1 Q1｜为什么做产品经理？建筑背景和产品岗怎么衔接？

> **改口要点**：把"系统化方案 + 可复用产品能力"落到**复杂 B 端 + 私域 SOP 化**的语境，让面试官听出**抽象能力可迁移到运营剧本与多角色协作**。

我做产品经理的动机来自建筑学训练——建筑做的就是**在复杂约束下做系统化方案**：空间关系、多专业协同、从概念到可交付图纸。硕士在 UCL 接触了 AI 平台策略、交互设计和 Unity 开发后，我清楚自己更想做的不是单栋建筑，而是**可被很多人复用的产品能力**。

这种思维与**句子互动的 B 端场景**是同构的：私域运营的本质也是"客户 + 标签 + 数据 + SOP"的系统问题——把一线运营/客服经验抽象成**可配置的剧本、可复用的话术、可治理的知识库**。建筑背景让我对多角色边界、流程闭环、对象语义更敏感，这在大客企微多账号、多角色协作里反而是优势。

所以背景不是劣势，而是做复杂 B 端产品时的**差异化抽象能力**——擅长把现场经验沉淀成模块、基线和实施路径，再交付给销售、运营和实施去复用。

**兜底句**：建筑教我的是"在不确定里搭可交付的系统"，做 B 端 PM 是延伸，不是转行。

### 9.2 Q2｜为什么专注 B 端 AI？

> **改口要点**：淡化"数字孪生"标签，强化"AI 进 B 端业务闭环"的方法论；用 §6.3 一句话定位 —— 运维 Agent 闭环 ≈ 私域客服闭环。

三个原因。

**第一，问题够硬。** B 端 AI 不是"加个聊天框"——无论是设施运维还是私域运营，都要 IoT/对话数据、知识库、SOP、人工兜底一起设计闭环，问题足够复杂，PM 的价值才够大。

**第二，价值可衡量。** B 端 AI 有清晰的商业语言：实施周期、人效、一次解决率、POC 转合同。这些指标我习惯用来对齐产研和销售，避免做"看起来 AI 很酷"但**不可验收**的功能。

**第三，经验可迁移、能放大。** 我过去做的是"告警 → 检索台账 → 诊断建议 → 人工确认 → 知识回流"这条运维 Agent 闭环；句子互动做的是"消息 → 检索商品/政策 → 回复草稿 → 客服确认 → 话术沉淀"这条私域客服闭环——**Agent 分工、RAG 分层、人机协同、bad case 治理**完全是同一套产品纪律。继续深耕 B 端 AI，能让我把方法论从设备场景放大到**人与客户的对话场景**，而不是重复从零开始。

**兜底句**：B 端 AI 的核心是"可控落地 + 可验收"，这是我两年多最想沉淀的事。

### 9.3 Q3｜你未来 3 年的职业规划？

> **改口要点**：短/中/长三段都向"IM + Agentic 数字员工"语境收口。

**短期（1 年）**：在**企业 IM + Agentic 数字员工**这条线上，把 Agent 能力从单客户验证推到**可规模复制的标准模块**——比如可控对话、知识库治理、SOP 触发与人工兜底的配置化基线，沉淀一套**验收指标和 bad case 治理机制**，让销售、实施、运营都能复用。

**中期（2–3 年）**：成为能独立负责**平台型 B 端 AI 产品**的 PM——不只是写 PRD，还能定义路线图、商业验证路径、跨产品形态（SaaS / PaaS / 私有化）的架构边界，主导从 POC 到规模化交付的全链路。

**长期**：希望在**「行业标准 + 产品化交付」的交叉点**上持续深耕——让复杂行业的 B 端软件，不再靠纯项目制堆人力增长，而是靠**可配置的产品能力 + 可控 Agent + 可验收的交付**实现规模化。句子互动正好在这条曲线上，这也是我希望加入的原因。

**兜底句**：3 年内我想从"做好一条产品线"，变成"能为一类 B 端 AI 产品定义路线图与商业验证路径"。

### 9.4 Q4｜为什么投句子互动？

> **改口要点**：完全展开，三段式——方向延伸 + 经验放大 + 缺口可补。与 §8.2「为什么句子互动（4 点精简版）」互补，可选其一或合并使用。

我投句子互动有三个层次的理由。

**第一，方向上是延伸而不是跳跃。** 你们做的是**企业级 Agentic 数字员工**——在企微 / 微信 IM 上，用 RPA 把运营动作稳定执行，用 SOP 把剧本配置进系统，用大模型把不确定的对话变成可控的回复。我过去做的是**工业/设施运维的数字员工**——告警接入、检索台账、诊断建议、工单生成、人工确认、知识回流。**底层方法论高度一致**：都是 Agent 分工 + RAG + 人机协同 + 可验收的 B 端闭环；场景从"设备运维"换到"人与客户的对话运营"。

**第二，我的经验在这里能直接放大。** 我在 Inspector 上做过三件事，对应你们当前的产品挑战：

1. **0→1 平台化**——70% 定制需求沉淀为标品，运营商交付从 5–6 个月压到 1–2 个月；对应你们"**平台能力 vs 行业包 / 大客定制**"的边界拿捏。
2. **可控 Agent 设计**——可解释、低置信兜底、强制人工复核、bad case 回流；正是私域 AI 客服里**合规、不乱承诺、不乱推荐**最关键的产品纪律。
3. **POC 转合同 + 发布 SOP**——日本头部建工 POC 收窄到可验证范围、发布周期 4 周→1–2 周；对应你们"**用可验证指标驱动签约 + 用 SOP 支撑规模化交付**"。

**第三，我承认两个缺口，并准备好补。** 一是企微生态和私域运营细则（加粉成本、触达、转化漏斗），我已经在通读官网案例和 Wechaty 边界；二是 IAM / 电商订单，本质是**多角色 × 资源 × 动作 × 审计** 和 **订单 → 标签 → SOP** 的逻辑——我不装懂，但能用大客交付里的多角色协作边界类比，**两周内补齐规则、入职后跟试点客户学**。

**兜底句**：方向延伸、方法迁移、缺口可补——这是我投句子互动的三层理由。

### 9.5 Q5｜怎么理解「AI 与产品融合」？

> **改口要点**：把"告警闭环"映射到"私域客服闭环"，并把"融合"定义为**纪律**而不是功能。

我理解的融合不是"接一个大模型 API"，而是回答三个问题：**解决什么业务闭环、输入什么上下文、如何可控地失败**。

在 Inspector 运维场景，我做过一条完整链路：告警接入 → 上下文检索（台账/时序/工单/SOP）→ 诊断辅助（多 Agent 分工）→ 方案/工单 → 执行 → 知识回流；产品机制上坚持**可解释输出、低置信兜底、强制人工复核高风险设备**。

这套方法论搬到句子互动的**私域客服/运营场景**完全同构：消息进来 → 检索（商品库 / 政策 / 历史会话 / 客户标签）→ 生成回复草稿 → 客服确认或直接发出 → 审结会话回流知识库。同样有**可解释**（引用商品/政策依据）、**可兜底**（价格、疗效、合规承诺必须规则或人工）、**可回流**（审结才入库，避免未审会话污染）的产品纪律。

我的核心观点：B 端 AI 的"融合"不是功能，是**纪律**——**什么可以自动回、什么必须转人工、什么进知识库**，这三条线划清楚了，AI 才是嵌在闭环里的生产力，而不是 demo。

**兜底句**：AI 与产品融合的本质是"三条线划清楚"——可自动、要兜底、可沉淀。

### 9.6 Q6｜平台化具体怎么做？什么该进标品？

> **改口要点**：在原 4 维框架上**追加"对底座架构的伤害程度"**，并把案例延展到私域 SOP / 话术 / 集成三个标品方向。

我用的判断框架是**高频 × 跨客户 × 研发可复用 × 商业优先级**——再叠加一个 B 端特有维度：**对底座架构的伤害程度**。

实操分三步：

1. **盘点**：把历史定制需求按"业务对象 + 状态机 + 数据接口"归类，不看 UI 细节。
2. **抽象**：找跨客户重复的部分进标品，找单客户独占的进配置层或明确报价的增量。
3. **基线化**：写进 Roadmap 的"底座能力 + 实施基线"，实施同学按 SOP 配置而不是从零写代码。

在 Inspector 上的结果是：约 **70%** 定制诉求沉淀为通用模块，MVP 覆盖标杆客户 80%+ 高优诉求，运营商三期实施从 5–6 个月压到 1–2 个月。

如果搬到句子互动语境，我会重点关注三类沉淀机会：

- **SOP 剧本基线**：教育、消费、零售各有典型的"加粉 → 培育 → 转化 → 复购 → 唤醒"模板，做行业包比一客户一配置可规模化得多。
- **话术库与合规规则**：每个行业都有不能乱说的话（价格、疗效、保险条款），把"禁词 + 转人工触发器"做成可配置标品。
- **集成连接器**：CRM、订单、营销自动化的对接，把"订单 → 标签 → SOP"这条数据链做成标品适配器，而不是一次一客户的定制开发。

这是我做"项目驱动 → 产品沉淀"的典型路径，**句子互动的多行业客户结构正好需要这种沉淀机制**。

**兜底句**：判断进不进标品，看"业务对象 + 状态机 + 数据接口"——UI 细节不算数。

### 9.7 Q7｜资源有限时怎么做优先级取舍？

> **改口要点**：在原 4 维基础上**追加"风控与合规边界"** —— IM/私域场景常见的封号、合规底线必须显式纳入。

我对齐四个维度：**商业窗口（合同/POC 节点）> 底座复用度 > 技术风险 > 团队产能**，遇到 IM/私域场景再加一条：**风控与合规边界**——不能为了某个客户的需求把封号风险或合规底线打穿。

落地时我会给销售和研发一张"**做 / 延期做 / 替代方案**"三档清单，每条都附**影响分析**（对合同、对架构、对工期、对风控）。

在 Inspector 多项目并行时的真实案例：某客户提了一个会伤害底座空间语义模型的定制，我推动用配置 + 已有模块组合替代，把增量需求记到下版本；某 POC 卡在签约窗口，则集中产研做最小可验证闭环，而不是追求完美功能。

类比到私域场景的常见取舍：

- 销售要"所有客户可任意频率群发" vs 平台稳定性（封号风险）→ 必须给频控规则，宁可拒绝个别"非常规请求"，也不能让标品掉号。
- 大客户要"完全替我们写 SOP" vs 团队产能 → 提供 SOP 模板 + 配置培训，而不是定制开发。
- 行业包 vs 通用平台 → 看签约金额和复用度，能进 3 个客户的优先做行业包，单客户专属走定制报价。

我的角色不是简单传话——而是把业务压力**翻译成可执行的版本范围和验收标准**，让取舍是结构化的，不是情绪化的。

**兜底句**：取舍 = 4 维优先级 + 1 条风控底线 + 三档透明清单。

### 9.8 Q8｜你怎么做商业化与交付协同？

> **改口要点**：把"售前—POC—交付"三阶段直接映射到句子互动的售前 ROI / POC 链路 / 实施基线 / 持续运营 4 段交付（参见 §3.7）。

B 端 AI 产品的商业化关键是**对齐验证路径和交付边界**，避免售前承诺与产品现实脱节。三个阶段我都有具体打法：

**售前阶段**：输出**可演示的 Demo + 方案 PPT + 口径一致的 value story**——丰田、Goodman、日本建工等 10+ 份深度方案我都参与过；与销售对齐"能做 / 延期 / 加价"的边界，避免无限承诺。

**POC 阶段**：**收窄到可验证范围**——日本头部建工的 POC 我们砍到 1–2 个可闭环场景，写清数据前置、成功标准、试点周期；POC 是合同证据，不是无限试用。结果是从试点直接转入正式合同。

**交付阶段**：用 **SOP + 发布清单 + 内部交付基线**降低一线 variance——DLC 发布周期从 4 周缩到 1–2 周，本质就是商业化交付效率。

这一套对应到句子互动我看到的链路：

- **售前** = 用 ROI（获客成本 / 人效 / 转化）+ 一条端到端 SOP Demo 打动客户决策层；
- **POC** = 选 1 个品牌 / 1 条链路（如"新客加粉 → 培育 → 首单"），定可验证的成功标准；
- **实施** = 配企微、话术、标签、账号、培训，按基线走而不是从零；
- **持续运营** = 改 SOP、风控参数、知识库迭代，沉淀 bad case 进下版本标品。

我的角色一直是**产研枢纽 + 商业守门员**：对内翻译需求与边界，对外用可验证指标推动从 POC 到签约。

**兜底句**：商业化协同 = 售前 value story + POC 可验证闭环 + 交付 SOP，不脱节、不无限承诺。

## 十一、建议反问面试官（选 3–4 个）

1. 岗位当前负责 **哪条产品线**？最紧的 **业务指标** 是什么？  
2. **平台能力** 与 **大客定制/行业包** 边界？考核偏 ARR/续费还是交付人效？  
3. **Agent** 与 **传统 RPA/SOP** 在路线图上的优先级？过去一年最大产品教训？  
4. AI 对话的 **合规与封号/风控** 如何产品化兜底？  
5. PM 与运营、售前、算法的协作模式？新人前 90 天期望产出？

---

## 十二、面试前 48 小时清单

- [ ] 通读官网 + AI/RPA 功能页，能说出 2–3 个可点评能力点  
- [ ] 搜「句子互动 + 客户案例 / Wechaty」  
- [ ] 口述 2 遍：90 秒自我介绍 + Inspector Agent STAR  
- [ ] 手写半页 **私域竞品粗分析**  
- [ ] 准备 1 个 **搞砸/收窄 POC 范围** 的真实故事  
- [ ] 带问题：企微、IAM、电商弱项 → 用 **学习计划** 收尾  

---

## 十三、风险提醒

- 别整场讲 3D/孪生技术细节，除非对方问；拉回 **闭环、指标、协作**。  
- 别贬低现公司；理由用「在更大规模 IM+AI 运营场景放大方法论」。  
- IAM/电商 **别装懂**。  
- 客户名注意 **脱敏**（头部车企 / 头部运营商）。

---

## 十四、复习用三句话

1. **句子互动** = IM 上的 **RPA（稳定执行）+ SOP（运营剧本）+ Agent（不确定对话）**，底层 Wechaty，上层接 CRM/订单。  
2. **我 match 的是**「闭环 + 可控 AI + 标品化交付」，用 **运维告警闭环** 类比 **私域客服闭环**。  
3. **短期** 卖人效与触达，**长期** 卖懂客户的数据 + 懂业务的 AI；**IAM** 是千人企业敢用的前提。
