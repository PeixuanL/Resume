## 1. 60 秒自我介绍参考结构

> 注意：这一段不是每个岗位直接复用的固定答案。生成 HTML 时要根据目标公司、JD、产品定位重写，但可以沿用“当前背景 -> AI 闭环能力 -> 目标岗位迁移”的结构。

面试官您好，我是李沛宣，目前在 DataMesh 做 B 端产品经理，主要负责数字孪生、设施运维和 AI 产品化方向。

我和 AI 产品经理岗位最相关的能力，是把一个 AI 能力从概念拆成真实业务里的任务闭环。过去在 Inspector 设施运维 Agent 项目里，我不是只做一个 AI 聊天入口，而是把告警、设备台账、空间位置、SOP、历史工单、诊断建议、工单草稿、人工确认和结果回写串起来，并定义每一步的输入、输出、边界和验收方式。

我比较擅长的不是单纯写需求，而是在不确定的场景里把问题拆清楚：用户到底要完成什么任务，AI 需要什么上下文，结果怎么证明可信，什么情况下必须人工确认，出了 bad case 怎么和研发继续调优。这个经历可以迁移到很多 AI 产品经理岗位，因为不管是 B 端运维、AI Agent 工具，还是海外效率产品，本质都是让 AI 从“能回答”变成“能完成一段可控、可解释、可迭代的任务”。

## 2. 通用项目介绍：Inspector AI Agent

我可以讲一个比较具体的 AI 项目：Inspector 设施运维 Agent。

这个项目的背景是，客户现场其实不是没有系统，而是系统很多但信息分散。告警系统知道异常，设备台账知道是哪台设备，空间系统知道它在哪个区域，知识库里有 SOP，历史工单里有以前怎么修过。但一线人员处理告警时，还是要自己翻资料、问资深工程师、再判断要不要派工。

所以我当时没有把 AI 做成一个孤立聊天框，而是先拆“告警到工单”的业务闭环：

```text
告警理解
-> 上下文检索
-> 诊断建议
-> 处置方案
-> 工单草稿
-> 人工确认
-> 结果回写
```

我的产品侧工作主要有几块。第一，我拆知识源和上下文：实时告警、传感器和设备状态用于判断现在发生了什么；设备台账和空间位置用于确认对象是谁、在哪里；SOP 和厂商手册作为标准处理依据；历史维修记录只能作为类似经验参考。第二，我拆 Agent 职责：告警理解 Agent 负责把原始告警转成结构化问题；RAG / 知识检索模块负责找 SOP、历史工单和设备资料；诊断 Agent 输出可能原因、依据和置信度；方案生成 Agent 给建议检查项；工单生成 Agent 只生成草稿，不自动派发。第三，我定义输出结构和人机边界，要求 AI 输出必须包含可能原因、判断依据、置信度、缺失信息、建议动作和人工确认标记。

我和研发配合时，不会只说“这个 AI 不准”。我会把问题拆成字段、接口、状态、输出 Schema 和验收脚本。比如设备 ID 能不能稳定拿到，SOP 有没有结构化，历史工单里有没有真实原因字段，AI 输出能不能按固定 JSON 返回，没有 SOP 时是否必须提示依据不足，低置信时是否禁止给确定结论。

最后结果上，我会谨慎表达：这段 AI 能力是产品侧方案验证、Demo/POC 和可交付规划，不包装成模型训练或大规模全量上线。但它证明了我能把一个很虚的“大模型诊断”拆成研发能做、测试能验、客户能理解、现场人员敢用的产品闭环。

## 3. RAG 详细标准答案

面试官您好，我具体讲一下 RAG 这块。

这个项目里我没有把 RAG 理解成“上传一堆文档让模型回答”，而是把它当成告警诊断的上下文检索层。因为现场运维里，AI 要判断一个告警，不能只靠通用知识，它必须知道：这是哪台设备、在哪个空间、属于什么系统、过去有没有类似故障、标准 SOP 是什么。

所以我当时把知识源分成四类：

- 设备台账：设备 ID、型号、位置、系统、维保周期、责任人。
- 空间和关系数据：设备在哪一层、哪个区域、上下游设备是什么。
- SOP / 厂商手册：标准排查步骤和安全注意事项。
- 历史工单：类似故障、真实原因、处理动作、处理结果。

判断“召回正确”，我不会只看语义相似度，而是看它有没有命中当前告警真正需要的证据。比如一个 B3 层空调机组温度异常告警，正确召回应包括：当前这台空调机组的设备资料；对应 HVAC / 空调机组温度异常的 SOP；同类型设备、同区域或同故障现象的历史工单；如果有时序数据，还要关联温度、水流量、阀门状态这类上下文。

如果它召回的是“空调相关文档”，但不是这类设备或不是这个告警类型，我会认为不合格。比如召回了普通空调清洁 SOP，却没有召回温度异常排查 SOP，这就不能算正确；或者只因为关键词相似召回了别的楼层、别的设备历史工单，也不能直接作为依据。

## 4. AI 回复准确性的标准答案

AI 回复准确，我分两层看。

第一层是 grounded：有没有基于召回材料回答。比如它说“可能是冷却水阀异常”，就必须能看到依据：当前温度异常和冷却水流量下降同时出现，历史工单中同区域出现过类似阀门卡滞，SOP 也要求先检查阀门开度和管路压力。

第二层是 cautious：不确定时有没有说清边界。如果缺少现场阀门开度或压力数据，它不能直接说“就是阀门坏了”，而应该说“中置信，建议先检查阀门开度、过滤器和管路压力”。

所以对我来说，准确的 AI 输出不是永远给确定答案，而是把依据、缺失信息和下一步检查路径说清楚。

## 5. 具体例子：B3 空调机组温度异常

举个具体例子。原始告警是：B3 层空调机组温度异常，持续 10 分钟。

一开始 AI 可能会回答：可能是冷却水阀异常，建议维修。这个答案太粗，现场人员不会信。

调优后我希望输出变成：

```text
可能原因 1：冷却水阀异常或局部管路堵塞。
判断依据：
1. 当前温度异常与冷却水流量下降同时出现。
2. SOP-AC-03 中温度异常排查流程要求优先检查冷却水阀和过滤器。
3. 历史工单中同区域曾出现冷却水阀卡滞问题。
置信度：中。
缺失信息：现场阀门开度、管路压力、过滤器状态。
建议动作：先检查阀门开度和过滤器，确认后生成维修工单草稿。
人工确认：需要运维人员现场确认后才能派单。
```

这个例子的重点不是 AI 说出了一个更长的答案，而是它把依据、风险和下一步动作讲清楚了，现场人员知道该信哪部分，也知道哪里还不能直接下结论。

## 6. 和研发配合调优的标准答案

我和研发配合调优时，不会只说“这个回答不准”。我会先把 bad case 分类：

| 类型 | 判断方式 | 对应动作 |
|---|---|---|
| 知识缺失 | 知识库里没有对应 SOP 或历史工单 | 补资料，明确 MVP 必需知识 |
| 切片问题 | SOP 被切得太碎，召回了步骤但丢了适用条件 | 调整 chunk、标题层级和上下文窗口 |
| 标签问题 | 缺少设备类型、空间、告警类型标签，导致跨楼层或跨设备误召回 | 补 metadata 和过滤条件 |
| 检索问题 | Top3 / Top5 没有正确资料 | 调检索策略、召回权重或重排 |
| 生成问题 | 资料召回对了，但模型表达太确定或乱引用 | 收紧输出 Schema、prompt 和兜底规则 |

我会给研发类似这样的验收表：

```text
测试告警：B3 空调机组温度异常
应召回：设备 ID、HVAC 温度异常 SOP、同区域同类历史工单
不应召回：其他楼层电梯/消防/普通保养 SOP
输出必须包含：原因、依据、置信度、缺失信息、建议检查项、人审标记
失败类型：知识缺失 / 切片问题 / 标签问题 / 检索问题 / 生成问题
```

这样研发拿到的不是抽象反馈，而是可以定位和迭代的产品验收标准。

## 7. 通用 Agent 交付答案

如果面试官问“这是不是只交了一个项目”，我会这样说：

它不是单客户一次性项目，而是从一个典型场景切入，沉淀成 Inspector 设施运维 Agent 的通用能力框架。

第一版会选一个典型告警，比如 HVAC / 空调机组异常，先跑通最小闭环。但产品架构不是只服务这一条告警，而是抽象成：

```text
告警理解
-> 上下文检索
-> 诊断建议
-> 处置方案
-> 工单草稿
-> 人工确认
-> 结果回写
```

后续可以扩展到电梯、消防、强电、冷站等不同设备类型。差异在于每类设备的 SOP、风险等级、知识库和人审规则不同，但底层 Agent 流程是通用的。

交付上，我会分三层：

- Demo-ready：用样例数据讲清“告警到诊断再到工单”的故事，支持售前演示。
- POC-ready：接入客户的部分真实设备台账、SOP 和历史工单，选 1-2 类高频告警跑场景验收。
- Delivery-ready 前置：需要客户提供稳定数据源、权限、知识库治理、工单字段、人工确认流程和 bad case 回流机制。

所以它不是“交付一个神奇 AI”，而是交付一套可落地的 AI Agent 闭环：数据前置、知识源、RAG 召回、结构化输出、人审机制、验收脚本和持续调优方法。
