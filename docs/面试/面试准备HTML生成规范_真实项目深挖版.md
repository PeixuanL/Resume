# 面试准备 HTML 生成规范：真实项目深挖版

> 用途：以后为任何新岗位生成面试准备 HTML 时，优先读取本文件。
> 目标：让页面里的答案默认达到“真实经历过项目的人在面试现场口述”的细节程度，而不是泛泛的简历 bullet。

## 1. 生成前必须先做的事

每次生成新岗位面试准备 HTML，先完成四步：

1. 读取目标 JD，解码岗位真正要招的人。
2. 读取我的简历和既有面试准备材料。
3. 读取可复用答案库：`docs/面试/AI产品经理可复用答案库_真实项目深挖版.md`。
4. 判断哪些问题可以复用通用答案，哪些问题必须按目标公司重新定制。

## 2. HTML 里的答案风格

所有重点答案默认采用“真实项目深挖版”：

- 用第一人称口述，像真实面试现场回答。
- 先讲业务场景，再讲产品设计，不先堆 AI 术语。
- 每个回答要包含：背景痛点、我的具体任务、我的产品设计、推进协作、验收方式、最终结果、边界。
- AI 项目必须落到具体机制：知识源、上下文、RAG 召回标准、Agent 职责、输出结构、置信度、人审、bad case、指标或阶段性验收。
- 不虚构上线数据。没有真实数字就明确说 Demo、POC、方案验证、可交付规划。
- 每个核心答案都要带“追问防守包”，方便面试官继续深挖。

## 3. 可复用与定制规则

### 可以直接复用的模块

以下问题在 AI 产品经理、AI Agent 产品经理、B 端 AI 产品经理、海外 AI 工具产品经理岗位中，可以直接复用本答案库的主体内容，再根据目标岗位替换最后的岗位落点：

- RAG 怎么设计。
- 怎么判断 RAG 召回正确。
- 怎么定义 AI 回复准确。
- 怎么和研发配合调优。
- 怎么做 Agent 职责拆解。
- 怎么定义人机边界。
- 没有完整线上指标时怎么证明价值。
- AI 项目的成果和边界。
- 具体 AI 项目深挖：尤其是 Inspector AI Agent / RAG / bad case 调优这组追问。

### 可以参考结构，但不能直接套用的模块

以下内容可以复用表达结构、能力主线和项目素材，但必须根据目标公司、JD 和产品定位重写：

- 60 秒自我介绍。
- 项目介绍的开头和结尾。
- “为什么适合这个岗位”的落点。
- 英文自我介绍。

### 必须按新岗位定制的模块

以下内容每次必须重新写：

- 为什么投这家公司。
- 这个岗位对应的产品定位。
- 这家公司具体产品的理解。
- JD 关键词映射。
- 我的经历如何迁移到这家公司。
- 90 天入职计划。
- 反问问题。
- 海外、增长、商业化、行业场景等岗位特有要求。

## 4. 页面结构要求

面试准备 HTML 里至少包含这些卡片：

1. 快速定位：公司、岗位、岗位本质、面试主线。
2. 60 秒自我介绍：按目标岗位重写，但可以引用 Inspector AI Agent 作为能力证据。
3. 核心项目故事：Inspector AI Agent，默认使用真实项目深挖版，再按目标岗位调整收口。
4. AI 深挖题：RAG、Agent、准确性、调优、交付，优先复用答案库中的标准答案。
5. 岗位匹配题：为什么是这家公司、为什么是这个岗位。
6. 产品理解题：目标公司产品定位、用户、核心链路、改进方向。
7. 压力追问：你个人做了什么、有没有上线、没有指标怎么办。
8. 反问准备：围绕岗位目标、Agent 主线、指标、协作边界。

## 5. 重点答案模板

每个重点答案按这个结构生成：

```text
面试官您好，我具体讲一个真实项目。

背景是……
当时真正的问题不是……而是……

我的任务不是……而是……

我具体做了几件事：
第一……
第二……
第三……

这里我和研发/设计/测试/客户的协作方式是……

我对“做完”的定义不是……而是……

最后结果上，我会谨慎表达……
这段经历迁移到贵公司的岗位，就是……
```

## 6. AI 深挖题必须覆盖的验收口径

AI / Agent / RAG 相关问题必须讲清：

- 业务闭环：用户从什么任务开始，到什么结果结束。
- 上下文：AI 需要哪些业务对象和资料。
- 知识源：哪些是事实、哪些是标准依据、哪些只是经验参考。
- 召回正确：是否命中正确对象、正确问题类型、正确知识源、有效证据。
- 生成准确：是否 grounded、是否谨慎、是否说明依据和缺失信息。
- 输出结构：原因、依据、置信度、缺失信息、建议动作、人审标记。
- 调优方式：知识缺失、切片问题、标签问题、检索问题、生成问题。
- 交付边界：Demo-ready、POC-ready、Delivery-ready 前置条件。

## 7. 禁止写法

不要写成：

- “我负责 AI 产品规划和落地，提升了用户体验。”
- “我通过 RAG 提高了回答准确率。”
- “我和研发沟通优化模型效果。”
- “最终显著提升效率。”

必须改成：

- 我拆了哪些业务对象、字段、状态、输出 Schema、验收脚本。
- RAG 到底召回什么才算正确，什么情况算不合格。
- AI 回复为什么可信，什么时候不能给确定结论。
- 和研发如何按 bad case 分类调优。
- 如果没有真实上线数据，就讲阶段性验收和可交付边界。
