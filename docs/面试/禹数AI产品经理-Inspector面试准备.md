## 2. Inspector Agent 闭环主故事

### 2.1 一句话定位

> Inspector 项目里，我参与的是把设施运维从“告警展示 + 人工处理”升级为“AI 辅助诊断 + 工单闭环 + 知识沉淀”的 Agent 产品化过程。

### 2.2 记忆主线

8 步闭环：

```text
告警触发
→ 告警理解
→ 上下文检索
→ 根因诊断
→ 方案生成
→ 工单草稿
→ 人工确认与执行
→ 结果回写与知识沉淀
```

5 个 Agent：

```text
告警理解 Agent
诊断推理 Agent
方案生成 Agent
工单生成 Agent
知识沉淀 Agent
```

3 条边界：

```text
AI 辅助诊断，不替代人决策
AI 生成草稿，不自动派发高风险工单
AI 输出必须有依据、置信度和缺失信息
```

## 3. STAR 完整版本

### S：背景

Inspector 面向设施运维 / 楼宇机电管理场景。我们看到的问题不是客户没有告警，而是告警之后没有诊断和处置闭环。

FAI 资料里有两个典型场景：

1. **B3 空调机组凌晨停机**：系统只提示“温度过高”。值班人员需要人工排查压缩机、冷凝器、阀门、管路、历史维修记录，最终才发现可能是冷却水阀异常。
2. **3 号电梯轿厢抖动**：同时 BMS 显示区域电力波动。工程师需要在设备台账、BMS 曲线、历史工单和厂商资料之间切换，才能判断是否是电压不稳或设备老化引起。

共性问题：

```text
告警只说明“发生了什么”
但不能解释“为什么发生”
也不能告诉用户“下一步怎么处理”
更不能把处理经验沉淀下来
```

传统工单系统能流转任务，但很多工单最后只记录“已处理”，真实根因、处置步骤、耗时、备件、SOP 是否有效都没有结构化沉淀。

### T：任务

我的任务不是做一个独立 AI 聊天框，而是把 AI 嵌入 Inspector 的主业务流程，设计一套可落地、可验证、可交付的 Agent 闭环。

目标拆成四个：

1. 让 AI 能基于告警对象理解设备、空间、系统和风险等级。
2. 让 AI 能检索设备台账、BMS/IoT 曲线、历史工单、SOP、模板库和 3D/MR 剧本。
3. 让 AI 输出可解释的诊断建议、处置方案和工单草稿。
4. 让执行结果回写，经过人工审结后沉淀为知识库或 SOP 模板。

### A：行动

#### A1. 拆业务闭环

我把运维链路拆成 8 个环节：

| 环节 | 产品设计重点 |
|---|---|
| 告警接入 | 告警来源、设备 ID、等级、时间、来源系统 |
| 告警理解 | 识别设备、空间、系统、告警类型、复发/并发关系 |
| 上下文检索 | 查台账、曲线、历史工单、SOP、图纸、MR 剧本 |
| 根因诊断 | 输出 TopN 原因、依据、置信度、缺失信息 |
| 方案生成 | 推荐检查项、SOP、备件、安全提示 |
| 工单草稿 | 自动填充设备、位置、故障描述、步骤、优先级 |
| 人工确认 | 运维主管确认、编辑、派发 |
| 回写沉淀 | 记录真实原因、执行结果、是否入库/存为模板 |

#### A2. 设计 Agent 分工

我没有把 Agent 设计成一个万能助手，而是按任务拆分：

| Agent | 做什么 | 输出 |
|---|---|---|
| 告警理解 Agent | 把原始告警转成结构化问题 | 设备、空间、系统、风险等级 |
| 诊断推理 Agent | 结合 RAG 和 Skill 判断可能原因 | 根因 TopN、依据、置信度 |
| 方案生成 Agent | 生成处置步骤和安全提示 | 检查项、SOP、备件建议 |
| 工单生成 Agent | 把方案转成工单草稿 | 标题、描述、设备、步骤、负责人建议 |
| 知识沉淀 Agent | 从审结工单中提取经验 | 知识条目、SOP 模板、事故报告草稿 |

一句话解释 Agent / Skill / RAG：

> Agent 是推进端到端任务的智能体，Skill 是 Agent 调用的专业能力，RAG 是 Agent 回答前先查现场资料。比如空调告警诊断 Agent 会通过 RAG 查设备台账、BMS 曲线、历史工单和 SOP，再调用空调诊断 Skill、SOP 匹配 Skill、工单生成 Skill，最后输出诊断建议和工单草稿。

#### A3. 设计人机协同和风险边界

风险策略：

| 风险等级 | 策略 |

| 低风险 | AI 推荐 SOP 和检查项，人工快速确认 |
| 中风险 | AI 生成方案和工单草稿，主管审核 |
| 高风险 | 电梯、消防、强电等只辅助诊断，强制专家/主管复核 |

```text
置信度
建议检查项
风险提示
```


#### A5. 把 Inspector 原有工单 / SOP / MR 能力接进闭环

Inspector 本身有工单、点巡检、预防性维护、模板库、MR 剧本、扫码签到、执行记录等能力。

所以 AI 不是孤立功能，而是连接这些模块：

```text
告警/故障提报
→ AI 自动填充基础信息和处置方案
→ 推荐 SOP 模板或生成步骤
→ 生成工单草稿
→ 人工指派
→ 一线按步骤执行，必要时扫码签到 / 调用 MR 剧本
→ 上传图片、视频、检查结果
→ 主管验收
→ 优质工单沉淀到模板库
```

记忆句：

> 模板库和 MR 剧本解决的是“怎么标准化执行”，Agent 解决的是“从告警到处置方案怎么生成”。两者结合，才能形成从诊断到执行再到知识沉淀的完整闭环。

### R：结果

结果不要说成“AI 完全替代人工”，而要说成产品化验证和能力沉淀：

> 这套设计支撑了 Inspector 从工单流转工具向智能运维闭环能力升级。它把告警、设备、空间、历史工单、SOP、模板库和 MR 作业引导串起来，让产品不只是“看见告警”，而是能进入诊断、方案、工单、执行和复盘。对交付侧来说，也把项目中的定制经验沉淀为通用 Agent 链路、SOP 模板和验收标准，为后续类似楼宇/厂务/设施运维场景复用打基础。

可衡量指标：

```text
诊断 TopN 命中率
工单创建耗时
平均故障处理时间 MTTR
人工修改率 / 驳回率
SOP 推荐采纳率
重复故障识别率
优质案例入库数量
标品模块复用率
```

业务测试怎么做：

```text
测试样例准备
→ 历史告警回放
→ 端到端链路验证
→ 专家/运维主管抽检
→ 记录人工修改点
→ bad case 分类
→ 进入下一轮迭代
```

我会把业务测试拆成 4 类：

| 测试类型 | 测什么 | 例子 |
|---|---|---|
| 单点能力测试 | 每个 Agent 是否完成自己的任务 | 告警理解是否识别正确设备、空间、系统、告警类型 |
| RAG / 知识测试 | 检索是否命中正确资料，引用是否可信 | 是否引用正确 SOP、历史工单、设备台账，而不是泛泛回答 |
| 端到端流程测试 | 告警到工单再到回写是否跑通 | 告警进入后能否生成诊断、方案、工单草稿，人工确认后能否派发和审结 |
| 风险兜底测试 | 低置信、高风险、无数据时是否安全降级 | 电梯/消防/强电不自动派单；数据不足时输出缺失信息 |

以“空调机组温度过高”为例，业务测试脚本会这样写：

```text
输入：B3 空调机组温度过高告警，包含设备 ID、空间 ID、告警等级、时间。
预期 1：系统识别为空调/HVAC 相关告警，并定位到正确设备和楼层空间。
预期 2：RAG 引用设备台账、最近 BMS 曲线、历史同类工单和空调温度异常 SOP。
预期 3：输出可能根因 TopN、依据、置信度、缺失信息和建议检查项。
预期 4：生成工单草稿，自动填充设备、位置、故障描述、建议步骤和优先级。
预期 5：人工确认后才能派发；执行完成后真实原因和处理结果能回写。
```

根据反馈怎么迭代：

我不会把所有问题都归因于“模型不准”，而是做 bad case 分类：

| 反馈类型 | 典型表现 | 迭代动作 |
|---|---|---|
| 数据缺失 | 没有设备 ID、空间关系、历史曲线、工单字段不完整 | 补字段、补数据源、调整 MVP 范围 |
| 检索未命中 | 明明有 SOP / 历史工单，但 Agent 没引用到 | 优化知识切片、标签、检索策略和同义词 |
| 知识过期/错误 | 引用了旧 SOP 或未审结工单 | 增加版本、发布状态、主管审结后入库机制 |
| 输出不可用 | 结论太泛、没有依据、工单字段缺失 | 固定输出模板，补字段校验和必填项 |
| 推理偏差 | 根因排序不合理，忽略关键告警并发关系 | 增加规则/Skill、调整 Prompt、补专家样例 |
| 流程问题 | 生成了方案但无法派单、无法回写 | 补工单接口、状态机、人审节点和异常路径 |
| 风险边界问题 | 高风险场景建议过于直接 | 强化设备分级、置信度阈值、强制人工复核 |

可直接背的 R 段补充：

> 在结果验证上，我不会只看 AI 回答是否流畅，而是用业务测试去验证闭环。比如空调机组温度异常场景，我会准备历史告警样例，检查系统是否识别正确设备和空间，RAG 是否引用正确的 BMS 曲线、历史工单和 SOP，诊断输出是否包含依据、置信度和缺失信息，工单草稿字段是否完整，人工确认后是否能派发，执行结果是否能回写。上线或试点后，如果一线反馈“AI 不准”，我会先做 bad case 分类：是数据缺失、检索没命中、知识过期、输出结构问题、推理问题，还是工单流程和权限问题。不同问题走不同迭代路径，而不是简单归因给模型。

最终交付给客户的成果：

## 4. 2 分钟背诵版

> Inspector 是设施运维方向的产品，我参与的是把它从“告警展示 + 工单流转”升级为“AI 辅助诊断 + 工单闭环 + 知识沉淀”的 Agent 产品化过程。
>
> 背景是商业楼宇和厂务运维里并不缺告警，BMS、IoT、设备系统都会提示温度过高、设备离线、电梯抖动这类问题。但告警只说明发生了什么，不能解释为什么发生，也不能告诉一线人员下一步怎么处理。FAI 资料里有个典型场景，B3 空调机组凌晨停机，系统只报温度过高，工程师还要人工查设备台账、BMS 曲线、历史工单、SOP 和图纸，最后才定位到冷却水阀异常。
>
> 所以我没有把 AI 设计成一个聊天框，而是嵌入 Inspector 的主流程。我把闭环拆成告警触发、告警理解、上下文检索、根因诊断、方案生成、工单草稿、人工确认、执行回写和知识沉淀。AI 负责整合上下文、生成诊断和草稿，人负责确认、派发和高风险复核。
>
> 在 Agent 设计上，我拆了告警理解 Agent、诊断推理 Agent、方案生成 Agent、工单生成 Agent 和知识沉淀 Agent。诊断 Agent 会通过 RAG 检索设备台账、BMS/IoT 曲线、历史工单、SOP、模板库和 MR 剧本，再调用空调诊断、电梯分级、SOP 匹配、工单生成这些 Skill，输出可能根因、依据、置信度、缺失信息和建议检查项。
>
> 和研发对接时，我重点明确输入输出、接口依赖、状态机、失败兜底和验收标准。比如工单 Agent 只能生成草稿，正式派发必须人工确认；电梯、消防、强电这类高风险场景必须升级主管或专家；AI 输出必须有引用来源和置信度。业务测试也不是只看回答是否流畅，而是看告警识别是否正确、RAG 是否引用正确资料、工单字段是否完整、执行结果是否能回写。
>
> 最后，这套设计把 Inspector 的工单、SOP 模板库、MR 剧本和执行反馈串起来，让产品从“告警展示”走向“可诊断、可处置、可复盘”的智能运维闭环。

## 5. 具体场景怎么讲

### 5.1 空调机组温度过高

```text
B3 空调机组温度过高告警
→ 识别设备、空间、所属暖通系统
→ 查台账：型号、维保周期、上次维护
→ 查 BMS 曲线：温度、冷却水流量、阀门开度、压力
→ 查历史工单：是否出现过阀门卡滞、过滤器堵塞
→ 查 SOP：空调机组温度异常排查步骤
→ 输出 TopN 根因、依据、缺失信息、检查项
→ 生成工单草稿，人工确认后派发
→ 执行结果回写，主管审结后进入模板/知识库
```

可解释输出：

```text
可能原因：冷却水阀异常或局部管路堵塞。
依据：温度升高与冷却水流量下降同时出现；历史工单中同区域曾出现冷却水阀卡滞；该部件缺少最近一次维护记录。
建议动作：检查冷却水阀开度、管路压力和过滤器状态；如确认阀门异常，按 SOP-AC-03 创建维修工单。
置信度：中。仍需现场补充阀门状态和管路压力。
```

### 5.2 电梯抖动 / 高风险设备

```text
3 号电梯轿厢抖动告警
→ 识别为高风险设备
→ 关联 BMS 区域电力波动
→ 查台账：型号、年限、厂商资料、责任人
→ 查历史工单：是否重复发生
→ 输出风险提示、影响范围、建议检查项
→ 强制主管或专家复核
→ 只生成工单草稿，不自动执行控制动作
```

重点：

> 电梯、消防、强电这类高风险场景，AI 的价值是帮助人更快获得上下文和检查路径，不是替代人做安全决策。

### 5.3 MR 剧本辅助维修

```text
维修工单下发
→ 工单引用 SOP 模板
→ 模板步骤关联 3D/MR 剧本和参考资料
→ 一线人员扫码签到，确认设备位置
→ 按步骤执行，上传图片/视频/检查结果
→ 离线场景缓存工单、设备、步骤、剧本资源和定位码数据
→ 执行后反馈该流程是否有效
→ 主管审结后沉淀为新模板或新版本
```

重点：

> 这段说明我不只懂 AI，也理解一线执行体验、离线缓存、3D/MR 资源、SOP 模板和版本控制。

### 7.2 Inspector 项目深挖

**Q7：这个项目解决什么问题？**

> 解决的是“告警之后没有诊断和处置闭环”的问题。传统系统能告诉用户温度高、设备离线、电梯抖动，但不能解释原因，也不能自动带出历史工单、SOP、设备台账和处置方案。Inspector Agent 闭环就是把这些上下文串起来，生成可解释的诊断、方案和工单草稿，并把执行结果沉淀成知识。

**Q8：你具体做了什么，不是研发做的吗？**

> 研发负责模型、接口和前后端实现。我主要负责把业务流程拆成产品结构和验收标准：哪些场景触发 Agent，Agent 需要哪些上下文，每一步输出什么，哪些动作必须人工确认，什么算验收通过。我会沉淀 PRD、流程图、字段说明、接口依赖和业务测试脚本。

**Q9：你最关键的产品判断是什么？**

> 最关键的判断是：工业/建筑运维场景里，AI 不能脱离业务对象和流程。它必须绑定设备、空间、告警、工单、SOP 和权限。否则就是一个回答好听但不可交付的聊天框。我的设计是让 Agent 嵌入主路径，输出结构化结果，并通过人审、回写和知识沉淀形成闭环。

### 7.3 Agent / RAG / Skill

**Q10：Agent、RAG、Skill 怎么区分？**

> Agent 是负责推进端到端任务的智能体，Skill 是 Agent 调用的专业能力，RAG 是 Agent 回答前先查现场资料。比如空调告警诊断 Agent 会通过 RAG 检索设备台账、BMS 曲线、历史工单和 SOP，再调用空调诊断、SOP 匹配、工单生成这些 Skill，最后输出诊断建议和工单草稿。

**Q11：为什么不用一个大聊天框？**

> 运维场景要的是可追责的决策辅助，不是开放域问答。大聊天框容易脱离设备对象、权限和工单流程。我的设计是告警触发后带着设备、空间、系统、历史工单、SOP 和曲线上下文进入诊断，再生成方案和工单草稿。这样输出有依据、有状态、有人工确认，也更容易验收。

**Q12：RAG 知识库怎么设计？**

> 我会分层处理：设备台账、空间关系、实时/历史告警、BMS/IoT 曲线、历史工单、SOP、厂商手册、模板库、MR 剧本。历史工单必须主管审结后再进入知识库，客户专属知识和通用知识也要隔离，避免错误经验污染模型输出。

### 7.4 研发对接与交付

**Q13：你怎么和研发对接？**

> 我会把 Agent 能力拆成输入、输出、接口、状态和异常处理。比如输入包括告警 ID、设备 ID、空间 ID、系统类型、告警等级；接口包括设备台账、BMS/IoT 曲线、历史工单、SOP、工单模板；输出包括根因、依据、置信度、检查项和工单草稿字段。低置信、无数据、高风险都要有兜底路径。

**Q14：业务测试怎么做？**

> 我会按真实场景写测试脚本，不只看 AI 回答是否流畅。比如空调温度异常场景，要验证是否识别到正确设备和空间，是否引用正确 BMS 曲线、历史工单和 SOP，是否输出依据和置信度，工单草稿字段是否完整，人工确认后是否能派发，执行结果是否能回写。

**Q15：怎么做交付验证？**

> MVP 阶段可以用场景脚本 + 黄金样例 + 专家抽检。上线后看诊断 TopN 命中率、工单创建耗时、MTTR、人工修改率、SOP 采纳率、重复故障识别率和优质案例入库数。对客户来说，更重要的是链路可解释、可复核、可追溯，而不是模型说得多流畅。

**Q16：使用方反馈怎么迭代？**

> 我会先做 bad case 分类：检索未命中、知识过期、Prompt/Skill 问题、推理错误、工单流程问题、数据质量问题、权限人审问题。不同问题走不同路径，比如补知识库、优化切片、调整输出结构、修改流程或补接口，而不是一概说模型不准。

### 7.5 风险与边界

**Q17：怎么控制 AI 风险？**

> 四层控制。输入侧要求基于设备、空间、告警和知识源；输出侧固定结构，包括结论、依据、置信度、缺失信息、建议检查项、风险提示；操作侧不自动执行高风险动作，强电、消防、电梯等必须人工确认；系统侧要有权限、租户隔离、工具调用日志和审计记录。

**Q18：AI 会不会误导一线人员？**

> 有风险，所以输出必须是“建议 + 依据 + 置信度 + 缺失信息”，不能只给一个确定结论。低置信度只给排查方向，高风险场景强制人工复核。执行后的真实结果也要人工审结后再入库，避免错误经验污染后续诊断。

### 7.6 FMEA / 禹数方向

**Q19：你怎么理解 FMEA 智能体？**

> 我理解 FMEA 智能体不是简单生成 FMEA 表，而是建立故障模式、原因、影响、检测方式、风险等级、预防措施和实际处置结果之间的因果关系。真正落地时，它要接入日常告警、巡检、维保和维修工单，把现场发生的 case 反向沉淀到知识库里。否则 FMEA 只停留在文档里，不能融入日常运维。

**Q20：Inspector 和禹数方向怎么连接？**

> Inspector 可以理解成可靠性管理的执行层。告警和巡检异常触发风险识别，AI 推荐可能故障模式和排查步骤，工单执行结果回写真实原因，再反向优化 FMEA 知识库、SOP 和模板。这和禹数强调的可靠性、FMEA、IoT 报警、巡检/维保/维修数据贯通是同一个方向。

### 7.7 Python / 技术兴趣

**Q21：你 Python 掌握到什么程度？**

> Python 我定位在产品验证和数据处理辅助上，比如整理告警样例、清洗历史工单字段、构造测试集、分析 bad case 分布、辅助接口联调或 mock 数据。我不会说自己做模型训练，但能用 Python 帮研发和测试把场景样例、字段校验和验证数据准备得更清楚。

**Q22：你对大模型技术怎么理解？**

> 我不会把所有能力都交给大模型。24x7 监控更适合规则、阈值、趋势和健康评分，因为它要稳定、低成本、可解释。LLM/Agent 更适合用户点击告警后的解释、检索、方案生成和工单草稿。这样能兼顾成本、稳定性和用户体验。

### 7.8 业务负责人深挖：指标、成果、搭建、准确率

**Q23：Inspector 中 Agent 的交付指标是什么？**

> 我会把 Agent 指标分成四层，而不是只说“准确率”。第一层是链路指标，比如告警接入成功率、RAG 检索成功率、接口可用性、P95 响应时延；第二层是效果指标，比如诊断 Top1/Top3 命中率、答案可用率、引用来源正确率、工单草稿字段完整率；第三层是效率指标，比如平均诊断耗时、工单创建耗时、人工修改率、SOP 采纳率、MTTR；第四层是业务指标，比如试点验收通过率、标品模块复用率、交付周期缩短和客户是否愿意进入下一阶段。
>
> 对设施运维这种场景，我认为最关键的不是报一个孤立准确率，而是“能不能进入业务闭环”：能否识别正确设备和空间，能否引用正确知识，能否生成可用工单，能否被人工复核，能否把真实结果回写。

**Q24：如果问你 Inspector Agent 的准确率是多少，怎么回答？**

> 我不会虚报一个生产级准确率。因为设施运维 Agent 的准确率必须有样本集、故障类型、专家标注和统计周期，否则 95%、99% 这种数字没有意义。更稳的说法是：在我负责的阶段，重点是 MVP/POC 的场景闭环验证，简历里可以讲的是 MVP 覆盖标杆客户 80%+ 高优先级诉求、吸引 5 家以上潜在商机，以及支撑实施周期从 5-6 个月压到 1-2 个月。
>
> 如果要设计准确率口径，我会用历史告警回放 + 专家抽检，比如准备 50-100 条典型告警样例，标注标准根因、可接受排查路径和必需引用来源，再统计 Top1/Top3 命中率、可用答案率、引用正确率和高风险误建议率。

**Q25：上线结果或产出成果怎么讲？**

> 我会分“产品产出”和“业务结果”两类讲。产品产出包括场景卡片、Agent 流程图、PRD、知识源清单、接口字段、工单草稿字段、状态机、人审规则、测试脚本、验收表和 bad case 分类。业务结果按简历真实口径讲：Inspector MVP 覆盖标杆客户 80%+ 高优诉求，形成告警、资产、工单、3D 沙盘和 Agent 闭环的 0→1 底座；吸引 5 家以上潜在商机；并支撑部分项目缩短实施周期。
>
> 我不会把它包装成“已经大规模上线并有长期准确率数据”。更稳的表达是：完成了 MVP/POC 级别的业务闭环验证，并沉淀成可复用产品能力和交付基线。

**Q26：Agent 是用什么搭建的？**

> 产品口径上，我不会把重点放在“用了哪个框架”，而是讲清楚 Agent 的工程形态：它不是一个独立聊天框，而是通过 AI Engine / Agent 服务接入主业务系统，调用设备台账、告警、时序数据、历史工单、SOP、工单模板等工具或接口，按“理解-检索-诊断-方案-工单-回写”的流程编排。
>
> 如果面试官追问框架，我可以说：PM 层面我理解 LangChain、LangGraph、Dify 这类工具的差异。像 Inspector 这种有条件分支、低置信兜底、人工暂停和回写的流程，更适合用图编排思路，比如 LangGraph 这类有状态编排；但在项目中我的职责不是指定底层框架，而是定义状态机、输入输出、工具调用、失败降级和验收标准，具体技术选型由研发结合现有 AI Engine 和系统架构决定。

**Q27：你和研发的协作机制是什么？**

> 我会按五步协作：第一，和业务/客户确认场景卡片，明确用户、触发条件、数据源、成功标准和不做范围；第二，和研发/算法做可行性评审，判断规则、RAG、多 Agent、接口和数据质量是否支持；第三，输出 PRD 和 AI 行为规格，包括输入输出、知识源、工具调用、状态机、人审边界、兜底策略；第四，联调阶段用真实样例跑业务测试，检查告警、检索、诊断、工单、回写是否端到端通；第五，上线或试点后按 bad case 分类迭代。
>
> 我在这里的价值不是替研发写模型代码，而是把复杂业务拆成研发能实现、测试能验证、客户能验收的规格。

**Q28：如果业务负责人问你怎么搭一个最小可用 Agent，你怎么答？**

> 我会先选一个高频、低到中风险、数据相对完整的场景，比如空调机组温度异常。MVP 不追求全设备全故障，而是跑通最小闭环：告警 ID 进来，识别设备和空间，检索设备台账、近期曲线、历史工单和 SOP，输出可能原因 TopN、依据、置信度、检查项，再生成工单草稿，由人工确认派发，最后把真实原因和处理结果回写。
>
> MVP 验收不是“回答看起来像专家”，而是这条链路能不能稳定跑通、输出是否可复核、低置信是否能兜底、高风险是否能强制人审。

**Q29：业务负责人问“你做 Agent 最难的点是什么”？**

> 最难的不是让模型回答，而是让回答进入可交付的业务链路。主要难点有四个：第一，现场数据不完整，设备 ID、空间、告警、工单和 SOP 可能对不上；第二，专业知识分散在工程师经验、历史工单和文档里，质量不一；第三，AI 输出必须可解释、可追责，不能幻觉式给结论；第四，工单、权限、人审和回写会影响真实落地。我的处理方式是先做数据盘点和场景分级，再用结构化输出、人审边界、引用来源和 bad case 机制逐步迭代。

**Q30：建筑转产品再转 AI，原因怎么回答？**

> 我的路径不是跳来跳去，而是同一条线逐步延展：建筑训练让我理解空间、系统和复杂约束；数字孪生产品把这种空间理解变成软件里的对象、数据和交互；到设施运维场景后，我发现客户不只需要“看见设备和告警”，更需要系统帮助判断原因、推荐处置、生成工单并沉淀经验。AI Agent 正好是在这个基础上进一步做诊断、推荐和流程编排。
>
> 所以我不是从建筑突然转到 AI，而是从“设计空间”延展到“设计空间运行系统”，再延展到“用 AI 帮助这个系统更智能地运行”。

**Q31：FMEA 智能体是什么？**

> FMEA 是 Failure Mode and Effects Analysis，中文一般叫故障模式与影响分析。传统 FMEA 会把设备或系统可能的故障模式、原因、影响、检测方式、严重度、发生度、探测度、风险优先级和预防措施整理成表。但传统问题是它经常停留在文档里，和日常告警、巡检、维保、维修工单没有打通。
>
> 我理解 FMEA 智能体，就是把 FMEA 从静态文档变成能参与日常可靠性管理的 Agent：当告警或巡检异常发生时，它能基于设备、故障模式、历史案例、SOP 和实时数据，推荐可能原因和排查路径；处理完成后，再把真实原因、处置结果和经验反向沉淀到 FMEA 知识库里。核心价值是让可靠性知识进入“发现-诊断-处置-复盘”的闭环。

**Q32：如果禹数问“你来了能怎么做我们的 FMEA 智能体”？**

> 我会先从一个可验证业务闭环切入，而不是一上来做大而全知识库。第一步，选定高价值设备或系统，比如洁净厂房空调、配电、关键工艺设备；第二步，盘点数据源，包括 FMEA 表、设备台账、FMCS/BMS/IoT 告警、历史工单、巡检记录、SOP、专家经验；第三步，定义 Agent 输出结构，包括可能故障模式、影响范围、依据、置信度、检查项、建议工单和风险提示；第四步，设计人审和回写机制，让专家确认后的结果反向更新 FMEA/SOP/知识库；第五步，用历史案例回放和专家抽检做试点验收。
>
> 这套方法和我在 Inspector 里的经验是相邻的：都是把静态知识、实时告警和执行工单打通，形成可复用的可靠性闭环。

### 7.10 简历补充追问：Workflow、系统接入、技术学习

**Q34：你简历里写 Agent/RAG/Workflow，Workflow 编排具体是什么意思？**

> 我理解 Workflow 编排不是简单画一条流程线，也不是一定绑定某个框架，而是把 Agent 完成业务任务的过程拆成可执行、可观测、可兜底的状态流。产品经理要定义每个节点做什么、输入输出是什么、调用哪些工具、什么时候分支、什么时候人工介入、失败怎么降级、结果怎么回写。
>
> 以 Inspector 告警诊断为例，我会把 Workflow 拆成：
>
> ```text
> 告警触发
> → 告警标准化：识别设备、空间、系统、等级
> → 上下文打包：拉取台账、曲线、历史工单、SOP
> → RAG 检索：召回相关知识和案例
> → 诊断推理：输出根因 TopN、依据、置信度
> → 风险分流：低风险给方案，高风险强制人工复核
> → 工单草稿：生成标题、描述、步骤、优先级
> → 人工确认：编辑、派发或驳回
> → 执行回写：记录真实原因、结果、是否入库
> ```
>
> 我在这里关注的不是“用了 LangGraph 还是 Dify”，而是状态机和业务边界。比如低置信时不能继续生成确定结论，只能输出缺失信息和检查项；电梯、消防、强电这类高风险设备要进入人工复核节点；检索不到 SOP 时要提示知识缺口，而不是编造答案。研发可以用现有 AI Engine、LangGraph 这类有状态编排，或其他内部方案实现，但产品侧必须先把节点、条件、异常和验收定义清楚。

**Q35：Workflow 编排和 Agent 分工有什么区别？**

> Agent 分工回答“谁负责什么”，Workflow 编排回答“这些能力按什么顺序、什么条件、什么状态流转起来”。比如告警理解 Agent、诊断 Agent、工单 Agent 是角色分工；而“诊断低置信时回到补充信息，高风险时暂停等待主管确认，人工修改后再回写知识库”就是 Workflow 编排。
>
> 一句话：Agent 是能力单元，Workflow 是把能力单元放进业务闭环里的执行路径。

**Q36：你简历里写 REST API、视频流、第三方系统接入边界，怎么讲？**

> 我不会说自己做底层集成开发，我的产品职责是定义系统接入的产品边界和前置条件。比如 Inspector 要让 Agent 理解告警上下文，前提是设备台账、IoT/告警、历史工单、SOP、视频流或第三方系统数据能被统一关联到设备和空间对象上。
>
> 我会和研发对齐几类内容：接口能提供哪些字段、字段枚举和状态怎么定义、哪些数据实时拉取、哪些缓存、失败时怎么提示、客户侧系统是否需要适配器、视频流只是辅助查看还是能参与 AI 识别。对 AI 来说，最重要的是不要把“接入了系统”当成“数据可用”，而是要确认 Agent 能否用这些数据完成诊断和工单闭环。

**Q37：SaaS / 私有化、多租户、权限安全怎么讲？**

> B 端 AI 产品不能只考虑模型效果，还要考虑部署和数据边界。简历里写 SaaS/私有化，是因为不同客户对数据和系统集成要求不同。设施运维场景里，客户的设备台账、告警、工单、SOP、视频流都可能是敏感数据，所以产品上要考虑客户/租户隔离、角色权限、工具调用日志、人工确认和审计记录。
>
> 对 Agent 来说，这意味着它只能访问当前客户、当前权限范围内的知识和数据；高风险操作不能自动执行；输出要能追溯引用来源和人工修改记录。

**Q38：简历里有竞品分析、上线复盘，面试要怎么讲？**

> 这两个点不用主动展开，但被问到可以讲成产品基本功。竞品分析不是简单看页面，而是对照客户流程、模块边界、数据对象和交付方式，判断哪些能力应该做成标品，哪些只是客户定制。上线复盘则是看版本是否达到验收标准、客户/一线反馈集中在哪些 bad case、哪些问题进入下一版 Roadmap。
>
> 和 AI 产品结合时，复盘重点会包括：检索命中率、人工修改率、工单字段完整率、低置信兜底是否触发、高风险人审是否有效。

**Q39：Designer / 工业仿真项目如果被追问，和这个岗位有什么关系？**

> Designer 不是这次主故事，但可以作为“AI 所需数据和空间语义底座”的补充。它证明我不只是理解 Agent，也理解数字孪生里的空间对象、设备关系、仿真记录、3D 可视化和数据闭环。比如 Plant Simulation 对接、运行路径、热力图、仿真结果回顾，本质上都是把工业现场转成可计算、可解释、可复盘的数据对象。后续 Agent 要做诊断和推荐，也需要这种结构化上下文。

**Q40：技术学习能力怎么证明？不会深入写代码会不会影响？**

> 我不会把自己包装成开发或算法工程师。我的技术能力主要体现在能读懂产品相关的技术约束，比如接口文档、字段表、状态机、数据接入边界、AI 输出结构和验收日志。简历里提到 Unity、C# 基础阅读、Cursor/Codex，是为了说明我有快速理解技术系统和做原型验证的能力。
>
> 如果研发说做不了，我会先判断是技术不可行、数据不具备、范围太大，还是当前版本资源不够；然后把需求拆成 MVP、降级方案和后续迭代，而不是简单坚持原方案。

## 9. 不能踩的坑

- 不要说“我做过 FMEA 智能体”。更稳：我做过告警、工单、SOP、知识沉淀闭环，和 FMEA 智能体的日常运维落地层很接近。

- 不要说“AI 自动派单/自动控制设备”。更稳：AI 生成草稿和建议，高风险必须人工确认。
- 不要把 Demo-ready 说成 Product-ready。更稳：MVP/POC 验证，沉淀为可复用底座。
- 不要虚报准确率。没有样本、周期、口径就不要讲 99% 这类数字。
