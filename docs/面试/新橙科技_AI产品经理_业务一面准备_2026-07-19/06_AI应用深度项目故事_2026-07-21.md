# 新橙科技面试 · AI 应用深度项目故事

> 日期：2026-07-21
> 用途：回答“你在项目中具体怎么做 AI 应用、怎么拆业务流、怎么定义 Agent、怎么和研发配合、怎么验收和调优”。
> 口径边界：这段讲的是产品侧方案验证 / Demo / POC / 可交付能力规划，不包装成模型训练或大规模全量上线。

---

## 1. 一句话定位

> 我不是把 AI 当成一个功能点，而是把它当成一条业务链路里的协作角色。它要有知识源、有依据、有边界、有人工复核，也要能从真实反馈里持续变好。

---

## 2. 面试可直接讲的完整故事

我做 Inspector 里 AI 应用的时候，其实一开始没有先想“我要做一个 Agent”。我当时先看的，是设施运维人员真实的一天。

客户现场已经有告警系统、设备台账、工单系统、SOP、历史维修记录，但问题是这些东西是散的。比如一个空调机组报警，系统能告诉你“温度异常”或者“设备离线”，但不会告诉你为什么异常、过去有没有类似问题、应该先查哪个部件、要不要派工单。一线人员还是要自己翻台账、问老师傅、查历史工单、看 SOP。于是我意识到，这个 AI 能力不能做成一个孤立聊天框，而应该嵌进“告警到工单”的业务主流程里。

我先把业务流拆成几步：告警触发以后，系统先识别告警对象是谁，在哪个空间，属于哪个设备系统；然后补上下文，包括设备台账、空间关系、历史工单、SOP、实时或历史数据；再让 AI 生成可能原因、判断依据、置信度、缺失信息和建议检查项；如果用户认可，再生成工单草稿；最后由人工确认、执行、回写结果，并把高价值案例沉淀回知识库。

所以我的第一步不是定义 Agent，而是定义业务闭环：

```text
告警
-> 上下文补全
-> 诊断建议
-> 处置方案
-> 工单草稿
-> 人工确认
-> 执行回写
-> 知识沉淀
```

然后我再反过来拆 Agent。我的理解里，Agent 不是一个万能机器人，而是业务链路里不同职责的智能模块。比如告警理解 Agent 负责把原始告警转成结构化问题；RAG / 知识检索模块负责找 SOP、历史工单、设备资料；诊断 Agent 负责输出可能原因和依据；方案生成 Agent 负责给检查步骤；工单生成 Agent 只生成草稿，不自动派发。

我给每个 Agent 都定义了四件事：输入是什么、输出是什么、什么时候触发、边界在哪里。比如诊断 Agent 的输入不是一句“帮我看看这个告警”，而是告警 ID、设备 ID、空间位置、告警等级、历史曲线、历史工单、SOP 检索结果。它的输出也不能是一句“可能是水阀问题”，而要包含可能原因、判断依据、置信度、缺失信息、建议动作和风险提示。

这也是我跟研发配合最核心的地方。我不会跟研发说“我们做个智能诊断吧”，这句话太虚了，研发也很难落地。我会把它拆成字段、接口、状态和验收脚本。比如我会问研发：设备 ID 能不能稳定拿到？SOP 有没有结构化？历史工单里有没有真实原因字段？AI 输出能不能按 JSON 返回？低置信时怎么走分支？没有检索到 SOP 时能不能强制不生成确定结论？

当时我们也遇到过一个很典型的问题：大家很容易把 AI 想成“给一个完整答案”。但现场运维不是这样工作的。一线人员不一定需要 AI 直接告诉他“就是某个部件坏了”，他更需要的是：我应该先查什么？为什么先查这个？有没有历史类似案例？这个建议有多可靠？哪些地方还缺信息？所以我后来把输出从“结论型”调成“排查型 + 依据型”。

比如原来可能是：

> 可能是冷却水阀异常，建议维修。

这就太粗了。调优后我希望它变成：

> 可能原因 1：冷却水阀异常。依据：当前温度异常与冷却水流量下降同时出现；历史工单中同区域出现过类似情况。置信度：中。缺失信息：现场阀门开度和管路压力。建议动作：先检查阀门开度、过滤器和管路压力，确认后再生成工单草稿。

这样一线人员会更容易信任，因为它不是在“替我决定”，而是在帮我组织排查路径。

业务目标我也拆过。对客户来说，目标不是“用了 AI”，而是告警处理更快、经验能沉淀、新人也能按流程排查。对产品来说，目标是把一次性的客户方案沉淀成可复用能力。对研发来说，目标是先跑通一个最小闭环，而不是一口气覆盖所有设备、所有故障、所有 SOP。

所以我会把目标拆成三层。第一层是 MVP 闭环：选一个典型设备或告警类型，跑通“告警识别 - SOP/历史工单检索 - 建议输出 - 工单草稿”。第二层是可信机制：引用、置信度、缺失信息、人工确认。第三层才是规模化：更多设备类型、更多客户知识库、更多历史案例回流。

至于怎么判断 AI 成果是否符合预期，如果还没有稳定线上指标，我不会硬讲准确率。我会用场景验收来判断。比如一个典型告警来了以后，它有没有识别对设备？有没有召回正确 SOP？有没有引用历史工单？输出字段是不是完整？低置信时有没有兜底？高风险场景有没有要求人工确认？工单草稿能不能被运维人员少量修改后使用？

我觉得 AI 产品早期最重要的不是“回答漂亮”，而是“失败得可控”。如果它不知道，就要说不知道；如果依据不够，就要提示缺什么；如果是高风险设备，就不能自动执行。

调优上，如果没有量化指标，我会主要看用户反馈和 bad case。我会把反馈拆成几类。第一类是输出太泛，比如用户说“这个建议我没法直接用”。那我会判断是不是输出结构太松，要改成固定检查项和工单字段。第二类是依据不足，比如用户问“为什么这么判断”。那说明要补引用来源、知识源优先级和可解释展示。第三类是现场规则不匹配，比如客户说“我们不是这么处理的”。那就要区分通用知识库和客户专属知识库，不能把所有客户经验混到一起。第四类是风险边界，比如电梯、消防、强电这类场景，AI 只能辅助判断，必须加人工确认或专家升级。

我指导研发调优时，也不会笼统说“模型不准”。我会先把 bad case 分类：是数据缺失、检索没召回、召回了但引用错、Prompt 越界、输出 Schema 不稳定，还是产品流程没有让用户补充必要信息。不同问题对应不同解法。知识缺失就补知识库；检索不准就调检索策略和标签；输出不稳定就收紧 Schema；用户不信任就增加引用和置信度；风险太高就加人工确认。

整个过程中最大的困难有两个。第一个是业务信息太散。没有设备、空间、SOP、历史工单这些上下文，AI 很容易变成泛泛建议。我的解决方式是先建业务对象和数据前置清单，明确哪些数据是 MVP 必须有的，哪些可以后补。第二个是大家对 AI 的期待容易过高，觉得 AI 应该一步到位给结论。我做的取舍是先不追求自动决策，而是先做“辅助诊断 + 工单草稿 + 人工确认”，让它在真实流程里可用、可控。

所以如果面试官问我这段经历，我不会说我训练了模型，也不会说 AI 已经完全替代人工。我会说，我做的是 AI 产品经理应该负责的那部分：把业务问题拆成流程，把流程拆成 Agent 职责，把 Agent 拆成输入输出和边界，再用场景验收、用户反馈和 bad case 迭代，让 AI 从一个概念变成研发能实现、客户能理解、现场敢用的产品闭环。

## 3. 90 秒压缩版

> 在 Inspector 的 AI 应用里，我一开始没有先定义“做 Agent”，而是先拆设施运维人员的真实业务流。客户的问题不是没有告警，而是告警之后还要人工查设备台账、历史工单、SOP 和现场数据，才能判断原因和处理路径。
>
> 所以我把 AI 放进“告警到工单”的主流程里：告警触发后，先识别设备和空间，再补设备台账、历史工单、SOP、时序数据这些上下文，然后输出可能原因、依据、置信度、缺失信息和建议检查项，最后生成工单草稿，由人工确认后派发。
>
> 我定义 Agent 时，不把它当万能机器人，而是拆成告警理解、知识检索、诊断、方案生成和工单草稿几个职责。每个职责都要说明输入是什么、输出是什么、边界在哪里。跟研发协作时，我会把它转成字段、接口、状态和验收脚本，比如设备 ID 能不能拿到、SOP 有没有结构化、低置信时怎么兜底、没有依据时能不能拒答。
>
> 验收上我不看 AI 回答是否漂亮，而看它是否识别对设备、是否召回正确依据、输出字段是否完整、低置信是否兜底、高风险是否人工确认。调优时我会把用户反馈和 bad case 分成知识缺失、检索不准、输出太泛、Schema 不稳、流程缺输入几类，再对应补知识库、调检索、收紧输出结构或增加人工确认。
>
> 所以这段经历的核心不是模型训练，而是我把 AI 能力从概念拆成研发可实现、测试可验收、客户能理解的产品闭环。

## 4. 追问防守

### Q1：你具体输出了什么？

> 我输出的不是一句“做智能诊断”，而是一组产品协议：业务流程图、业务对象清单、Agent 职责说明、输入输出字段、输出结构、异常分支、人机边界和验收脚本。这样研发能讨论接口和状态，测试能写用例，客户也能理解这个能力到底怎么用。

### Q2：怎么证明这个 AI 符合预期？

> 如果还在方案验证 / POC 阶段，我不会硬讲线上准确率，而是用场景验收：设备识别是否正确、SOP/历史工单是否召回、依据是否可解释、输出字段是否完整、低置信是否兜底、高风险是否人工确认、工单草稿是否能被少量修改后使用。

### Q3：如果用户反馈“AI 建议太泛”，你怎么处理？

> 我会先判断是任务定义太大、知识源不足，还是输出结构太松。如果是输出太松，就把答案改成固定结构，比如可能原因、依据、缺失信息、建议检查项和工单字段；如果是知识源不足，就补 SOP、历史工单和客户专属经验库；如果是场景太大，就收敛到某一类设备或某一类告警先跑通。

### Q4：如果用户不信任 AI 输出，怎么处理？

> 信任不是靠一句“AI 很准”，而是靠产品机制。我要让用户看到依据、引用来源、置信度、缺失信息和人工确认入口。尤其在高风险设备上，AI 只能生成建议或草稿，不能自动派发或替代人做决策。

### Q5：怎么指导研发调优？

> 我不会笼统说“模型不准”，而是把 bad case 分类：知识缺失、检索错误、引用错误、Prompt 越界、输出 Schema 不稳定、流程缺少必要输入。不同原因对应不同动作：补知识库、调检索策略、收紧输出协议、增加缺失信息提示，或加入人工确认。

## 5. 还需要补真实细节

这段故事已经能讲，但要想更真，后续需要补一条真实反馈。可以从下面任选一类补：

- 客户 / 内部评审是否说过“AI 输出太泛”？
- 是否有人质疑“为什么这么判断，依据在哪里”？
- 是否遇到过工单草稿字段不符合现场习惯？
- 是否有高风险设备场景被要求必须人工确认？
- 是否有客户自己的 SOP / 历史经验和通用知识不一致？

补到真实反馈后，这段故事可以继续升级成 STAR 版本。
