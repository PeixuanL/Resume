# AI 产品全流程补强与面试口径

> 用途：把 `~/Desktop/AI-Coding-Learning` 的学习内容，转成 B 端 AI 产品经理面试可讲的知识框架、项目闭环和追问回答。  
> 边界：可说「方案验证 / Demo / 原型闭环 / 产品侧全流程设计」，不说「已在生产产品中完整上线」。

---

## 一、总口径

面试时先把边界讲清楚，再把专业度讲满：

> 这块 AI 能力目前在公司产品里还处在方案验证和 Demo/POC 设计阶段，没有大规模生产上线。我参与和补全的是产品侧全流程：从业务场景拆解、Agent 分工、RAG 知识源、Prompt 业务协议、输出 Schema、低置信兜底、人工确认，到 bad case 验收和反馈回流。  
> 所以我不会把自己包装成模型训练或算法负责人，但我确实按 AI 产品落地的完整流程走过一遍，能和研发、交付、客户把边界和验收说清楚。

这套说法的优势：

- 不承诺「已上线」，避免被问线上指标时穿帮。
- 但强调「走过完整产品流程」，符合 AI PM 的能力要求。
- 把 AI 能力落在业务闭环、协议、验收和风险治理上，而不是停留在概念。

## 二、你需要补成能讲的 AI 知识地图

### 1. LLM / Prompt：Prompt 是业务协议

不要把 Prompt 说成「写提示词」，要说成「把业务系统翻译成模型可理解、可约束、可验证的协议」。

你要能讲 6 层：

| 层级 | 面试解释 | Inspector 对应 |
|------|----------|----------------|
| Role | AI 是什么角色，只建议不执行 | 告警诊断助手 |
| World Model | 业务对象、状态、关系 | 空间、设备、告警、时序、工单、SOP |
| Fact Priority | 数据冲突时信谁 | 实时告警 > 工单备注 > 历史归档 |
| Planning Rules | 内部诊断步骤 | 解析告警 -> 查设备 -> 检索 SOP -> 生成建议 |
| Output Contract | 输出必须机器可解析 | JSON、错误码、warnings、citations |
| Guardrails | 禁止事项 | 不编造设备 ID，不自动执行高风险动作 |

60 秒话术：

> 我理解的 Prompt 不是一句 system message，而是一份业务协议。尤其在 B 端运维场景里，模型必须知道自己负责什么、不负责什么，能相信哪些数据，冲突时信谁，输出什么结构，失败时返回什么错误码。比如 Inspector 里，AI 只能输出诊断建议和工单草稿，不能直接派单或下发控制指令。

### 2. RAG：不是上传文档，而是知识源治理

你要能说出 RAG 查什么，以及为什么要分层：

| 知识源 | 作用 |
|--------|------|
| 设备台账 | 设备型号、位置、责任人、维保周期 |
| 空间 / BrickSchema | 设备上下游、空间影响范围 |
| 实时告警 / 时序数据 | 判断当前状态和异常趋势 |
| 历史工单 | 相似故障、真实原因、处理动作 |
| SOP / 手册 | 标准处置步骤和安全注意事项 |
| 人工审结案例 | 高价值经验沉淀，避免错误工单污染知识库 |

常见坑：

- 整份 PDF 直接入库，检索噪音大。
- 未审结工单直接入库，错误经验污染。
- 没有引用来源，诊断结论不可追溯。
- 只做语义检索，不绑定告警对象和设备 ID。

60 秒话术：

> 我设计 RAG 时不会把它理解成“上传资料给模型看”，而是先做知识源分层。告警诊断要绑定设备、空间、时序、历史工单和 SOP。生成时必须带 citations，如果没有召回到有效 SOP 或历史案例，就不能输出确定结论，只能提示缺失信息或转人工。

### 3. Agent：多步工作流，不是聊天框

Inspector 的 Agent 可以这样拆：

| Agent / 模块 | 输入 | 输出 |
|--------------|------|------|
| 告警理解 | 原始告警、设备 ID、空间 | 结构化问题 |
| 上下文检索 / RAG | 设备、告警类型、时间范围 | SOP、历史工单、台账片段 |
| 诊断推理 | 上下文、时序、规则 | 根因假设、依据、置信度 |
| 方案生成 | 诊断结果、SOP | 检查步骤、注意事项 |
| 工单草稿 | 方案、设备、处理人规则 | 工单标题、描述、步骤 |
| 反馈沉淀 | 人工修正、执行结果 | bad case、知识库更新建议 |

关键边界：

- Agent 可以生成建议，不直接执行。
- 高风险动作必须人工确认。
- 工单是草稿，不是自动派发。
- 反馈回流必须人工审结后入库。

### 4. Schema / 白名单 / 置信度：把幻觉挡在系统外

这部分是你区分普通 AI 使用者和 AI PM 的核心。

| 层 | 问题 | 拦什么 |
|----|------|--------|
| Schema 校验 | 格式对不对 | JSON 字段缺失、类型错误、Markdown 混入 |
| 对象 / 动作白名单 | 系统里有没有、能不能做 | 不存在的设备 ID、未授权动作 |
| 置信度分支 | 敢不敢给完整建议 | 无 SOP、数据缺失、多候选、低相似度 |
| 人工确认 | 能不能进入业务系统 | 派单、关单、事故报告、停启设备 |

30 秒话术：

> 模型只输出结构化 JSON。先用 Schema 保证可解析，再用对象和动作白名单保证设备、SOP、工单类型真实存在，最后按置信度分流：高置信也只到人工确认，低置信只展示依据和缺失信息，不能进入一键采纳。

### 5. 指标与验收：不只说准确率

方案验证阶段可以讲「场景脚本 + bad case + 人工验收」，不要硬讲线上指标。

| 类型 | 指标 |
|------|------|
| 结构化输出 | JSON 解析成功率、Schema 合规率 |
| RAG 质量 | 检索命中率、引用正确率、无依据回答比例 |
| 产品体验 | 建议采纳率、人工修改率、转人工率 |
| 风险控制 | 低置信兜底触发率、高风险人工确认覆盖率 |
| 业务效果 | 告警响应时长、工单创建耗时、一次解决率 |
| 运营迭代 | bad case 收敛速度、回归通过率 |

验证口径：

> 在没有完整线上数据时，我不会包装模型准确率，而是用场景脚本验收。比如准备 20-30 条标准告警，覆盖设备不存在、SOP 缺失、同名设备、时序缺失、告警已恢复等 bad case，看系统是否按预期失败、兜底或转人工。

## 三、项目故事：Inspector AI Agent 方案验证闭环

### 1. 30 秒版

> Inspector 这段我会讲成设施运维 AI Agent 的方案验证。真实业务里，我们已经有告警、工单、资产和 3D 沙盘链路，问题是告警只能告诉用户“出问题了”，但不能解释原因和下一步动作。  
> 我的工作是把 AI 嵌进这条链路：先定义告警、设备、时序、历史工单和 SOP 的世界模型，再设计 RAG 检索、诊断建议、工单草稿、Schema 校验、低置信兜底和人工确认。它目前没有说成生产全量上线，而是作为产品侧 AI 闭环设计和验证方案。

### 2. 2 分钟 STAR 版

**S 背景**

设施运维客户已经有 BMS/SCADA/IoT 告警，但告警本身只描述表象。一线人员还要查设备台账、历史工单、SOP、图纸和现场数据，才能判断原因和处置动作。项目制交付也容易高度定制，难沉淀可复用能力。

**T 任务**

我的目标不是做一个聊天框，而是把 AI 放进告警到工单的业务主链路，形成一个可验证、可兜底、可回流的诊断辅助闭环。

**A 行动**

我按产品侧全流程拆了几层：

1. 梳理世界模型：空间、设备、告警、时序、工单、SOP，以及对象之间的关系。
2. 定义 RAG 知识源：设备台账、BrickSchema、实时/历史告警、历史工单、SOP、人工审结案例。
3. 设计 Agent 分工：告警理解、上下文检索、诊断推理、方案生成、工单草稿、反馈沉淀。
4. 设计输出协议：结论、依据、置信度、warnings、缺失信息、建议动作，必须结构化输出。
5. 设计风险控制：Schema 校验、对象/动作白名单、低置信兜底、高风险人工确认。
6. 设计验收方式：用场景脚本和 bad case 表验证，不只看回答是否流畅。

**R 结果**

这套方案把 AI 能力从概念转成了研发和交付能讨论的产品协议：哪些数据要前置、哪些接口要提供、什么情况必须失败、什么情况能生成工单草稿。它支撑了 Inspector 从告警展示、工单流转走向诊断辅助闭环的产品化方向。商业结果仍主要归因于标品化底座和交付流程提效，AI 这部分我会表述为方案验证和能力规划，不夸大成已全量上线。

## 四、Vibe Coding 口径：从“没跑通”改成“正在补全方法论”

### 1. 面试边界

可以说：

> 我正在用 Vibe Coding 补一个端到端小项目，把它作为 AI PM 的工程协作训练：从 PRD、技术方案、任务拆解、结构化 Prompt、开发验证、测试门禁到文档回写。现在它更适合作为我的个人训练和原型验证方法，不会包装成公司生产项目成果。

不要说：

- 我已经用 Vibe Coding 交付了公司生产功能。
- 我能独立负责前后端生产级实现。
- Cursor/Codex 替代了研发。

### 2. 8 步方法论

| 步骤 | 你要讲什么 |
|------|------------|
| Define | 先写目标、非目标、验收标准 |
| Design | 再做技术方案、接口、数据结构、异常路径 |
| Decompose | 拆成 30-90 分钟可闭环任务 |
| Drive | 用角色、目标、上下文、边界、输出、验收驱动 AI |
| Develop | 小步开发，每步可运行 |
| Defend | 防幻觉：事实/推断/待确认分开 |
| Debug/Test | 先缺陷分析，再修复，再回归 |
| Document/Deploy | README、变更记录、复盘、可复现交付 |

### 3. 60 秒话术

> 我理解的 Vibe Coding 不是让 AI 随便写代码，而是“AI 执行、人类决策”的工程协作流程。我会先定义 PRD 和非目标，再产出技术设计，把任务拆成小闭环；执行时用结构化 Prompt，要求先方案后实施。质量上用测试、冒烟、缺陷分析和文档回写做门禁。现在我正在用这个流程补一个小项目，目标是让自己不仅能写 AI 产品需求，也能理解研发协作、验证和交付风险。

## 五、面试追问：稳答版本

### Q1：这个 AI Agent 真实上线了吗？

> 还没有包装成大规模生产上线。我更准确的说法是：这块处在产品方案验证和 Demo/POC 设计阶段。我做的是产品侧完整闭环设计，包括业务流程、数据前置、Agent 分工、RAG、输出协议、验收和兜底。它不是模型训练成果，但已经不是停留在“想法”，而是可以进入研发评审和场景验证的产品方案。

### Q2：你具体做过什么，不是只学了概念吗？

> 我做的不是算法实现，而是 AI 产品经理应该负责的协议和闭环：  
> 第一，把告警、设备、工单、SOP 抽成世界模型；  
> 第二，定义 RAG 知识源和事实优先级；  
> 第三，拆 Agent 在业务链路中的职责边界；  
> 第四，设计输出 Schema、错误码、warnings 和人工确认；  
> 第五，用 bad case 表做验收，比如设备不存在、SOP 缺失、告警已恢复、同名设备、多候选等。

### Q3：如果模型幻觉怎么办？

> 我不会只靠 Prompt 解决幻觉，而是做四层机制：Schema 保证格式，白名单保证设备和动作真实存在，置信度分支控制展示路径，人工确认挡住高风险执行。比如模型给出不存在的设备 ID，会在对象白名单层被拦截；没有召回 SOP 却给确定步骤，会被强制降置信并转人工。

### Q4：RAG 怎么做评估？

> 我会分两层评估。检索层看是否命中正确 SOP、历史工单和设备资料，引用是否可追溯；生成层看结论是否基于引用、是否列出缺失信息、是否在低置信时拒绝确定回答。早期用 20-30 条黄金场景和 bad case 做脚本验收，进入上线后再看建议采纳率、人工修改率、工单一次解决率等业务指标。

### Q5：你怎么和研发对齐？

> 我不会只说“做个 Agent”，而是给研发明确协议：输入有哪些字段，输出 JSON 长什么样，错误码有哪些，哪些动作必须白名单，低置信阈值怎么配置，日志要记录哪些 trace，发版时评测集怎么回归。这样研发讨论的是协议、接口和验收，不是抽象地争论模型聪不聪明。

### Q6：你怎么区别 AI PM 和算法 / 工程职责？

> 算法和工程负责模型、检索、服务、接口实现；AI PM 负责把业务问题定义成可验证的 AI 能力。我会定义场景边界、数据源优先级、人机分工、输出协议、验收指标和风险兜底。我的价值是保证 AI 能安全地进入业务流程，而不是只生成一个看起来很聪明的回答。

### Q7：你现在 Vibe Coding 到什么程度？

> 我不会说它已经是公司生产成果。更准确地说，我正在把它作为个人 AI PM 工程协作训练：按 PRD、设计、拆任务、结构化 Prompt、测试、文档回写跑小项目。我的目标是能和研发用同一套工程语言沟通，并能判断 AI 生成内容是否越界、是否可验证、是否需要回滚。
