# 设施运维 Agent 闭环项目示例：面试复盘稿

> 用途：用于准备 AI 产品经理面试中的 B 端 AI Agent 项目表达。  
> 说明：本文是基于设施运维 / 预测性维护业务逻辑整理的高真实度模拟口径。正式面试时，应替换为自己的真实客户、设备、数据源、指标和交付边界，避免把未实际发生的细节说成事实。

---

## 1. 一句话项目定位

我做的不是一个单独的 AI 聊天助手，而是一个嵌入设施运维流程的异常诊断与工单处置 Agent 闭环。

它的核心价值是：把原来“看到告警后靠人工查数据、翻 SOP、问专家、再手动建工单”的流程，变成“系统基于告警、设备数据、历史工单、SOP 和资源信息，辅助完成诊断、方案生成、资源匹配、工单草稿、人工确认、执行回写和知识沉淀”的闭环。

---

## 2. 业务场景

场景假设是一个园区或工厂的中央空调系统，客户重点关注洁净车间的温湿度稳定性。

某天上午 9:17，系统监测到 3 号洁净车间的 AHU-03 空调机组送风温度连续 15 分钟高于阈值，从 22℃ 升到 26.8℃。

在传统流程里，告警系统只能提示：

```text
AHU-03 送风温度过高
```

但这条告警并不能回答：

```text
为什么温度过高？
是冷源问题、阀门问题、风机问题、过滤网问题，还是传感器误报？
是否影响生产区域？
需要谁去处理？
需要准备什么备件？
按照什么步骤处理？
处理完成后怎么验证恢复正常？
```

因此，我把产品目标定义为：让 Agent 把告警从一个“异常提示”，转化成一个“可诊断、可派工、可执行、可复盘”的运维闭环。

---

## 3. 端到端闭环流程

```text
告警触发
→ 异常识别与降噪
→ 诊断分析
→ 处置方案生成
→ 资源调度
→ 工单草稿生成
→ 人工确认与派发
→ 工单执行
→ 结果回写
→ 知识沉淀
```

---

## 4. 用户每一步看到什么，背后调了什么能力

| 步骤 | 用户看到的结果 / 交互 | 背后调用的 Agent 或能力 | 产品设计要点 |
|---|---|---|---|
| 1. 告警触发 | 运维看板出现“AHU-03 送风温度持续升高”的异常提醒 | 监控系统、规则引擎、时序数据采集 | 告警是原始信号，不直接等同于故障；需要记录设备、指标、阈值、持续时间、来源 |
| 2. 异常识别与降噪 | 多条温度波动告警被合并成一个“异常事件卡片” | 告警归并规则、异常检测、重复告警过滤 | 24x7 监控不宜每条告警都进大模型，先用确定性规则减少噪音 |
| 3. 异常事件卡片 | 页面显示异常对象、影响区域、风险等级、建议响应时限 | 设备台账查询、空间/区域映射、风险分级规则 | 用户第一眼要知道“哪里出问题、影响谁、急不急” |
| 4. 点击查看诊断 | 系统展示可能根因、置信度、证据和排除项 | 诊断 Agent、设备数据查询工具、历史工单检索、知识库 RAG | AI 输出必须可解释，不能只给一句结论 |
| 5. 查看诊断证据 | 用户看到“阀门指令 80%，反馈 35%”“冷冻水供水温度正常”等证据卡片 | 时序数据对比、规则判断、相似案例检索 | 现场人员更相信证据，而不是相信“AI 说” |
| 6. 生成处置方案 | 用户点击“生成处置方案”，系统输出维修步骤、备件、工具、安全注意事项 | 方案 Agent、SOP 检索、设备手册检索、历史案例总结 | 方案要可执行，不能只是泛泛建议 |
| 7. 资源调度 | 系统推荐执行人、备件位置和建议派工时间 | 资源调度 Agent、人员排班、技能标签、备件库存 | 调度类信息优先走确定性规则，Agent 负责解释和编排 |
| 8. 生成工单草稿 | 系统自动生成标题、设备、故障描述、步骤、备件、验收标准 | 工单生成 Agent、字段映射、工单系统 API | 输出必须结构化，能写入工单系统 |
| 9. 人工确认 | 运维主管确认或修改工单后再派发 | 审批流、权限控制、风险规则 | 高风险动作不能自动执行，必须保留人工确认 |
| 10. 工单执行 | 维修人员在移动端按步骤勾选执行，上传结果 | 移动工单、SOP 步骤、现场反馈 | 工单不仅要派出去，还要能指导现场执行 |
| 11. 结果回写 | 执行人填写实际根因、处置方式、耗时、备件消耗 | 工单回写、设备履历更新、知识库更新 | 闭环的关键是结果回写，否则 Agent 无法持续优化 |
| 12. 知识沉淀 | 新案例进入知识库，后续类似问题可被检索引用 | 知识沉淀 Agent、案例结构化、人工审核 | 知识入库前要有人审，避免错误经验被长期复用 |

## 5. 一个完整的用户体验示例

### 5.1 告警进入系统

上午 9:17，运维看板上出现一条异常事件：

```text
异常对象：AHU-03 空调机组
异常现象：送风温度持续升高
影响区域：3 号洁净车间
风险等级：中高
建议响应时限：30 分钟内确认
```

用户不再只看到一条孤立的“温度过高”告警，而是看到一个已经归并后的异常事件。

这里背后的产品设计是：告警先经过归并和降噪。比如同一设备、同一指标、同一时间窗口内的多条告警会被合并成一个异常事件，短时间的瞬时波动不会立刻触发 Agent 诊断。

### 5.2 Agent 进行诊断

用户点击“查看诊断”，系统开始拉取上下文。

Agent 调用的数据包括：

```text
送风温度、回风温度、冷冻水供水温度
冷冻水阀门指令开度和反馈开度
风机频率
设备台账
历史维修工单
SOP
设备手册
```

诊断结果展示为：

```text
初步判断：冷冻水阀门执行器卡滞可能性较高
置信度：78%

核心证据：
1. 系统下发阀门开度 80%，但实际反馈长期停留在 35%
2. 冷冻水供水温度正常，排除冷源不足的可能性
3. 送风温度升高与阀门反馈异常时间高度一致
4. 近 30 天内该阀门曾出现两次动作延迟告警

排除项：
1. 冷源供水温度正常，暂不优先判断为冷源故障
2. 风机频率稳定，暂不优先判断为风机异常
3. 温度传感器与相邻点位趋势一致，传感器误报可能性较低
```

这里我在 PRD 里要求 Agent 输出“结论 + 证据 + 排除项 + 置信度”，因为客户现场不会因为 AI 给了结论就直接相信，必须知道它为什么这么判断。

### 5.3 生成处置方案

用户点击“生成处置方案”，方案 Agent 基于诊断结论、SOP 和历史案例生成建议：

```text
建议处置：
1. 到 AHU-03 现场检查冷冻水阀门执行器供电状态
2. 检查执行器反馈信号是否异常
3. 手动切换阀门开度，确认是否存在机械卡滞
4. 如执行器无响应，建议更换同型号执行器
5. 更换后重新校准阀门开度反馈
6. 观察送风温度是否在 20 分钟内恢复至 22-24℃

所需备件：冷冻水阀门执行器 1 个
所需工具：万用表、扳手、调试终端
安全注意：操作前确认该支路不影响核心生产区域
预计耗时：45-60 分钟
```

这里的产品重点是：方案不能只写“建议检查阀门”，而要能变成现场人员可执行的步骤，包括备件、工具、安全注意和验收方式。

### 5.4 资源调度与工单草稿

系统继续调用资源调度能力，查询人员排班、技能标签、当前位置和备件库存。

推荐结果是：

```text
推荐执行人：张工
推荐原因：具备暖通维修资质，当前无高优先级任务，距离现场最近
备件状态：仓库 A 有冷冻水阀门执行器库存 2 个
建议派工时间：立即派发
```

随后 Agent 生成工单草稿：

```text
工单标题：AHU-03 冷冻水阀门执行器疑似卡滞处理
关联设备：AHU-03
关联异常事件：送风温度持续升高
故障描述：送风温度超过阈值，阀门指令开度与反馈开度不一致
推荐处置步骤：引用 SOP-AC-VALVE-02
所需备件：冷冻水阀门执行器 1 个
推荐执行人：张工
优先级：P1
验收标准：送风温度恢复至 22-24℃，阀门反馈与指令偏差小于 5%
```

我在产品上没有让 Agent 直接自动派单，而是设置了人工确认。因为涉及现场维修、备件消耗和可能的生产影响，高风险动作必须由运维主管确认。

### 5.5 工单执行与回写

维修人员在移动端收到工单，按步骤执行：

```text
已检查供电
已检查信号反馈
已确认执行器卡滞
已更换执行器
已完成阀门校准
```

执行完成后，系统要求回写：

```text
实际根因：阀门执行器机械卡滞
实际处置：更换执行器并重新校准
实际耗时：52 分钟
是否解决：是
备件消耗：1 个
```

这些信息会进入设备履历、历史工单和知识库。后续再遇到类似问题，Agent 可以把这个案例作为相似案例引用。

## 6. 背后的产品设计拆解

### 6.1 为什么不是所有告警都直接调用大模型

设施运维监控是 24x7 场景，对稳定性和成本都敏感。如果每条告警都调用大模型，会带来三个问题：

```text
成本高
响应不稳定
误报和重复告警会放大噪音
```

所以我把能力分层：

```text
第一层：规则和时序算法做监控、阈值判断、异常检测、告警归并
第二层：Agent 在需要诊断和生成方案时介入
第三层：人工确认高风险动作
```

### 6.2 Agent 的职责边界

Agent 主要负责：

```text
整合上下文
解释异常原因
检索 SOP 和历史案例
生成处置建议
生成工单草稿
总结执行结果
沉淀知识
```

Agent 不负责：

```text
直接控制设备停机
直接执行高风险维修动作
绕过主管审批自动派发关键工单
在数据缺失时强行给确定结论
```

### 6.3 产品经理需要和研发对齐的内容

我会和研发重点对齐这些内容：

| 模块 | 需要对齐的内容 |
|---|---|
| 告警归并 | 时间窗口、设备维度、指标维度、重复告警过滤规则 |
| 数据查询 | 需要哪些时序数据、接口延迟、数据缺失时如何提示 |
| 诊断 Agent | 输入字段、Prompt 模板、工具调用顺序、输出 JSON 结构 |
| 知识库 RAG | 文档来源、分段方式、召回数量、引用来源展示 |
| 方案生成 | SOP 优先级、历史案例优先级、禁止生成的高风险建议 |
| 工单生成 | 工单字段映射、必填字段、默认值、人工可编辑字段 |
| 权限审批 | 哪些动作必须人工确认，哪些角色可以派单 |
| 日志审计 | Agent 调用了什么数据、给了什么建议、用户是否采纳 |
| 验收指标 | 诊断采纳率、工单完整率、平均响应时间、人工修改率 |

## 7. 客户反馈与产品迭代

### 反馈 1：告警太多，AI 分析反而更吵

第一版如果每条告警都触发 AI 诊断，客户会觉得系统更复杂。

调整后：

```text
增加告警归并
增加异常事件对象
只有满足持续时间、风险等级、影响范围条件时才触发诊断 Agent
```

### 反馈 2：AI 输出太长，现场人员没时间看

第一版 AI 输出偏自然语言，像一段报告。

调整后：

```text
改成“结论 + 证据 + 下一步动作”
证据用卡片展示
处置步骤按现场执行顺序排列
```

### 反馈 3：客户不希望 AI 自动派发高风险工单

客户担心 AI 误判导致错误维修或影响生产。

调整后：

```text
普通巡检建议可以生成工单草稿
涉及停机、切换系统、关键部件更换的动作必须主管确认
所有 Agent 建议保留审计记录
```

### 反馈 4：现场 SOP 和设备手册不完全一致

设备手册是通用的，但客户现场有自己的运维习惯。

调整后：

```text
知识优先级调整为：客户现场 SOP > 历史工单 > 设备手册 > 通用知识
Agent 输出必须标明引用来源
知识入库前需要人工审核
```

## 8. 可以直接背的面试讲法

我当时做这个设施运维 Agent 时，核心不是做一个聊天机器人，而是把 Agent 嵌入告警到工单的真实业务闭环里。

客户原来已经有告警系统，但告警只能告诉运维人员设备出现了异常表象，比如 AHU 空调机组送风温度过高，却不能告诉他根因是什么、影响范围是什么、需要什么备件、应该按什么步骤处理。所以我把这个能力设计成从告警触发开始，经过异常识别、诊断分析、方案生成、资源调度、工单草稿、人工确认、执行回写和知识沉淀的一整条链路。

在用户侧，运维人员看到的是一个异常事件卡片，而不是一堆孤立告警。点击诊断后，Agent 会结合实时运行数据、设备台账、历史工单和 SOP，输出可能根因、证据、排除项和置信度。比如系统判断冷冻水阀门执行器卡滞，是因为阀门指令开度和反馈开度长期不一致，同时冷冻水供水温度正常，可以排除冷源不足。之后方案 Agent 会生成处置步骤、备件、工具和安全注意事项，资源调度能力会推荐合适人员和备件位置，最后生成工单草稿，由主管确认后派发。

我和研发配合时，重点不是只写页面需求，而是对齐每一步的输入、输出、接口和边界。比如告警归并规则怎么定，Agent 能调用哪些数据源，诊断结果的 JSON 结构是什么，置信度低于多少要转人工，高风险动作哪些必须审批，工单字段怎么映射，执行结果如何回写到历史工单和知识库。

后面根据客户反馈，我们也做了几轮调整。比如客户觉得告警太多，所以我们先做异常事件归并，不是每条告警都触发 AI；客户觉得 AI 输出太长，我们就改成结论、证据、下一步动作三段式；客户不希望 AI 自动派高风险工单，我们就加了人工确认和权限控制；客户现场 SOP 和设备手册不一致，我们就把知识优先级调整为现场 SOP 优先。

所以我理解的工业 Agent 产品，不是让大模型替人做所有决定，而是在关键节点提供诊断、解释、推荐、编排和写回能力，把人的经验、设备数据和业务流程串成一个可执行、可审计、可持续优化的闭环。

## 9. 作为产品经理我是怎么推进的

如果面试官追问“你作为产品经理具体做了什么”，我不会只说自己写了 PRD，而是会从项目推进过程来讲。

### 9.1 先回到现场流程，而不是先接大模型

我推进的第一步不是让研发先接模型，而是先和运维主管、一线维修人员、交付同事一起梳理现有流程：

```text
告警从哪里来？
谁第一时间看到？
看到后怎么判断？
通常要查哪些系统？
哪些问题必须问老师傅？
工单怎么创建和派发？
维修结果有没有回写？
```

梳理后，我把客户原来的流程画成：

```text
告警出现 → 人工查数据 → 判断根因 → 找 SOP → 查备件 → 派工 → 执行 → 回写
```

这样做的目的是先确认真实痛点：客户不是缺一个聊天框，而是缺少从告警到诊断、处置和工单执行的闭环。

### 9.2 找到 AI 最适合介入的节点

我会把流程拆开判断，哪些地方适合 AI，哪些地方更适合规则或传统系统。

```text
适合规则 / 确定性系统：
1. 24x7 告警监控
2. 阈值判断
3. 告警归并
4. 权限校验
5. 备件库存查询
6. 人员排班查询

适合 Agent / AI：
1. 结合上下文做异常解释
2. 检索历史工单和 SOP
3. 生成诊断证据和排除项
4. 推荐处置方案
5. 生成工单草稿
6. 总结执行结果并沉淀知识
```

所以我没有把第一期定义成“AI 自动维修”，而是收敛成一个最小可交付闭环：

```text
告警归并 → 异常诊断 → 方案生成 → 工单草稿 → 人工确认 → 执行回写
```

### 9.3 拆 PRD，明确每个模块的输入、输出和边界

PRD 里我会把需求拆成几个模块：

```text
1. 异常事件卡片
2. 诊断 Agent
3. 处置方案 Agent
4. 资源调度能力
5. 工单草稿生成
6. 人工确认与权限
7. 执行结果回写
8. 知识沉淀
```

每个模块都要定义清楚输入、输出、状态和边界。比如诊断 Agent 的输入不是一句“帮我分析告警”，而是：

```text
告警信息
设备实时数据
设备台账
历史工单
SOP
设备手册
```

输出也不能是一段自由文本，而要是结构化字段：

```text
可能根因
诊断证据
排除项
置信度
建议下一步
引用来源
```

这样后续才能进入工单系统、审批系统和知识库。

### 9.4 和研发对齐：数据、能力分层、输出结构

和研发推进时，我重点对齐三件事。

第一，数据能不能拿到。

```text
告警系统是否有接口？
时序数据字段是否稳定？
设备台账是否能按设备 ID 查询？
历史工单是否能检索到故障类型和处置结果？
SOP 是否能做知识库切分和召回？
备件和人员排班是否能实时查询？
```

第二，哪些用规则，哪些用 Agent。

```text
告警归并、权限校验、备件库存、人员排班：优先用规则和接口查询
诊断解释、SOP 检索、处置建议、工单描述生成：使用 Agent / RAG
```

第三，Agent 输出必须结构化。

因为后面要写入工单系统，Agent 不能只返回一段自然语言，而要输出可解析字段，例如：

```text
root_cause
evidence_list
excluded_causes
confidence
recommended_steps
spare_parts
work_order_draft
source_references
```

### 9.5 用一个真实案例推动，而不是泛泛做平台能力

在推进开发时，我会选择一个典型故障场景作为第一期样例，例如 AHU 空调机组送风温度异常。

用这个案例把链路跑通：

```text
告警进来
→ 系统形成异常事件
→ Agent 给出根因和证据
→ 生成处置步骤
→ 生成工单草稿
→ 人确认
→ 执行结果回写
```

这样团队不会陷入“AI 能做很多事”的空泛讨论，而是先做出一个能讲清楚、能验证、能交付的小闭环。

### 9.6 客户验证与迭代

客户验证阶段，我会带着原型或 Demo 去和运维人员确认三件事：

```text
诊断结果像不像现场真实判断？
处置步骤能不能直接执行？
工单字段是否符合他们原来的工作习惯？
```

如果客户反馈 AI 输出太长，我会把结果改成：

```text
结论
证据
下一步动作
```

如果客户反馈告警太多，我会加强告警归并和触发条件。

如果客户担心 AI 自动派单有风险，我会增加人工确认和审批权限。

如果客户说 SOP 和现场习惯不一致，我会调整知识优先级，让客户现场 SOP 优先于通用设备手册。

### 9.7 用指标验收，而不是只说功能上线

最后验收时，我不会只说“功能上线了”，而是看几个指标：

```text
告警到工单草稿生成时间是否缩短
AI 诊断建议采纳率
工单字段完整率
人工修改率
执行结果回写率
相似案例复用率
```

这几个指标能证明 Agent 不是一个 Demo，而是真的进入了业务流程。

### 9.8 可以直接背的 PM 推进口径

我作为产品经理推进这个项目的方式，是先从客户真实运维流程出发，找到 AI 最适合介入的诊断、解释、推荐和工单生成节点；再把它拆成最小可交付闭环，定义清楚每一步的输入、输出、权限、人工确认和验收指标；最后通过真实案例和客户反馈不断收敛，而不是一开始就做一个泛泛的 AI 助手。

## 10. STAR 版压缩表达

### S：背景

客户已有设施运维告警系统，但告警只能提示设备异常表象，无法解释根因、影响范围、处置步骤和备件需求，运维人员需要依赖经验人工排查。

### T：任务

设计一个设施运维 Agent 闭环，将告警、设备数据、历史工单、SOP、备件库存和人员排班串联起来，辅助完成异常诊断、处置方案生成和工单闭环。

### A：行动

我把流程拆成告警触发、异常识别、诊断分析、方案生成、资源调度、工单草稿、人工确认、执行回写和知识沉淀。产品上定义了 Agent 的输入数据、结构化输出、工具调用、置信度、人工审批、工单字段映射和知识回写规则。同时根据客户反馈优化了告警归并、证据展示、权限控制和知识优先级。

### R：结果

这个设计让运维人员不再只是看到告警，而是能基于证据快速理解可能根因，并获得可执行的处置方案和工单草稿。项目价值体现在缩短告警响应时间、提升工单填写完整度、降低对老师傅经验的依赖，并通过执行回写持续沉淀知识。

## 11. 面试官希望听见的关键词

```text
不是聊天机器人，而是业务闭环
告警是症状，异常事件是诊断入口
规则和时序算法负责稳定监控，大模型负责解释、推荐和编排
AI 输出要结构化，可写入工单系统
高风险动作必须人工确认
诊断结果要有证据和引用来源
执行结果要回写，形成知识沉淀
产品经理要定义输入、输出、边界、权限、验收和兜底
```
