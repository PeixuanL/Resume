# STAR 项目故事：Inspector 与 Designer

> 用途：B 端 AI 产品经理投递、面试口述、简历项目经历补强。  
> 口径：以事实闭环为主，不夸大为“主导模型训练”或“AI 自动执行高风险操作”。

---

## 总体定位

这两段经历不要平均用力。面试时主打 **Inspector / AI 设施运维 Agent 闭环**，Designer / DFS 作为第二项目，用来证明你不仅理解 AI 概念，也理解数字孪生、数据接入、3D 场景、分析画布和交付闭环。

推荐总叙事：

> Inspector 更偏业务闭环和 Agent 产品化，回答“AI 怎么参与运维决策”；Designer / DFS 更偏数据底座和孪生场景，回答“AI 需要什么业务上下文和数据基础”。前者是应用层闭环，后者是数据与空间语义底座。两者合起来，就是我对 B 端工业 AI 产品的理解：不能只做模型输出，而要从数据接入、业务对象、流程状态、人机确认、执行回写到知识沉淀完整设计。

## 项目一：Inspector / AI 设施运维 Agent 闭环

### 一句话定位

我参与的是把设施运维从“告警展示 + 人工处理”升级成“AI 辅助诊断 + 工单闭环 + 知识沉淀”的 0 到 1 产品化过程。

### STAR 结构

#### S：背景

传统设施运维里，BMS、IoT、设备系统能产生大量告警，但告警往往只能告诉用户“出问题了”，不能解释“为什么、影响哪里、下一步怎么处理”。一线人员还要在设备台账、历史工单、SOP、维修手册、现场数据之间切换，处理效率高度依赖专家经验。

典型痛点：

- 单点告警无法解释根因。
- 设备台账、历史工单、SOP、时序数据割裂。
- 同类故障经验难沉淀，工单常只记录“已处理”。
- 复杂故障依赖专家，新人处理成本高。
- 管理侧缺少高频故障、成本归因和预防性维护依据。

#### T：任务

我的目标不是做一个独立 AI 聊天框，而是把 AI 嵌入运维主流程，设计一条可控闭环：

```text
告警触发
  -> AI 诊断
  -> 处置方案
  -> 工单草稿
  -> 人工确认
  -> 执行回写
  -> 知识沉淀
```

核心任务包括：

- 定义 Agent 在运维链路中的触发时机、输入、输出和边界。
- 设计 AI 与工单、知识库、设备数据、库存/排班等系统的协作关系。
- 明确哪些动作可以由 AI 辅助，哪些必须人工确认。
- 把能力拆成研发可实现、测试可验收、客户可理解的需求。

#### A：行动

**1. 场景拆解**

把运维链路拆成几个阶段：告警理解、上下文检索、诊断推理、资源调度、工单生成、执行反馈。

对应 Agent 分工：

- 告警理解 Agent：识别设备、空间、系统、告警类型、复发和并发关系。
- 诊断 Agent：结合设备时序数据、历史工单、知识库和 SOP 输出可能根因。
- 调度 Agent：查询备件、人员排班、维修窗口。
- 工单生成 Agent：把诊断结论和调度结果转成结构化工单草稿。

**2. AI 边界设计**

确定 AI 不直接替代人做高风险决策，而是输出：

- 诊断结论
- 判断依据
- 置信度
- 缺失信息
- 推荐 SOP
- 工单草稿

正式派发、库存扣减、高风险处置必须由运维经理或主管确认。

**3. 研发协作**

和研发对齐 Agent 调用链路与 API 能力，包括：

- 设备时序数据：获取振动、电流、温度、功率等运行数据。
- 历史工单：获取维修履历、上次处理时间、实际故障原因。
- 知识库 / RAG：检索 SOP、厂商手册、故障对照表。
- 库存与排班：查询备件、预占库存、匹配人员和维修窗口。
- 工单模板：拉取模板、生成草稿、人工确认后正式发布。
- 审计与权限：记录工具调用、用户身份、租户边界和操作结果。

**4. 产品验收**

验收标准不是只看“AI 能不能回答”，而是看闭环是否完整：

- 每一步是否有明确输入输出。
- 是否有状态、错误处理和加载反馈。
- AI 结论是否有依据和可解释性。
- 高风险动作是否有人审。
- 工单执行结果是否能回写。
- 反馈是否能沉淀为后续知识库。

#### R：结果

这套设计支撑了设施运维 Agent 底座的 0 到 1 验证，形成可复制的告警、工单、资产、3D 沙盘和 Agent 闭环能力。简历口径中可结合已有事实表达为：

- MVP 覆盖标杆客户 80%+ 高优诉求。
- 支撑 5 家+ 潜在商机。
- 运营商类项目实施周期从 5-6 个月压缩到 1-2 个月。
- 支撑新加坡数百万级方案交付。

注意：这些结果要作为“产品化和交付效率提升”的结果讲，不要说成单个 AI 功能独立创造全部商业结果。

### 30 秒口述版

> 在 Inspector 项目里，我负责把设施运维场景从“告警展示”升级为“AI 辅助诊断与工单闭环”。传统楼宇或数据中心运维里，系统只能提示温度、振动等异常，但一线人员还要人工查台账、历史工单和 SOP。我把链路拆成告警理解、RAG 检索、诊断推理、方案生成、工单草稿和人工确认几个环节，并和研发对齐设备数据、知识库、工单模板、权限审计等接口。最终形成了从告警到处置再到知识沉淀的闭环，而不是一个孤立的 AI 对话入口。

### 2 分钟展开版

> Inspector 是设施运维方向的产品。我们看到的核心问题不是客户没有告警，而是告警之后没有诊断和处置闭环。比如空调、冷机、电梯或数据中心设备出现异常时，系统只能显示温度高、振动异常、设备离线等表象，一线人员还要查台账、历史工单、SOP、设备手册和现场数据。
>
> 所以我在做产品设计时，没有把 AI 定义成一个独立聊天入口，而是嵌入告警到工单的主路径。我把流程拆成告警理解、上下文检索、诊断推理、处置方案、工单草稿、人工确认、执行回写和知识沉淀。AI 负责做信息整合和建议生成，人负责最终确认和派发。
>
> 和研发协作时，我重点明确每个 Agent 需要调用哪些业务能力：诊断 Agent 需要设备时序数据、历史维修记录和知识库；调度 Agent 需要库存、排班和维修窗口；工单 Agent 需要模板、SOP 和草稿保存接口。同时我也定义了风险边界，比如高风险设备不能自动派单，必须输出依据、置信度和缺失信息，并由运维主管确认。
>
> 这套设计最终支撑了设施运维 Agent 底座的 0 到 1 验证，也让产品不只是“看告警”，而是能从告警进入诊断、工单、执行和复盘。

### 高频追问与回答

#### Q1：你具体做了什么，不是研发做的吗？

可以答：

> 研发负责具体实现算法、接口和前后端能力。我主要负责把业务流程拆成研发能实现的产品结构：哪些场景触发 Agent，Agent 需要哪些上下文，每一步输出什么，哪些动作必须人工确认，怎么验收。我会把这些沉淀到 PRD、流程图、接口字段说明和验收标准里。比如诊断 Agent 要调用时序数据、历史工单和知识库；工单 Agent 只能生成草稿，正式派发必须由人确认。

#### Q2：这个项目里的 AI 到底智能在哪里？

可以答：

> 它不是单点模型智能，而是流程智能。第一，它能根据告警对象检索设备档案、历史维修和 SOP；第二，它能把诊断结果转成结构化处置方案；第三，它能调用工单、库存、排班等业务系统形成可执行草稿；第四，执行后的真实结果还能回流知识库，后续同类告警可以复用。

#### Q3：你怎么控制 AI 风险？

可以答：

> 我主要做三类边界。第一，输出边界：AI 必须给出依据、置信度、缺失信息，而不是只给一句结论。第二，操作边界：高风险动作不能自动执行，只能生成草稿或建议，正式派发由人确认。第三，系统边界：工具调用需要权限、租户隔离、审计和敏感字段裁剪，避免越权或提示注入导致错误操作。

#### Q4：你怎么定义验收标准？

可以答：

> 我不会只验“回答是否流畅”，而是拆成业务链路验收。比如告警触发后能否带出正确设备和空间；RAG 是否能引用对应 SOP 和历史工单；诊断结论是否有依据和置信度；工单草稿是否字段完整；人工确认后是否能正式派发；执行结果是否能回写并进入复盘。这样研发、测试和客户都知道怎么判断这个能力是否可交付。

## 项目二：Designer / DFS 数据闭环与智能楼宇 Demo

### 一句话定位

Designer 项目可以讲成：我参与构建“3D 数字孪生场景 + DFS 数据接入与分析 + 告警/优化闭环”的 AI/数据产品能力。

### STAR 结构

#### S：背景

客户想看见的不只是 3D 场景，而是“真实数据如何驱动场景变化，并产生业务决策”。例如智能楼宇空调系统里，会议室温度异常、空调功率升高、COP 下降，这些数据分散在传感器、设备、天气和能耗系统里。

如果只有 3D 模型，客户只能看到展示效果；如果只有数据表，客户又很难理解空间影响和业务价值。产品需要把场景、数据、分析和运维动作串起来。

#### T：任务

需要把 Designer 的场景能力和 DFS 的数据接入、数据探索、看板预警能力串起来，形成一个能讲清楚业务价值的端到端 Demo / 方案：

```text
3D 场景搭建
  -> 设备 / 孪生体绑定
  -> 数据接入
  -> 异常发现
  -> 根因分析
  -> 优化建议
  -> 工单 / 运维闭环
```

#### A：行动

**1. 场景设计**

定义办公区、会议室、空调机组、传感器等孪生对象，以及对应动态属性：

- 会议室：温度、舒适度、空间位置。
- 空调机组：运行状态、功率、水阀开度、效率指标。
- 传感器：温湿度、采集频率、设备序列号。
- 场景表现：温度异常时颜色从绿色变红，设备故障时触发告警动画。

**2. 数据链路规划**

梳理 MQTT / Node-RED / DFS 数据流，明确设备点表、字段格式、更新频率和孪生体属性绑定。

关键产品动作：

- 定义数据点表：设备 ID、属性名、数据类型、采样频率。
- 规划数据格式：传感器数据、空调数据、天气数据、能耗数据。
- 设计绑定关系：设备数据字段绑定到孪生体属性。
- 明确数据异常时的前端表现和看板反馈。

**3. 分析与可视化设计**

通过 DFS 数据探索和看板配置，把数据转成业务判断：

- 温度趋势对比：今日 vs 昨日。
- 空调机组功率变化。
- COP 等效率指标计算。
- 预警列表：温度超过阈值触发。
- 根因分析：结合温度、功率、室外天气、效率变化判断设备或环境因素。

**4. 交付协作**

把美术模型、Designer 场景、DFS 适配器、数据脚本、看板配置、测试用例拆给不同团队：

- 美术团队：办公区、会议室、设备、传感器模型。
- 客成 / 实施：场景搭建、设备绑定、Demo 环境。
- 研发：适配器、数据接入、看板和数据探索能力。
- 测试：端到端数据驱动、预警触发、分析结果验证。

#### R：结果

这个项目证明你能把“3D 可视化”讲成“数据驱动的业务闭环”。它支撑了 Designer / DFS 面向智能楼宇、数据中心冷却、商场空调调度等场景的演示与方案复用，也能和 Inspector 的工单闭环形成上下游关系。

### 30 秒口述版

> Designer 这段我会重点讲数据闭环。我们不是只做一个 3D 场景，而是用 Designer 建楼宇和设备孪生体，再通过 DFS 接入传感器、空调、天气和能耗数据，把数据绑定到孪生体属性上。比如会议室温度异常时，3D 场景颜色变化，看板触发预警，再通过数据探索分析空调功率和 COP 变化，定位可能的设备效率问题。我的价值在于把场景、数据、分析和运维动作串成一条可交付链路。

### 2 分钟展开版

> Designer / DFS 这段我会从数据闭环讲。客户通常不只是想看一个 3D 模型，而是想知道真实业务数据怎么驱动空间状态变化，并产生运维决策。以智能楼宇空调系统为例，会议室温度、空调功率、天气、能耗这些数据原本分散在不同系统里。
>
> 我在方案里把链路拆成几步：先用 Designer 构建办公区、会议室、空调机组和传感器孪生体；再通过 DFS 接入 MQTT 或 Node-RED 数据流；然后把设备数据字段绑定到孪生体属性上，比如温度驱动房间颜色变化，空调状态驱动设备动画；最后通过 DFS 数据探索和看板配置温度趋势、COP、功率变化和预警列表。
>
> 和团队协作时，我会把交付物拆清楚：美术团队需要什么模型，客成团队怎么搭场景，研发需要什么数据格式和接口，测试如何验证端到端链路。验收不是只看场景能打开，而是看数据脚本运行后，3D 场景是否正确变化，看板是否预警，数据探索是否能算出 COP 和趋势。
>
> 所以这个项目的价值是把数字孪生从展示工具推进到数据驱动的分析和运维闭环，也为后续 Inspector Agent 做诊断提供了空间语义和数据基础。

### 高频追问与回答

#### Q1：Designer 这个项目和 AI 产品有什么关系？

可以答：

> 它不是一个纯大模型项目，但它是 AI/数据产品的前置基础。AI 要做诊断和推荐，必须先有结构化的设备、空间、时序数据和业务对象。Designer 提供空间语义和孪生体，DFS 提供数据接入和分析，后续 Agent 才能基于这些上下文做诊断、解释和工单生成。

#### Q2：你在里面的产品判断是什么？

可以答：

> 我的判断是重点不能停留在“展示酷炫 3D”，而要转到“数据驱动决策”。所以我会定义哪些设备需要建模、哪些属性必须动态绑定、哪些指标能体现业务价值，比如温度异常、功率升高、COP 下降、预警触发、工单生成。这样 Demo 不是看模型，而是看一条真实业务链路。

#### Q3：你和研发怎么配合？

可以答：

> 我会把需求拆成几类交付物：模型需求、点表、数据格式、适配器配置、绑定规则、看板组件、数据探索任务和测试用例。研发、客成、美术各自有明确输入输出，最后用端到端测试验收，比如数据脚本运行后，3D 场景是否变化、看板是否预警、COP 是否计算正确。

#### Q4：这个项目的结果怎么讲更稳？

可以答：

> 我不会把它说成单独的 AI 项目结果，而是说它沉淀了数字孪生数据闭环能力，支撑 Designer / DFS 在智能楼宇、数据中心冷却、商场空调调度等场景中的演示和方案复用。它的价值在于把空间语义、实时数据和分析结果打通，为后续 Agent 诊断提供上下文。

## 表达边界：必须避免的坑

- 不要说“我主导训练模型”。更稳的是“定义 AI 能力边界、输入输出、评估与验收”。
- 不要说“AI 自动派单”。更稳的是“AI 生成工单草稿，人工确认后派发”。
- 不要把 Designer 说成纯 AI 项目。它更适合讲“数字孪生 + 数据闭环 + AI 上下文底座”。
- 不要只讲功能列表。每次都回到：业务痛点、产品判断、研发协作、落地验证。
- 不要把 Demo-ready 说成 Product-ready。可说“支撑 Demo / MVP 验证，并沉淀为可复用能力”。

## 模拟面试练习顺序

建议按以下顺序训练：

1. 用 30 秒讲 Inspector。
2. 用 2 分钟讲 Inspector。
3. 回答“你具体做了什么，不是研发做的吗？”
4. 回答“AI 到底智能在哪里？”
5. 回答“怎么控制 AI 风险？”
6. 用 30 秒讲 Designer / DFS。
7. 回答“Designer 和 AI 有什么关系？”
8. 最后把两个项目串成你的 B 端 AI 产品方法论。
