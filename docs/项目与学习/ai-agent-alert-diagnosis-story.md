# AI Agent 告警诊断能力：面试故事与产品框架

## 1. 一句话定位

面向商业楼宇设备运维场景，将 Inspector 从“告警展示与工单流转工具”升级为“AI Agent 驱动的告警诊断与运维决策辅助平台”，帮助一线运维人员完成从告警理解、根因分析、方案生成、工单执行到知识沉淀的闭环。

## 2. 业务场景与真实痛点

参考《商业楼宇设备运维工作场景-FAI》中的两个典型场景：

- 凌晨 2 点，B3 层空调机组突发停机报警。系统只提示“温度过高”，值班人员需要依次检查压缩机、冷凝器、电子阀门等多个部件，最终发现是冷却水阀异常，但系统没有记录该部件上次维护时间，抢修耗时 3 小时。
- 工作日晚高峰，商业中心 3 号电梯触发“轿厢抖动”报警，同时 BMS 显示区域电力波动。工程师需要在设备台账、BMS 曲线、工单系统和厂商资料之间切换，最终发现历史同类故障没有被系统关联，导致次日再次发生。

这些场景反映的核心问题不是“没有告警”，而是：

1. **单点告警无法解释根因**：设备内置报警只描述表象，无法判断是部件故障、上游设备异常、环境影响还是历史维护问题。
2. **数据分散在多个系统**：BMS、设备台账、工单系统、图纸文档、纸质日志和微信记录相互割裂。
3. **历史经验难以复用**：同类故障被粗粒度记录为“已处理”，无法统计高频故障、重复发生次数和真实成本归因。
4. **复杂故障处理依赖专家经验**：新人缺少图纸、历史维修记录和诊断路径，遇到复杂故障时沟通成本高。
5. **管理侧缺少复盘依据**：工程主管难以做预防性维护，财务/采购人员难以判断维修费用上涨的真实原因。

## 3. 面试中可以讲的产品故事

在 Inspector 项目中，我关注到商业楼宇运维场景里，传统系统通常只能展示 IoT 或 BMS 告警，但一线人员真正需要的是知道“为什么告警、影响哪里、下一步怎么处理”。例如空调机组停机时，系统只提示温度过高，运维人员仍然要在设备面板、历史工单、纸质日志和图纸之间来回切换；电梯抖动时，设备报警、BMS 电力曲线和历史维修记录也无法自动关联。

因此，我把这个能力定义为 AI Agent 驱动的告警诊断与运维决策辅助，而不是简单的 AI 问答。产品设计上，我会把设备台账、空间位置、BrickSchema 关系数据、孪生体属性、实时告警、历史工单、SOP 和维修知识库连接起来。当 DFS 或 BMS 产生业务预警时，Agent 先理解告警对象和上下文，再检索相似历史案例和相关 SOP，分析上下游设备、空间影响范围和可能根因，最后输出可解释的诊断建议、处置方案和工单草稿。

为了保证可靠性，我不会让 AI 直接替代运维人员做高风险决策，而是设计人机协同机制：AI 负责生成初步诊断、方案和工单，人工负责确认、编辑和最终下发。对于低置信度或消防、电梯、强电等高风险场景，系统必须提示人工复核或升级专家处理。输出上也不只给结论，而是给出判断依据、引用来源、置信度、缺失信息和建议检查项。

最后，我会把执行后的工单结果和人工反馈回流到知识库中。运维人员可以标记 AI 建议是否准确、实际故障原因是什么、SOP 是否有效、是否需要补充备件或专家建议。经过人工审结的高价值工单会沉淀为后续 Agent 的资料来源。这样，产品形成了“告警-诊断-方案-工单-执行-报告-知识沉淀”的闭环，既提升一线响应效率，也帮助工程主管识别高频故障和预防性维护机会。

## 4. AI 产品经理视角下的 Agent 架构

### 4.1 告警理解 Agent

负责把原始告警转化为结构化问题：

- 识别设备、空间位置、所属系统和告警类型。
- 判断告警等级、持续时间、是否重复发生。
- 关联同一空间、同一系统或上下游设备的并发告警。
- 将表象故障转化为可诊断的问题描述。

### 4.2 上下文检索与 RAG 模块

负责从业务知识源中补充上下文：

- 设备台账：型号、安装时间、维保周期、责任人、备件信息。
- 空间与关系数据：楼层、区域、设备上下游关系、BrickSchema 关系。
- 实时与历史数据：BMS 曲线、IoT 时序数据、DFS 预警记录。
- 历史工单：同类故障、处理过程、实际原因、耗时、成本。
- SOP 与知识库：标准检查步骤、维修方案、事故报告、厂商文档。

### 4.3 诊断推理 Agent

负责生成可解释的诊断结论：

- 可能原因 Top N。
- 每个原因的判断依据。
- 影响范围和风险等级。
- 需要补充检查的数据或现场信息。
- 是否建议升级专家、厂商或主管。

### 4.4 方案生成 Agent

在人工确认异常后，调取 MCP、知识库和历史案例生成初步处置方案：

- 现场检查步骤。
- 所需工具、备件和安全注意事项。
- 推荐 SOP。
- 预计处理路径。
- 需要人工确认的关键风险点。

### 4.5 工单生成 Agent

在人工确认方案后，生成工单草稿：

- 工单标题、故障描述、设备信息、空间位置。
- 推荐处理人或班组。
- 关联 SOP、备件和历史案例。
- 预期完成时间和风险等级。
- 下发前由人工审核和编辑。

### 4.6 事故报告与知识沉淀 Agent

在必要场景下生成事故报告草稿，并将可复用案例沉淀为知识库：

- 事故经过、影响范围、处理过程。
- 根因分析、成本影响、预防措施。
- 人工审结后的高价值工单进入客户知识库。
- 区分标准知识库与客户专属经验库，避免污染通用知识。

## 5. Agent 调优建议

### 5.1 业务规则调优

- 不同设备类型使用不同诊断路径，例如空调机组、电梯、消防、强电系统不能共用同一套规则。
- 高风险告警采用保守策略，只提供辅助判断，不自动执行控制。
- 连续多次告警、同空间多设备异常、近期维修后复发等情况提高优先级。
- 将“表象故障”和“真实根因”分开记录，避免所有问题都被粗略归类为“已处理”。

### 5.2 Prompt 与输出策略调优

- 要求 Agent 必须基于检索到的资料回答，不能凭空生成维修建议。
- 输出固定结构：结论、依据、置信度、建议动作、风险提示、缺失信息。
- 置信度低时，不输出确定结论，只输出可能方向和建议检查项。
- 对高风险设备，强制提示人工确认或专家升级。

### 5.3 知识库调优

- 将历史工单拆分为故障现象、设备信息、诊断过程、真实原因、处理动作、结果验证。
- 将 SOP 拆成可检索步骤，而不是只上传整份文档。
- 为设备、空间、故障类型、备件和处理方式建立标签。
- 人工审结后再将工单沉淀进知识库，避免错误案例污染后续生成。

## 6. 可解释性设计

AI 诊断结果不能只给“建议更换某部件”，而应给出可验证依据：

- 为什么判断这个原因。
- 引用了哪些历史工单或 SOP。
- 关联了哪些 BMS/IoT 数据。
- 哪些上下游设备可能受影响。
- 当前置信度是多少。
- 哪些信息缺失，需要人工补充。

示例输出结构：

```text
可能原因：冷却水阀异常或局部管路堵塞。
判断依据：当前空调机组温度过高告警与近 10 分钟冷却水流量下降同时出现；历史工单中同区域曾出现冷却水阀卡滞问题；该部件缺少最近一次维护记录。
影响范围：B3 层空调区域，可能影响部分租户办公区温控。
建议动作：检查冷却水阀开度、管路压力和过滤器状态；若现场确认阀门异常，按 SOP-AC-03 创建维修工单。
置信度：中。需要现场补充阀门状态和管路压力数据。
```

## 7. 可靠性与风险控制

### 7.1 人机协同边界

- AI 生成初步方案，人工确认后才能创建正式方案。
- AI 生成工单草稿，人工审核后才能下发执行。
- AI 生成事故报告草稿，人工审阅后才能归档。

### 7.2 分级处理机制

- 低风险：AI 推荐 SOP 和检查项，人工快速确认。
- 中风险：AI 生成方案和工单草稿，主管审核。
- 高风险：消防、电梯、强电、燃气等场景必须升级人工专家或值班主管。

### 7.3 置信度与兜底

- 高置信度：输出明确建议和依据。
- 中置信度：输出候选原因和建议检查项。
- 低置信度：提示信息不足，不给确定结论。
- 数据缺失时，明确指出缺哪些数据，而不是编造结论。

## 8. 反馈机制与持续优化

闭环设计：

1. Agent 生成诊断建议、方案或工单草稿。
2. 运维人员确认、修改或驳回。
3. 执行人员记录处理过程、实际原因、备件使用和结果。
4. 主管审结工单，判断是否进入知识库。
5. 高价值案例沉淀为客户专属经验库。
6. 后续同类告警调用该案例，提升诊断准确率。

可跟踪指标：

- 告警诊断命中率。
- 平均故障处理时间 MTTR。
- 工单创建耗时。
- SOP 推荐采纳率。
- 人工修改率和驳回率。
- 重复故障识别率。
- 高价值案例入库数量。
- 运维人员满意度。

## 10. 概念解释：Agent、Skill、RAG

### Agent 是什么

Agent 可以理解为具备目标、工具调用能力和多步决策能力的 AI 执行单元。它不只是回答一句话，而是能围绕一个任务拆步骤、查资料、调用系统、生成方案并根据反馈继续推进。

在 Inspector 场景中，Agent 可以是：

- 告警理解 Agent：把原始告警转成结构化问题。
- 诊断推理 Agent：结合上下文判断可能根因。
- 工单生成 Agent：把方案转成可执行工单。

### Skill 是什么

Skill 更像是给 Agent 使用的一套固定能力、流程或操作手册。它通常封装了某类任务的最佳实践，让 Agent 在处理特定任务时遵循稳定方法。

可以这样区分：

- Agent：负责“谁来做、多步怎么推进”。
- Skill：负责“某一步具体按什么方法做”。

类比到运维场景：

- Agent 像一个智能运维助理。
- Skill 像它掌握的标准技能，例如“空调机组故障诊断流程”“电梯报警分级规则”“事故报告生成模板”。

### RAG 是什么

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。它的核心思想是：模型在回答前先从知识库、文档、工单、SOP 或数据库中检索相关资料，再基于检索结果生成回答。

RAG 适合楼宇运维这类场景，因为 AI 不能只靠通用模型知识来判断现场故障，必须参考客户自己的设备台账、历史工单、SOP、BMS 数据和维保记录。

简单说：

- 没有 RAG：AI 可能凭经验泛泛回答。
- 有 RAG：AI 先查“这栋楼、这台设备、这类历史故障、这份 SOP”，再给出更可解释、更贴近现场的建议。
