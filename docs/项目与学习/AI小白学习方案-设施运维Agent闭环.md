# AI 小白学习方案：设施运维 Agent 闭环

> 目标：从 AI 小白出发，围绕“设施运维 Agent 闭环”建立可面试、可写简历、可和研发对齐的产品理解。  
> 学习原则：不先硬啃模型原理，先从业务闭环入手，再逐层补齐数据、Agent、RAG、工具调用、权限审计和验收。

---

## 一、最终要达到的能力

你不是要变成算法工程师，而是要达到 AI 产品经理面试里的“可交付理解”：

1. 能用 1 分钟讲清楚设施运维 Agent 闭环是什么。
2. 能解释为什么 24x7 监控不用大模型，而用确定性算法和规则。
3. 能说清 Agent 在诊断、调度、工单生成里分别调用什么数据和系统。
4. 能画出从告警到工单再到知识沉淀的流程。
5. 能写出一个 AI 功能 PRD 的核心章节。
6. 能回答招聘官常问的风险、权限、验收、效果指标问题。

一句话目标：

> 我理解的设施运维 Agent，不是聊天框，而是把告警、设备数据、知识库、工单系统和人工确认串起来的智能处置闭环。

---

## 二、学习地图

设施运维 Agent 闭环可以拆成 7 层：

```text
第 1 层：业务场景
  设备为什么会告警，运维人员为什么痛苦

第 2 层：数据底座
  传感器、设备台账、历史工单、SOP、库存排班

第 3 层：异常识别
  确定性算法、规则、健康评分、趋势、告警降噪

第 4 层：Agent 与 RAG
  Agent 怎么查数据、查知识库、生成诊断和建议

第 5 层：工具调用
  Agent 怎么调用设备、工单、库存、排班等业务系统

第 6 层：人机协同与风控
  哪些能自动，哪些必须人工确认，怎么审计

第 7 层：产品化与验收
  PRD、状态机、错误处理、指标、客户验收
```

学习顺序一定要按这个来。不要一开始就问“LSTM 原理是什么”，那会把自己绕晕。

## 三、4 周学习计划

### Week 1：先听懂业务闭环

目标：你能把设施运维 Agent 的故事讲顺。

#### Day 1：理解设施运维痛点

学习问题：

- 什么是设施运维？
- 为什么设备告警后，人工处理很慢？
- 告警、工单、SOP、设备台账分别是什么？

重点理解：

- 告警只说明“有问题”，不说明“为什么”。
- 工单是把问题变成可执行任务。
- SOP 是标准作业步骤。
- 设备台账是设备的身份档案。

今日产出：

```text
用 5 句话解释：
1. 设施运维里谁是用户？
2. 他们每天面对什么问题？
3. 告警为什么不等于诊断？
4. 工单为什么重要？
5. AI 可以帮哪一步？
```

自测答案方向：

> 设施运维用户主要是运维经理和一线技师。他们面对大量设备告警，但告警通常只说明温度高、振动异常、设备离线，并不说明根因。处理人员还要查设备台账、历史维修记录和 SOP。工单的价值是把问题转成任务、责任人、时间和执行记录。AI 可以先帮用户整合上下文、判断可能原因、生成处置建议和工单草稿。

#### Day 2：背熟闭环主流程

你要记住这条链：

```text
告警触发
  -> 异常识别
  -> 诊断分析
  -> 资源调度
  -> 工单草稿
  -> 人工确认
  -> 执行回写
  -> 知识沉淀
```

今日产出：

把这条链改写成自己的话，不超过 120 字。

参考表达：

> 系统先通过传感器和规则识别设备异常，生成结构化告警；用户点击告警后，Agent 调用设备数据、历史工单和知识库分析原因，再查询备件和人员排班生成维修方案，最后生成工单草稿，由运维经理确认后派发，执行结果回写并沉淀为后续知识。

#### Day 3：分清“告警、诊断、处置、闭环”

学习重点：

- 告警：发生了什么。
- 诊断：为什么发生。
- 处置：下一步怎么做。
- 闭环：做完以后结果有没有回写。

今日练习：

用 AHU 精密空调例子填空：

```text
告警：AHU-02 振动异常，预计 72 小时内风险升高。
诊断：可能是 __________。
处置：建议 __________。
闭环：维修完成后需要记录 __________。
```

参考答案：

```text
诊断：可能是风机皮带松弛或打滑。
处置：建议在低负载窗口更换皮带，并执行断电挂牌等安全 SOP。
闭环：记录真实故障原因、更换备件、维修耗时、AI 建议是否准确、设备恢复后的振动指标。
```

#### Day 4：学会一句话讲项目

今日产出：

背熟这一版：

> 这个项目不是做一个 AI 聊天框，而是把 AI 嵌入设施运维的告警到工单流程。后台用规则和算法持续识别异常，用户点击告警后，Agent 再调用设备数据、历史工单、知识库、库存和排班系统，生成诊断结论、维修方案和工单草稿，最后由人工确认派发，并把执行结果回写成知识。

#### Day 5：第一次模拟面试

练习问题：

1. 这个 Agent 闭环解决什么问题？
2. 为什么不是做一个聊天机器人？
3. 工单为什么必须人工确认？

合格标准：

- 能说出“告警不等于诊断”。
- 能说出“AI 嵌入主流程”。
- 能说出“高风险动作必须人审”。

### Week 2：补齐 AI 和数据基础

目标：你能说清楚 AI 在这个项目里到底做什么，不再只说“赋能”。

#### Day 6：理解确定性算法 vs LLM

重点结论：

```text
24x7 监控：确定性算法 / 规则
用户解释与生成：LLM / Agent
```

为什么？

- 监控要稳定、便宜、一致、可离线。
- LLM 适合解释、总结、生成方案、写报告。
- 工业场景不能每条传感器数据都调大模型。

面试表达：

> 后台异常识别我不会全部交给大模型，而是优先用确定性算法、阈值、趋势和健康评分。LLM 更适合在用户点击告警后做解释、检索和方案生成。这样能兼顾成本、稳定性和可解释性。

#### Day 7：理解异常识别 Pipeline

只需要理解产品层，不需要推公式。

```text
传感器读数
  -> 数据清洗
  -> 特征提取
  -> 异常评分
  -> 告警 / 抑制
```

例子：

- ISO 10816：用振动速度判断设备状态。
- 轴间关联：看 X/Y/Z 三轴是否异常。
- 趋势检测：看是否逐步退化。
- 降噪：低置信度异常不推送，避免告警轰炸。

今日产出：

用 3 句话解释“为什么要告警降噪”。

参考表达：

> 工业现场有大量传感器噪声，如果简单超过阈值就告警，会造成告警轰炸。一线人员会疲劳，反而忽视真正重要的异常。因此系统需要结合趋势、置信度和多通道关联，把低置信度异常抑制，把高风险事件推送出来。

#### Day 8：理解 RAG

一句话：

> RAG 就是 AI 回答前先查资料，再基于资料回答。

设施运维里 RAG 查什么？

- 厂商手册
- SOP
- 历史工单
- 设备维护记录
- 故障模式知识库

今日产出：

回答：为什么不能让大模型直接凭经验给维修建议？

参考答案：

> 因为设施运维是客户现场强相关场景，不同设备型号、维护历史、SOP、安全要求都不同。大模型如果不查客户自己的资料，容易给泛化甚至错误建议。所以 Agent 必须先基于告警对象检索设备档案、历史工单和 SOP，再生成诊断和处置建议。

#### Day 9：理解 Agent

一句话：

> Agent 是能围绕目标拆步骤、查资料、调用工具、生成结果的 AI 执行单元。

本项目里的 Agent：

- 异常分析 Agent：建策略、下发规则。
- 诊断 Agent：查数据、查历史、查知识库。
- 调度 Agent：查库存、查排班、找维修窗口。
- 工单生成 Agent：拉模板、写 SOP、生成草稿。

今日产出：

用表格写出每个 Agent 的输入、动作、输出。

#### Day 10：理解 Tool / Skill

一句话：

> Tool / Skill 是 Agent 调用业务系统的能力。

例子：

- 查设备状态
- 查知识库
- 查历史工单
- 查库存
- 查排班
- 创建工单草稿

面试表达：

> Agent 的价值不只是生成文本，而是能通过工具调用业务系统。比如诊断 Agent 调设备时序和历史工单，调度 Agent 调库存和排班，工单 Agent 调模板和草稿接口。

### Week 3：学习产品设计和研发协作

目标：你能从 PRD 和研发协作角度讲清楚怎么落地。

#### Day 11：画触发机制

你要掌握三类触发：

```text
系统触发：定时任务、设备接入、基线重估
事件触发：异常告警、设备状态变化
用户触发：点击诊断、制定方案、草拟工单
```

今日产出：

写出三段：

- 什么时候触发诊断 Agent？
- 什么时候触发调度 Agent？
- 什么时候触发工单生成 Agent？

参考答案：

```text
诊断 Agent：用户点击告警进入设备详情，前端携带 alert_id 和 device_id 调用。
调度 Agent：用户确认诊断结论后点击制定维修方案，系统携带维修对象和所需备件调用。
工单生成 Agent：用户确认方案后点击草拟工单，系统携带诊断结论、备件、人员、时间等上下文调用。
```

#### Day 12：学习上下文传递

关键词：

- `alert_id`
- `device_id`
- `conversation_id`
- `draft_id`
- `work_order_id`

你要理解：

> 每一步都要有稳定 ID 串起来，否则 Agent 不知道自己处理的是哪个告警、哪台设备、哪个草稿。

今日产出：

画一条 ID 链：

```text
alert_id -> device_id -> conversation_id -> draft_id -> work_order_id
```

并解释每个 ID 的作用。

#### Day 13：学习 PRD 要写什么

AI 功能 PRD 至少包含：

- 背景与痛点
- 用户角色
- 触发条件
- 输入数据
- Agent 动作
- 输出格式
- 人工确认点
- 错误与降级
- 权限与审计
- 验收标准

今日产出：

写一个小 PRD 片段：

```text
功能：AI 诊断告警
触发：用户点击设备告警
输入：alert_id、device_id、近 7 天时序数据、历史工单、SOP
输出：可能根因、依据、置信度、建议检查项、缺失信息
人工确认：用户点击“制定方案”后进入下一步
异常处理：知识库无结果时提示信息不足，不输出确定结论
```

#### Day 14：学习权限和审计

你要掌握的最小知识：

- Agent 不能拥有无限权限。
- 权限属于当前用户、租户、角色。
- 只暴露用户能用的工具。
- 敏感操作要二次确认。
- 每次工具调用要有日志。

面试表达：

> 我会要求 Agent 工具调用经过网关控制，按用户角色和租户裁剪可用工具；敏感操作如派发工单、扣减库存、导出数据必须二次确认；每次调用记录 request_id、conversation_id、用户、租户、操作和结果摘要，方便审计和追溯。

#### Day 15：学习验收标准

错误示范：

> AI 能回答问题。

正确验收：

```text
1. 告警能正确关联设备。
2. Agent 能检索对应历史工单和 SOP。
3. 输出有根因、依据、置信度、缺失信息。
4. 低置信度不输出确定结论。
5. 工单草稿字段完整。
6. 正式派发前必须人工确认。
7. 执行结果能回写。
8. 失败时有错误提示和审计记录。
```

### Week 4：面试输出训练

目标：能像 AI 产品经理一样回答问题。

#### Day 16：准备 1 分钟回答

题目：

> 设施运维 Agent 闭环是怎么实现的？

标准答案：

> 我理解的设施运维 Agent 闭环，不是给系统加一个聊天框，而是把 AI 嵌入告警到工单的主业务流程。底层先通过传感器数据和确定性算法做异常识别，生成结构化告警；用户点击告警后，诊断 Agent 调设备时序、历史工单和 SOP 知识库，输出根因、依据和置信度；用户确认后，调度 Agent 查询备件、人员排班和维修窗口；最后工单 Agent 拉取模板生成工单草稿，由运维经理确认后派发。执行结果回写后，再沉淀成知识库，形成告警、诊断、处置、复盘、学习的闭环。

#### Day 17：准备“你具体做了什么”

标准答案：

> 我不是做算法实现，而是负责把业务流程产品化：定义 Agent 的触发条件、输入输出、调用哪些业务系统、哪些动作必须人工确认、失败怎么降级、怎么验收。我会把这些拆成 PRD、流程图、接口字段和验收标准，帮助研发明确实现边界。

#### Day 18：准备“AI 到底智能在哪里”

标准答案：

> 智能不只是生成一段话，而是能基于业务上下文完成多步任务。它能根据告警对象查设备数据、历史工单和 SOP，判断可能根因；再调用库存和排班系统形成维修方案；最后把方案转成工单草稿。它把跨系统的信息整合和操作编排自动化了。

#### Day 19：准备“怎么控制风险”

标准答案：

> 我会从三层控制风险。第一是输出层，必须有依据、置信度和缺失信息，低置信度不输出确定结论。第二是操作层，高风险动作只能生成建议或草稿，正式派发必须人工确认。第三是系统层，工具调用要经过权限、租户隔离、参数校验和审计。

#### Day 20：准备“怎么衡量效果”

指标：

- 告警误报率 / 降噪率
- 平均诊断时间
- 工单创建耗时
- SOP 推荐采纳率
- AI 建议人工修改率
- 工单一次解决率
- 重复故障识别率
- 知识库有效案例沉淀数量

标准答案：

> 我会同时看效率、质量和风险指标。效率上看诊断时间和工单创建耗时；质量上看 SOP 采纳率、一次解决率、重复故障识别率；风险上看低置信度兜底、人工修改率和错误建议率。这样能判断 AI 是真的改善业务，而不只是生成内容。

## 四、你需要掌握的最小术语表

| 术语 | 你需要会说的话 |
|------|----------------|
| Agent | 能围绕目标查资料、调用工具、生成结果的 AI 执行单元 |
| RAG | 回答前先查知识库，再基于资料生成答案 |
| Tool / Skill | Agent 可调用的业务能力，比如查设备、查库存、建工单 |
| Deterministic Pipeline | 不依赖大模型的规则/算法流水线，适合 24x7 监控 |
| Human-in-the-loop | AI 给建议，人做确认，尤其适合高风险业务 |
| Work Order | 工单，把问题转成责任人、时间、步骤和执行记录 |
| SOP | 标准作业程序，告诉一线人员怎么安全、规范地处理 |
| Audit Log | 审计日志，记录谁在什么时候让 Agent 调了什么工具 |
| Confidence | 置信度，表示 AI 或算法对结论的把握程度 |
| Write-back | 回写，指执行结果重新写回系统，形成复盘和学习 |

## 六、每日学习方法

每天只做三件事：

```text
1. 读 20 分钟
2. 用自己的话写 5 句话
3. 回答 1 个面试问题
```

不要追求一次看懂所有技术。你的学习目标是：

- 能判断业务链路是否闭环。
- 能说清 AI 放在哪一步。
- 能定义输入输出和人工确认点。
- 能和研发讨论接口、权限、状态和验收。

## 七、第一天现在就开始

今天只学一个问题：

> 为什么设施运维里“有告警”还不够？

你可以先写下自己的答案，然后对照下面这版：

> 因为告警只告诉用户出现了异常，比如温度过高、振动异常、设备离线，但它不解释根因、影响范围和下一步动作。一线人员仍然要人工查设备台账、历史工单、SOP 和现场数据。设施运维 Agent 的价值就是把告警变成可诊断、可处置、可复盘的任务闭环。
