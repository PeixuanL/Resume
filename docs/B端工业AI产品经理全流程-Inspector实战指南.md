## 1. 角色定位

B 端工业 AI 产品经理通常同时承担四类职责，而非仅「写 PRD」：

| 维度 | 职责 |
|------|------|
| **业务** | 场景是否真有价值、能否付费/续约、ROI 如何表述 |
| **产品** | 需求边界、体验路径、版本节奏、标品 vs 定制 |
| **AI/数据** | 能力边界、指标定义、何时调 RAG/Prompt/流程/精调 |
| **交付** | 售前承诺与研发现实对齐、试点→上线、客户成功闭环 |

**工业 B 端特点：**

- 往往**没有统一的 C 端式指标**（如全网统一的「99.5% 准确率」），需与客户共创场景级成功标准。
- 客户常需按**产线/楼宇/设备类型**做配置或知识精调。
- **安全与合规**权重大：电梯、消防、强电等高风险场景必须人机协同，不能全自动决策。

## 2. 一句话产品定义（Inspector）

**定位：** 将 Inspector 从「告警展示与工单流转工具」升级为 **AI Agent 驱动的告警诊断与运维决策辅助平台**。

**价值：** 帮助一线运维完成从 **告警理解 → 根因分析 → 方案生成 → 工单执行 → 知识沉淀** 的闭环；让工程主管获得可复盘、可统计的真实根因数据。

**不是什么：** 不是全局「大聊天框」，而是嵌入主业务链路的 **可控、可解释、可验收** 的 AI 能力。

### 2.1 典型痛点（业务场景）

| 场景 | 表象 | 本质问题 |
|------|------|----------|
| B3 空调机组停机 | 系统只报「温度过高」 | 需在多部件间人工排查；维护记录缺失；抢修耗时长 |
| 3 号电梯轿厢抖动 | 设备报警 + BMS 电力波动 | 台账、曲线、历史工单、厂商知识割裂；同类故障未关联 |

**共性结论：** 核心问题不是「没有告警」，而是：

1. 单点告警无法解释根因  
2. 数据分散在 BMS、台账、工单、图纸、微信等多系统  
3. 历史经验难以复用（工单常只记「已处理」）  
4. 复杂故障依赖专家，新人成本高  
5. 管理侧缺少预防性维护与成本归因依据

## 3. 端到端全流程总览

### 3.1 产品全生命周期（PM 视角）

```text
【1. 线索 & 需求进入】销售 / 售前 / 客户成功 / 试点反馈
        ↓
【2. 需求澄清 & 场景定义】场景卡片、数据盘点、成功标准、MVP 边界
        ↓
【3. 方案 & 产品化设计】Agent 分工、RAG 策略、人机协同、集成方式
        ↓
【4. PRD & 验收标准】功能 + AI 行为 + 评估集 + 非功能 + 试点清单
        ↓
【5. 研发协作 & 迭代】评审、排期、联调、灰度、客户环境
        ↓
【6. 试点 / PoC / 上线】基线数据、培训、运维手册
        ↓
【7. 度量 & 反馈闭环】定量 + 定性 → 调产品 / RAG / Prompt / 流程
        ↓
（回到 2 或 3，进入下一版本）
```

**「上链」含义：** 需求进入 backlog → PRD → 研发版本 → 试点数据与客户验收 → 正式交付与续约依据；每一环需**可追溯的决策记录**。

### 3.2 Inspector 业务闭环（运维链路）

```text
告警接入（DFS/BMS/IoT）
    → 告警理解（结构化 + 关联并发告警）
    → 上下文检索（RAG）
    → 诊断推理（根因 TopN + 依据 + 置信度）
    → 方案生成（检查项 / SOP / 风险点）
    → 工单草稿 → 人工确认 → 下发执行
    → 执行记录 / 事故报告（可选）
    → 反馈回流 → 审结 → 知识库
    ↑__________________________________|
```

## 4. 告警→工单闭环的 8 个环节

| 环节 | 负责模块 / Agent | PM 需明确的产出 |
|------|------------------|-----------------|
| ① 告警接入 | 平台集成 | 数据源、等级、延迟、失败降级 |
| ② 告警理解 | 告警理解 Agent | 设备/空间/类型/复发/并发关联 |
| ③ 上下文检索 | RAG | 引用来源可追溯（工单、SOP、曲线） |
| ④ 诊断推理 | 诊断推理 Agent | TopN 根因、依据、缺失信息、是否升级专家 |
| ⑤ 方案生成 | 方案生成 Agent | 检查步骤、备件、安全提示、推荐 SOP |
| ⑥ 工单生成 | 工单生成 Agent | 结构化草稿；**禁止未经确认自动下发** |
| ⑦ 执行与报告 | 工单系统 + 可选报告 Agent | 真实根因、耗时、成本字段完整 |
| ⑧ 知识沉淀 | 审结 + 知识库 | 仅高价值、已审结案例入库 |

### 4.1 人机协同原则

- **AI：** 生成初步诊断、方案、工单草稿。  
- **人：** 确认、编辑、下发；高风险场景强制复核或升级专家。  
- **输出：** 结论 + 判断依据 + 引用来源 + 置信度 + 缺失信息 + 建议检查项（不可只给一句「换某部件」）。

### 4.2 风险分级策略（示例）

| 等级 | 策略 |
|------|------|
| 高风险（消防、电梯、强电） | 仅辅助诊断，不自动执行控制；强制人工复核 |
| 中风险 | AI 生成方案与工单草稿，主管审核 |
| 低风险 | 可快速确认下发，仍保留编辑与追溯 |

## 5. Agent / Skill / RAG / 编排

### 5.1 三者关系（一句话）

> **Agent** 是完成端到端任务的智能体；**Skill** 是 Agent 调用的可复用专业能力；**RAG** 是 Agent 先从客户现场资料检索再生成答案的方法。

### 5.2 Inspector 中的 Agent 分工

| Agent | 职责 |
|-------|------|
| 告警理解 Agent | 原始告警 → 结构化问题 |
| 诊断推理 Agent | 结合上下文输出可解释根因 |
| 方案生成 Agent | 处置步骤、SOP、风险点 |
| 工单生成 Agent | 方案 → 工单草稿 |
| 知识沉淀 Agent | 事故报告草稿、审结后入库（可选） |

### 5.3 Skill 示例（产品化沉淀方向）

- 空调故障诊断 Skill  
- 电梯告警分级 Skill  
- SOP 匹配 Skill  
- 工单生成 Skill（字段规则、标题模板）  

**标品化目标：** 将 FDE/试点中反复调试的逻辑沉淀为 Skill 或配置模板，避免每客户一套黑盒。

### 5.4 RAG 知识源分层

| 层级 | 内容 |
|------|------|
| 设备与空间 | 台账、BrickSchema 关系、孪生体属性、楼层区域 |
| 运行数据 | 实时/历史告警、BMS/IoT 时序、DFS 预警 |
| 经验与流程 | 历史工单（结构化字段）、SOP 步骤、厂商文档 |
| 客户专属 | 审结后的高价值工单、客户经验库（与通用库隔离） |

**检索纪律：** 先结构化「问题是什么」，再按告警对象绑定检索；避免泛化问答。

### 5.5 LangGraph（编排层，PM 级理解）

当流程具备 **循环、条件分支、人工暂停**（如低置信 → 仅输出检查项 → 人补现场信息 → 再诊断）时，适合用图编排（LangGraph 等）实现。

**PM 交付物：** 状态机 / 流程图，标明每步输入输出、失败降级、最多循环次数、人工介入点——而非指定具体技术框架。

## 6. 按 PM 工作节奏的五阶段

### 阶段 1：需求进入与澄清

**输入来源：** 销售、售前、试点客户、客户成功、一线反馈。

**场景卡片（模板字段）：**

- 用户角色（值班工程师、工程主管）  
- 触发条件（哪类告警、哪类设备）  
- 当前痛点与替代方案（不用 AI 时怎么做）  
- 成功标准（定性 + 可抽检定量）  
- 数据盘点（台账、工单、SOP、时序是否可用）  
- 约束（私有化、延迟、权限、审计）  
- MVP 边界（**本版不做什么**）  
- 风险（幻觉、误建议导致安全/停产）  

**首轮对齐研发/算法（30 分钟）：** 规则+RAG 是否够用？是否需要多 Agent？是否涉及客户侧精调？

---

### 阶段 2：方案与产品化设计

**Inspector 关键产品决策示例：**

1. 通用底座 vs 项目制定制 → 告警可视化、工单状态机、空间语义优先底座化  
2. AI 嵌入主路径，而非独立聊天入口  
3. 人机边界：草稿 + 人工确认；高风险强制复核  

**技术路径（由浅到深）：**

```text
配置层（术语表、设备路由、风险分级、拒答规则）
    → RAG（知识库 + 与告警对象绑定检索）
        → 多 Agent 编排（理解→检索→诊断→方案→工单）
            → 模型精调（仅当 1+2 不足且具备数据与评测闭环）
```

---

### 阶段 3：PRD 与验收（见第 8 节）

与研发协作：方案评审 → 迭代计划 → 提供黄金样例 → 联调 checklist → 发布与回滚条件。

---

### 阶段 4：试点与交付（FDE 协同）

| 角色 | 分工 |
|------|------|
| PM | 场景优先级、PRD、版本节奏、指标与验收定义 |
| FDE | 环境对接、知识库冷启动、试点陪跑、case 复现 |
| 研发/算法 | 平台能力、评测工具、可复用组件 |
| 客户成功/售前 | 预期管理、合同范围、续约叙事 |

**试点机制：**

- 试点前采集**基线**（诊断时长、复发率、人工步骤数）  
- **双周三角会**（PM + FDE + 研发）：失败 case 分类、是否进标品路线图  
- **试点周报**：数据、Top bad case、归因（产品/配置/知识/模型）  

**标品化成果（可量化叙事）：** 约 70% 定制诉求沉淀为通用模块；头部客户交付周期从 5–6 个月压缩至 1–2 个月（需结合本项目实际数据表述）。

## 7. 业务目标与指标体系

### 7.1 目标拆解层级

```text
公司/产品线目标（收入、续约、市占）
    → 产品目标（场景渗透率、试点转正率）
        → 版本目标（本迭代解决的 Top 客户痛点）
            → AI/场景指标（可测、可抽检）
```

### 7.2 Inspector 推荐指标

| 类型 | 指标 | 说明 |
|------|------|------|
| **平台 SLA** | 链路可用性、告警接入成功率、P95 时延 | 「99.5%」类指标通常指**服务可用**，≠ 诊断正确 |
| **效果** | 诊断 Top1/Top3 命中率、答案可用率 | 需专家抽检或标注集 |
| **效率** | 平均诊断时长、工单创建耗时、人工接管率 | 对接客户 ROI |
| **质量** | 重复告警率、工单一次解决率 | 体现知识沉淀是否有效 |
| **安全** | 高风险误建议率、强制复核覆盖率 | 电梯/消防/强电 |
| **商业** | 试点转正率、标品模块复用率、交付周期 | 平台化结果 |

### 7.3 无行业标准时怎么办

1. 与客户共创 **《试点成功标准 / 验收表》**（可签字）  
2. 试点前记录 **基线**  
3. 用 **历史告警回放 + 专家抽检**（如 20 条中 ≥15 条可用）  
4. 双周复盘：决定调产品、调知识库、调 Prompt/Skill，还是上精调

## 8. PRD 要点（含 AI 专属章节）

### 8.1 标准章节

| 章节 | 内容 |
|------|------|
| 背景与目标 | 业务目标、版本目标、客户/行业 |
| 用户与场景 | 角色、触发、主路径/异常路径 |
| 功能说明 | 页面、权限、与 BMS/工单等集成 |
| 非功能 | 性能、安全、日志、多租户 |
| 发布与运维 | 灰度、回滚、监控、客户培训 |
| 不做清单 | scope out |

### 8.2 AI 产品必增章节

| 章节 | 内容 |
|------|------|
| **AI 行为规格** | 输入输出、固定结构、拒答策略、引用展示 |
| **知识 & 数据** | 知识库范围、更新机制、标注与切片规范 |
| **人机协同** | 确认权限、复核规则、高风险兜底 |
| **评估与验收** | 黄金样例、测试集、抽检方法、通过线 |
| **可观测性** | 日志能否定位到检索/doc/模型/环节 |

### 8.3 诊断输出结构（建议固定）

1. 结论（可能根因 TopN）  
2. 判断依据  
3. 置信度  
4. 建议动作 / 检查项  
5. 风险提示  
6. 缺失信息（还需现场补充什么）  
7. 引用来源（工单号、SOP 段落等）

## 9. 大模型与技术路径选型

PM 不必负责训练，但需参与决策表：

| 决策点 | 要问的问题 |
|--------|------------|
| 模型 | 中文与术语、上下文长度、成本、是否可私有化、合规 |
| RAG vs 微调 vs 规则 | 知识是否常变？是否有足够工单与评测集？ |
| 编排 | 是否多步、有循环、需人工 interrupt？ |
| 评估 | 离线评测集？上线后漂移如何监控？ |

**默认策略：** 优先配置 + RAG + Skill；精调是最后手段。

## 10. 与 FDE、研发、客户的协作

### 10.1 会议与产出

| 阶段 | PM 产出 | 协作要点 |
|------|---------|----------|
| 方案评审 | 流程图、边界、验收标准 | 澄清数据依赖与技术风险 |
| 迭代计划 | 优先级、依赖项 | 区分平台能力 vs 单客户定制 |
| 开发中 | 黄金样例、澄清记录 | 好/坏答案各 10 条起 |
| 联调 | 试点环境 checklist | 全链路日志可追溯 |
| 上线 | 发布说明、回滚条件 | 灰度 + 监控看板 |

### 10.2 bad case 分类标签（共用）

`检索未命中` | `知识过期` | `Prompt/Skill` | `推理错误` | `工单/流程` | `数据质量`

### 10.3 与研发沟通的原则

- 讲清用户路径与失败兜底，少空谈「模型不够聪明」  
- 用样例和指标驱动，而非感觉驱动  
- 每环节输入输出边界清晰，避免单 Prompt 包打天下

## 11. 反馈闭环与知识运营

### 11.1 闭环流程

```text
Agent 输出诊断 / 方案 / 工单草稿
    → 工程师确认或修改（记录与 AI 的差异）
    → 执行工单
    → 反馈：AI 是否准确、真实根因、SOP 是否有效
    → 主管审结
    → 高价值案例进入客户知识库
    → 后续同类告警检索命中 → 提升命中率
```

### 11.2 知识库三条纪律

1. **未审结工单禁止直接灌库**，避免错误经验污染  
2. **SOP 拆成可检索步骤**，不要只上传整份 PDF  
3. **区分标准知识库与客户专属库**  

### 11.3 工单结构化字段（便于 RAG 与统计）

故障现象、设备信息、诊断过程、真实原因、处理动作、结果验证、耗时、成本。

## 12. 关键概念速查

| 概念 | 在 Inspector 链路中的含义 |
|------|---------------------------|
| **Agent** | 推进「这一单告警」端到端任务的智能体单元 |
| **Skill** | 可复用的专业能力（空调诊断、电梯分级、工单规则等） |
| **RAG** | 基于台账/工单/SOP/时序等现场资料检索后再生成 |
| **LangGraph** | 多 Agent 有状态编排；PM 关心状态机与人工介入点 |
| **FDE** | 现场部署/交付工程师：对接、知识库、试点、case 回流 |
| **SLA（如 99.5%）** | 多指服务可用性，与诊断准确率分开度量 |
| **标品化** | 定制需求沉为底座模块与实施基线，缩短交付周期 |
| **人机协同** | AI 草稿 + 人确认；高风险强制复核 |

## 13. 分步学习路径（8 周）

| 周次 | 主题 | 作业产出 |
|------|------|----------|

| 第 5 周 | 编排与兜底 | 状态机（含人工确认点） |
| 第 6 周 | FDE 与交付 | 试点周报模板 + bad case 标签表 |
| 第 7 周 | 研发协作 | 联调 checklist + 发布/回滚说明 |
