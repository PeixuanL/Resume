# Agent、Skill、RAG：面向 Inspector AI 告警诊断场景的理解笔记

## 1. 先记住一句话

在 Inspector 的 AI 告警诊断场景里，可以这样理解三者关系：

> **Agent 是负责完成任务的智能体，Skill 是 Agent 执行任务时调用的专业能力，RAG 是 Agent 获取外部知识后再生成答案的方法。**

放到商业楼宇设备运维场景中：

- **Agent**：像一个智能运维助理，负责完成“分析告警、判断原因、生成方案、创建工单”等任务。
- **Skill**：像这个助理掌握的专业技能，例如“空调故障诊断流程”“电梯告警分级规则”“SOP 匹配方法”。
- **RAG**：像这个助理在回答前先查资料，例如查设备台账、历史工单、SOP、BMS 数据、BrickSchema 关系和孪生体属性。

## 2. 为什么这个问题在 Inspector 里重要

《商业楼宇设备运维工作场景-FAI》里有两个典型问题：

### 2.1 空调机组停机场景

凌晨 2 点，B3 层空调机组突发停机报警，系统只提示“温度过高”。运维人员无法直接判断原因，需要依次检查制冷剂、风机、管道、压缩机、冷凝器、电子阀门等多个部件。最终发现是冷却水阀异常，但系统没有记录该部件上次维护时间，抢修耗时 3 小时。

这个场景的本质问题是：

- 单点告警只能说明“发生了什么”，不能解释“为什么发生”。
- 设备、工单、维护记录、纸质日志和聊天记录分散。
- 历史经验无法复用，重复故障无法识别。

### 2.2 电梯抖动场景

工作日晚高峰，3 号电梯出现“轿厢抖动”报警，同时 BMS 显示区域电力波动。工程师需要手动查设备台账，发现电梯是 10 年前安装的 OTIS 3100 型；联系厂商后才知道该型号存在“电压不稳导致安全继电器误动作”的固有问题。但 BMS 历史电力数据与工单系统分离，无法证明这台电梯已经发生过多次同类故障。

这个场景的本质问题是：

- 设备告警、BMS 曲线、历史工单、厂商资料彼此割裂。
- 工单只记录“已处理”，没有关联真实根因。
- 管理者难以判断设备是否需要整改或更新。

所以，Inspector 的 AI 能力不能只是做一个聊天框，而应该围绕业务流程设计成：

> 告警识别 -> 上下文检索 -> 根因诊断 -> 方案生成 -> 工单生成 -> 执行记录 -> 事故报告 -> 知识沉淀

## 3. Agent 是什么

Agent 可以理解为具备目标、工具调用能力和多步决策能力的 AI 执行单元。它不是简单回答一句话，而是围绕一个目标持续推进任务。

在 Inspector 中，一个 Agent 可以做这些事：

1. 读取告警。
2. 判断告警属于哪类设备、空间和风险等级。
3. 查询设备台账、BMS 数据、历史工单和 SOP。
4. 分析上下游设备关系。
5. 输出可能根因、影响范围和处置建议。
6. 生成方案草稿或工单草稿。
7. 等待人工确认、编辑和下发。
8. 将人工反馈沉淀为后续知识。

### 3.1 Inspector 中可以有哪些 Agent

#### 告警理解 Agent

负责把原始告警转成结构化问题。

例如：

- 原始告警：B3 空调机组温度过高。
- 结构化理解：设备为空调机组，位置为 B3，告警类型为温度异常，可能关联冷却水阀、冷凝器、风机和压缩机。

#### 根因诊断 Agent

负责结合上下文判断可能原因。

例如：

- 发现该空调机组近期缺少冷却水阀维护记录。
- 历史工单中同区域曾发生过冷却水阀卡滞。
- 当前温度过高与冷却水流量异常同时出现。
- 因此将冷却水阀异常列为高优先级排查方向。

#### 方案生成 Agent

在人工确认异常后，调取 MCP、知识库和 SOP，生成初步处置方案。

例如：

- 检查冷却水阀开度。
- 检查管路压力。
- 检查过滤器是否堵塞。
- 必要时按指定 SOP 创建维修工单。

#### 工单生成 Agent

在人工确认方案后，把方案转成工单草稿。

例如：

- 工单标题。
- 故障设备。
- 空间位置。
- 故障描述。
- 推荐检查步骤。
- 所需工具或备件。
- 风险等级。
- 推荐处理班组。

#### 事故报告 Agent

在重大事故或必要场景下，生成事故报告草稿。

例如：

- 事故经过。
- 影响范围。
- 根因分析。
- 处理过程。
- 预防措施。
- 后续整改建议。

## 4. Skill 是什么

Skill 是 Agent 可以调用的专业能力、流程或操作手册。它通常比 Agent 更小、更固定、更可复用。

如果 Agent 像“智能运维助理”，Skill 就像这个助理掌握的具体技能。

### 4.1 Skill 的特点

- 面向某一类具体任务。
- 规则和流程相对固定。
- 可以被不同 Agent 复用。
- 帮助 Agent 降低自由发挥，提升稳定性。

### 4.2 Inspector 中可以有哪些 Skill

#### 空调故障诊断 Skill

用于处理空调机组相关告警。

可能包含：

- 温度过高时优先检查冷却水流量、冷却水阀、压缩机、冷凝器和风机。
- 如果近期维修过，优先判断是否为复发故障。
- 如果同区域多个设备异常，判断是否为区域级环境或供能问题。

#### 电梯告警分级 Skill

用于处理电梯相关告警。

可能包含：

- 轿厢抖动属于安全敏感告警。
- 如果同时存在 BMS 电力波动，应检查电压稳定性。
- 如果该型号存在厂商已知问题，应关联厂商知识库。
- 涉及乘客安全时必须升级人工主管或外部专家。

#### SOP 匹配 Skill

用于把诊断结果匹配到标准处置流程。

可能包含：

- 按设备类型匹配 SOP。
- 按故障现象匹配检查步骤。
- 按风险等级提示是否需要停机、隔离或专家确认。

#### 工单生成 Skill

用于把方案转换成结构化工单。

可能包含：

- 工单标题生成规则。
- 故障描述字段。
- 设备、空间、负责人、备件字段。
- 风险等级与处理时限。

### 4.3 Agent 和 Skill 的区别

| 维度 | Agent | Skill |
|---|---|---|
| 定位 | 负责完成一个端到端任务 | 负责完成任务中的某个专业步骤 |
| 粒度 | 更大 | 更小 |
| 灵活性 | 更强，可以多步决策 | 更固定，偏流程和规则 |
| 示例 | 告警诊断 Agent | 空调故障诊断 Skill |
| 类比 | 智能运维助理 | 助理掌握的一项技能 |

一个简单结构：

```text
空调告警诊断 Agent
  ├── 告警理解 Skill
  ├── 空调故障诊断 Skill
  ├── SOP 匹配 Skill
  ├── 工单生成 Skill
  └── 事故报告 Skill
```

## 5. RAG 是什么

RAG 是 Retrieval-Augmented Generation，中文通常叫“检索增强生成”。

它的核心逻辑是：

> AI 在回答前，先从外部资料中检索相关内容，再基于检索结果生成答案。

在商业楼宇运维里，AI 不能只靠通用模型知识回答。因为模型本身不知道：

- 这栋楼有哪些设备。
- 这台设备的型号是什么。
- 上次维护是什么时候。
- 历史上是否发生过类似故障。
- 当前 BMS 曲线是否异常。
- 客户自己的 SOP 是什么。
- BrickSchema 中这台设备与哪些上下游设备有关。

所以需要 RAG 先查资料。

## 6. RAG 在 Inspector 中查什么

RAG 可以检索的内容包括：

### 6.1 设备台账

- 设备名称。
- 型号。
- 安装时间。
- 所属系统。
- 所在楼层和空间。
- 上次维护时间。
- 责任人。
- 备件信息。

### 6.2 BMS / IoT / DFS 数据

- 实时告警。
- 历史告警。
- 温度、电压、电流、压力、流量等曲线。
- DFS 业务预警。
- 设备状态变化。

### 6.3 BrickSchema 关系数据和孪生体属性

PDF 中提到，Agent 可以综合分析 BrickSchema 关系数据和孪生体属性数据，判断设备上下游情况。

这类数据可以帮助 AI 回答：

- 这台设备上游是什么。
- 下游影响哪些区域。
- 同一系统中还有哪些关联设备。
- 是否可能是上游设备异常导致表象故障。

### 6.4 历史工单

- 同类故障是否发生过。
- 当时怎么处理。
- 真实原因是什么。
- 处理耗时多久。
- 是否使用备件。
- 是否复发。

### 6.5 SOP 和知识库

- 标准检查步骤。
- 安全注意事项。
- 厂商文档。
- 专家经验。
- 事故报告。
- 人工审结后的高价值工单。

## 7. RAG 的价值是什么

### 7.1 让回答更贴近现场

没有 RAG，AI 可能只会说：

> 空调温度过高可能由制冷剂不足、风机故障或管道堵塞导致。

有 RAG，AI 可以说：

> 当前 B3 空调机组温度过高告警与冷却水流量下降同时出现；历史工单显示同区域曾发生冷却水阀卡滞；该部件缺少最近维护记录。因此建议优先检查冷却水阀开度、管路压力和过滤器状态。

### 7.2 让结果可解释

RAG 可以告诉用户 AI 为什么这么判断：

- 引用了哪条历史工单。
- 参考了哪份 SOP。
- 关联了哪段 BMS 曲线。
- 使用了哪个设备台账字段。

### 7.3 降低幻觉风险

AI 不再凭空编造，而是基于可追溯资料生成建议。

### 7.4 支持客户专属知识

每个楼宇、每个客户、每套设备都有自己的差异。RAG 可以让 Agent 使用客户自己的资料，而不是只依赖通用模型知识。

## 8. 用一个完整流程串起来

以空调机组温度过高为例：

### 8.1 告警进入

DFS 或 BMS 产生告警：

> B3 空调机组温度过高。

### 8.2 Agent 接管任务

告警诊断 Agent 启动，目标是：

> 判断可能根因，并生成初步处置建议。

### 8.3 Agent 使用 RAG 查资料

它检索：

- 设备台账：这台空调机组的位置、型号、维护记录。
- BMS 数据：温度、流量、压力曲线。
- BrickSchema：上下游设备关系。
- 历史工单：是否有类似故障。
- SOP：温度过高排查流程。

### 8.4 Agent 调用 Skill 分析

它调用：

- 告警理解 Skill。
- 空调故障诊断 Skill。
- SOP 匹配 Skill。

### 8.5 Agent 输出诊断

输出：

- 可能原因。
- 判断依据。
- 影响范围。
- 推荐检查动作。
- 置信度。
- 缺失信息。

### 8.6 人工确认

运维人员结合现场情况确认：

- AI 判断是否准确。
- 是否创建方案。
- 是否生成工单。

### 8.7 Agent 生成方案和工单

人工确认后：

- 方案生成 Agent 输出初步方案。
- 工单生成 Agent 输出工单草稿。
- 人工审核后下发执行。

### 8.8 反馈回流

执行完成后：

- 记录真实原因。
- 记录处理动作。
- 记录备件使用。
- 记录是否复发。
- 主管审结后，将高价值工单沉淀进知识库。

## 9. 为什么不是让 Agent 全自动处理

商业楼宇运维包含安全风险，尤其是电梯、消防、强电、燃气等场景。因此 AI 产品经理要设计边界：

- AI 可以生成初步诊断，但不能替代人工做高风险决策。
- AI 可以生成方案草稿，但最终方案需要人工审核。
- AI 可以生成工单草稿，但工单下发需要人工确认。
- AI 可以生成事故报告草稿，但报告归档需要人工审阅。

这正好对应 FAI 文档里的设计：

- 预警后由人工判断异常情况。
- Agent 自动生成初步方案。
- 最终方案需要人工审核和编辑。
- 工单需要人工确认后下发。
- 事故报告也需要人工审核和编辑。
- 可复用工单需要人工审结后再进入知识库。

## 10. 面试中可以这样回答

如果面试官问：**Agent 和 Skill 有什么区别？**

可以回答：

> 我会把 Agent 理解为负责完成端到端任务的智能体，而 Skill 是 Agent 在执行任务时调用的专业能力。比如在 Inspector 的楼宇运维场景里，告警诊断 Agent 负责从告警理解、上下文检索、根因分析到工单草稿生成的完整流程；而空调故障诊断、电梯告警分级、SOP 匹配、工单生成这些更具体、更可复用的能力，可以沉淀为 Skill。这样既能让 Agent 具备多步处理能力，也能通过 Skill 约束它的执行路径，提升稳定性。

如果面试官问：**RAG 是什么，为什么这里需要 RAG？**

可以回答：

> RAG 是检索增强生成，也就是 AI 回答前先从外部知识源检索相关资料，再基于检索结果生成答案。楼宇运维是强业务上下文场景，大模型本身不知道某栋楼的设备型号、历史故障、BMS 曲线、SOP 和维护记录。如果没有 RAG，AI 只能给通用建议；有了 RAG，Agent 可以先查设备台账、历史工单、BrickSchema 关系、孪生体属性和 SOP，再输出有依据、可解释、贴近现场的诊断建议。

如果面试官问：**这个能力怎么保证可靠性？**

可以回答：

> 我会从四层控制可靠性：第一，RAG 约束模型必须基于客户现场资料和知识库回答；第二，Skill 约束不同设备类型的诊断流程和输出结构；第三，通过置信度、证据引用和缺失信息提示提升可解释性；第四，设计人工审核机制，高风险场景只做辅助诊断，不自动执行。最终方案、工单下发和事故报告都需要人工确认。

## 11. 最终记忆版本

可以直接背这个版本：

> Agent 是负责完成任务的智能体，Skill 是 Agent 调用的专业能力，RAG 是 Agent 查资料后再生成答案的方法。在 Inspector 的 AI 告警诊断场景里，告警诊断 Agent 会通过 RAG 检索设备台账、BMS/DFS 数据、历史工单、SOP、BrickSchema 关系和孪生体属性，再调用空调诊断、电梯分级、SOP 匹配、工单生成等 Skill，输出可能根因、影响范围、处置建议和工单草稿。为了保证可靠性，AI 只生成初步诊断和方案，最终方案、工单下发和事故报告都需要人工审核；执行完成后的高价值工单经过人工审结后进入知识库，成为后续 Agent 诊断同类问题的资料来源。
