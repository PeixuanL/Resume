03 · SETUP

告警-工单闭环, 是 AI 落地的最佳第一阶段切口

选择这个场景不是因为最大, 而是因为最可验证、最闭环、失败成本最低。

选择逻辑

  • 数据可得: IoT 告警、设备台账、历史工单已有积累
  • 流程闭环: 从告警到工单有明确状态机, 可端到端验收
  • 价值可感: 减少诊断时间、降低漏派工、沉淀维修知识

一句话闭环

告警触发 → AI 辅助诊断 → 维修方案筹备 → 工单草稿生成 → 人工确认 → 工单执行 → 结果回写 / SOP 沉淀

核心判断: 先做"告警到工单草稿", 而不是全链路无人化。闭环可验证, 风险可控。

04 · SETUP

传统运维四大痛点: 告警淹没、诊断靠经验、派工低效、知识流失

这些痛点来自与运维值班工程师、维修主管、IT 负责人的访谈归纳。

告警淹没

日均大量告警涌入, 无效告警占比高, 值班人员长期处于疲劳状态, 容易漏掉关键告警。
诊断依赖经验

老工程师退休后, 新人面对复杂告警无从下手。经验没有结构化沉淀, 每次故障都是重新摸索。
派工低效

从告警到工单平均耗时较长, 涉及多个系统切换, 信息反复录入, 响应速度受人为因素制约。
知识流失

维修记录散在 Excel 或纸质文档, 无法复用为 SOP。老师傅的笔记本丢了, 经验就断了。
根因: 人是瓶颈。AI 的价值不是替代人, 而是把人的时间从重复判断中释放出来。

05 · EVIDENCE

0-1 判断: 为什么先做告警→工单草稿, 而不是全链路无人化

客户想要"智能运维大脑", 我把它拆成三阶段, 第一阶段只做最小可验证单元。

保留 · 第一阶段 MVP

  • 告警接入与过滤规则
  • AI 诊断建议, 多假设 + 置信度
  • 工单草稿自动生成
  • 人工确认节点, 强制拦截

砍掉 · 降低交付风险

  • 自动派单, 组织未准备好
  • 预测性维护, 数据质量不足
  • 跨 ERP/采购系统联动, 集成成本过高
  • 全链路无人决策, 失败不可承受
交付策略: 4-6 周 Demo 验证核心链路, 再决定产品化投入。不赌全链路。

06 · EVIDENCE

业务闭环: 告警触发到工单执行回写的六步状态机

每一步都有明确的输入输出、验收标准和异常降级路径。

告警触发 → AI 诊断 → 方案筹备 → 工单草稿 → 人工确认 → 执行回写
告警触发IoT 阈值 / 规则引擎触发, 生成结构化告警事件输入物完整
AI 诊断RAG 检索知识库 + 设备参数匹配, 输出故障假设列表多假设并列
方案筹备调用 SOP 模板库 + 历史相似工单, 推荐维修步骤模板可复用
工单草稿自动填充设备、位置、建议措施、所需物料字段完整
人工确认工程师审核、修改、批准。高风险动作强制确认不可跳过
执行回写维修结果更新设备状态, 经验沉淀为新 SOP闭环验收
状态机不是概念图, 是研发验收标准。每步都有输入输出和异常分支。

设施运维 AI Agent
07 · EVIDENCE

AI Agent 架构: 诊断输出置信度, 人工确认后生成工单

AI 的输入、输出、工具调用、人工介入点全部明确定义, 不是黑盒。

诊断 Agent

输入: 告警文本 + 设备 ID + 实时参数 + 历史工单

输出: 故障假设列表, 每项带置信度 + 建议排查步骤

工具: RAG 检索知识库、设备资产查询接口

边界: 置信度 < 70% 标记为"需人工复核"

工单生成 Agent

输入: 人工确认后的诊断结果 + SOP 模板

输出: 工单草稿, 字段已填充 + 所需物料清单

工具: SOP 模板库、物料库存查询

边界: 草稿须经工程师修改批准后才生效

产品侧定义: 场景、数据输入输出、算法评估指标、验收标准、人机协同边界。不碰模型训练。

数据架构: IoT、设备资产、知识库、3D 场景的四层融合

设备 ID 是统一主键, 串联感知层、资产层、知识层与应用层。

四层数据

  • 感知层 · IoT 传感器, 时序数据库, 告警规则引擎
  • 资产层 · 设备台账, 空间关系, BIM/GIS 模型
  • 知识层 · SOP 模板库, 历史工单, 维修手册, RAG 向量库
  • 应用层 · 运维看板, 诊断面板, 工单系统, 3D 定位

3D 场景的作用

不是大屏装饰, 是告警定位工具:

· 点击告警 → 三维空间定位设备

· 显示楼层、邻近设备、关联系统

· 辅助维修路径规划

数字孪生不是好看, 是"告警定位"这个具体价值。底座能力来自 DFS / Designer。

09 · EVIDENCE

人机协同: 高风险动作强制人工确认, 低置信度自动降级

B端 AI 最大的风险不是不够聪明, 是太自信。三层控制 + 异常兜底。

输入控制告警过滤规则减少无效输入; 设备参数校验防脏数据
输出控制置信度阈值拦截; 多假设并列不强制单选; 高风险动作白名单
执行控制工单审批流; 操作权限分级; 审计日志全记录
异常兜底AI 超时 fallback 传统模板; 知识库为空标记专家介入
审计追溯每条 AI 建议关联原始数据源, 可追溯到具体知识库条目
原则: AI 可以建议, 不能替人决定。停机、断电等高风险动作必须弹出确认框。

10 · EVIDENCE

Vibe Coding: 用 PRD 和接口假设快速验证原型, 而非替代研发

产品经理的验证工具, 不是研发替代方案。价值在"提前发现错误"。

输入物

  • PRD: 用户故事、验收标准
  • 流程图: 状态机、异常分支
  • 接口字段假设: 请求 / 响应结构
  • UI 规则: 布局、交互、校验逻辑

输出物

  • 可交互原型, 快速走通主流程
  • 边界问题发现: 状态机遗漏、字段缺失
  • 需求澄清: 给研发的具体问题清单
  • 与研发开会前的共识基础
案例: 用 Vibe Coding 验证告警→工单草稿链路, 提前发现漏了"紧急程度"字段。
