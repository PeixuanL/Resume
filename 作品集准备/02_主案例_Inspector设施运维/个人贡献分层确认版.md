# 个人贡献分层确认版

更新时间：2026-07-01

## 结论

用户已确认原“个人贡献分层待确认”中的 A/B/C/D 项均可采用。其中：

- A 类可写成“主导/端到端负责”。
- B 类可写成“负责模块/深度参与”。
- C 类可写成“参与方案设计/产品化表达”，但作品集整理阶段可以继续补强，不等同于所有内容都来自既有工作交付。
- D 类只写成“了解原理、能讲清产品链路”，不写成算法、模型训练或工程实现 owner。

## 可直接放进作品集的贡献表达

### 1. 主导：0-1 产品闭环与产品文档沉淀

可写：

> 主导 Inspector 设施运维场景的 0-1 产品闭环设计，将资产、告警、3D 场景、工单处置和知识沉淀串联为可演示、可评审、可持续推进的业务链路，并沉淀 PRD、用户流程、状态机、页面结构和交互说明等产品文档。

适合放在：主案例总览页、贡献页、简历项目经历。

### 2. 负责：工单、告警、资产、3D 场景等关键模块

可写：

> 负责或深度参与工单列表/详情、状态流、创建分配、执行完成、退回改派、SLA/评价等工单闭环设计，并串联设备详情、移动端/离线执行、剧本/AR/3D 资源缓存等现场执行场景。

可写：

> 深度参与告警、资产、设备台账、3D 场景与运维流程之间的信息架构设计，使异常发现、设备定位、处置执行和记录沉淀能够进入同一业务流。

适合放在：工单闭环页、产品底座页、页面/原型展示页。

### 3. 参与/补强：AI Agent / RAG 产品化表达

可写：

> 围绕 AI Agent / RAG 在设施运维中的产品化方向，梳理告警诊断、SOP/手册检索、历史工单检索、处置建议、工单草稿生成和人机确认的辅助链路，用产品流程定义 AI 能力边界。

边界说明：

- 这部分可以在作品集整理阶段继续补强，重点是把方案讲成完整产品逻辑。
- 不写成“我主导算法方案”或“AI 模块已完整上线交付”。
- 需要把 AI 定位为辅助建议、人机确认和知识复用，而不是自动派单或自动维修。

适合放在：AI 产品化页、人机协同页、面试问答。

### 4. 工作方式：AI-Native PM

可写：

> 在产品方案整理过程中，使用 AI 工具辅助 PRD 结构化、流程图/状态机梳理、原型和前端 Demo 探索，提高复杂业务场景的表达和验证效率。

边界说明：

- 写成工作方法和验证方式。
- 不写成生产交付成果或工程上线证明。

### 5. 了解原理：能讲清但不写成 owner

可讲：

- 预测性维护多 Agent 方案的业务链路和产品边界。
- RAG 检索、知识库、SOP/手册召回逻辑的产品输入输出。
- API 调用链、数据触发机制、工程实现细节只讲产品层流程，不讲工程实现 owner。

## 禁止或需要避开的表达

| 不建议表达 | 推荐替代表达 |
|---|---|
| 我负责 AI 算法训练/模型开发 | 我定义 AI 在业务流程中的产品边界和人机确认点 |
| AI 已经自动完成诊断、派单和维修 | AI 辅助生成诊断解释、处置建议和工单草稿，由人确认后进入流程 |
| 工单与 AI 闭环已完整上线 | 基于产品底座完成方案设计、原型/Demo 验证和 POC 路径梳理 |
| 我负责 API 和工程链路实现 | 我梳理产品层的数据流、触发点和用户路径 |

## 面试时的贡献口径

> 我的核心贡献不是单纯画页面，而是把设施运维里分散的资产、告警、空间定位、工单执行和知识沉淀梳理成一条最小可闭环的产品链路。在这个基础上，我进一步把 AI 放到知识检索、处置建议、工单草稿和人机确认这些节点里，既能体现 AI 产品化能力，也保留了 B 端高风险场景必须有的人工确认和审计边界。
