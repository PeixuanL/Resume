# 现有资料回答度评估

更新时间：2026-06-29

## 结论

现有资料足够支撑作品集主案例的“业务场景、产品闭环、AI能力边界、工单/Workflow状态流、页面/PRD依据”。

但现有资料不能完全替代用户补充“个人贡献边界、真实实现阶段、可公开展示范围、验证材料真实性”。这些必须由用户确认，否则容易把方案、模拟口径或团队材料包装成个人已交付成果。

## 六个问题回答度

| 问题 | 现有资料能否回答 | 可用资料 | 仍需用户补充 |
|---|---|---|---|
| 1. 业务背景是什么 | 基本能回答 | 设施运维 Agent 复盘稿、PdM 落地方案、触发机制和执行链路 | 真实客户/行业背景是否可写，是否是实际项目还是方案化案例 |
| 2. 你参与了哪些模块/文档/流程/评审 | 只能部分推断 | 简历与项目学习文档、已有作品集 PPT、PRD 材料 | 必须确认本人实际参与模块、产出物、会议/评审角色 |
| 3. 哪些已实现，哪些是 PRD/方案/MVP/演示 | 部分能回答 | AI Agent 用户手册、工单/Workflow 用户指南、Inspector 工单/资产/3D PRD、PdM 架构文档 | 必须确认在你的案例里哪些已落地，哪些只是规划或 demo |
| 4. 是否有可脱敏展示材料 | 部分能回答 | 已有 PPT、HTML、PRD 内嵌图、功能架构图、流程图 | 必须确认哪些截图可公开、是否需要重新截图或脱敏 |
| 5. 是否有验证材料 | 现有资料较弱 | DEMO/验收/部署类文档、CSV 样例、旧 PPT 可作为旁证 | 缺真实 POC/客户反馈/验收/评审结论，需用户确认 |
| 6. 最想突出哪些标签 | 资料能给建议，最终需用户选择 | 既有作品集 source-notes、主案例工作稿、简历背景资料 | 需要用户确认优先标签，建议优先：0-1、AI 产品化、工单闭环 |

## 可直接进入作品集的材料

| 作品集内容 | 可用证据 | 表达建议 |
|---|---|---|
| 业务问题 | 告警后靠人工查数据、翻 SOP、问专家、手动建工单 | 可写为典型设施运维痛点，不写具体客户数据 |
| 产品闭环 | 告警触发 -> 诊断分析 -> 处置方案 -> 资源调度 -> 工单草稿 -> 人工确认 -> 执行回写 -> 知识沉淀 | 可作为主案例核心流程图 |
| AI 能力边界 | 确定性 Pipeline 24x7，LLM on-demand；AI 输出结论、证据、排除项、建议 | 可写为产品判断，不写全自动维修 |
| 工单状态流 | CREATED/ASSIGNED/IN_PROGRESS/COMPLETED/CANCELLED，支持退回、改派、评价、SLA | 可用于证明流程严谨性 |
| Workflow 审批 | 工单审批、调度审批、状态追踪、操作历史 | 可用于证明人机确认和审计边界 |
| Inspector 资产联动 | 设备详情新增工单 Tab，设备自动预填，设备位置带入 | 可用于证明资产-工单闭环 |
| 移动/离线执行 | 工单字段、操作步骤、剧本资源、离线缓存、执行记录 | 可作为执行端复杂度补充 |
| Vibe Coding | 触发机制、API Contract、模块架构可转成 AI 编程输入 | 只能写成原型/验证方法，不写生产交付成果 |

## 不建议直接写死的内容

| 内容 | 原因 | 建议处理 |
|---|---|---|
| 降低停机 X% / 提升 OEE X% | 现有资料未提供真实指标 | 不写，或标为待补充验证数据 |
| AI 自动派单/自动维修 | 与人机确认边界冲突 | 改成“处置建议 + 人工确认 + 工单流转辅助” |
| 本人主导全部 Agent 架构 | 个人贡献未确认 | 改成“参与/梳理/设计某模块”，等用户确认后再升级 |
| 完整商业化上线 | 现有资料包含 PRD、demo、方案、用户手册混合 | 必须区分已实现、MVP、方案、演示 |
| 客户真实名称和内部数据 | 需要脱敏与可公开确认 | 使用行业/场景替代 |

## 下一步需要用户最小补充

只需要补 4 件事即可开始写主案例 1.0：

1. 你本人实际参与的 3-5 个产出物。
2. 主案例中哪些是已实现，哪些是方案/demo/PRD。
3. 哪些截图或页面可以脱敏展示。
4. 有没有任何真实评审/POC/反馈/验收材料可以写成阶段性验证。
