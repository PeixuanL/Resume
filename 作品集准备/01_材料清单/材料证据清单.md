# 材料证据清单

更新时间：2026-06-29

## 使用方法

每个材料都要标注：

- **证据用途：** 用来证明什么。
- **可展示性：** 可公开 / 需脱敏 / 仅内部参考。
- **证据强度：** 强 / 中 / 弱 / 待确认。
- **作品集表达：** 可以直接写、保守写、还是不要写。

## 主案例：Inspector / 设施运维 / AI 辅助闭环

| 材料类型 | 需要材料 | 当前状态 | 证据用途 | 表达边界 |
|---|---|---|---|---|
| 业务背景 | 设施运维痛点、客户/行业场景、项目目标 | 待补充 | 解释为什么要做这个产品 | 不写具体 ROI，除非有真实数据 |
| 用户角色 | 运维人员、设施主管、工程师、客户管理员画像 | 待补充 | 证明你理解 B 端多角色差异 | 可基于产品设计总结，但要避免写成真实调研结论 |
| 流程材料 | 告警、诊断、处置、工单、记录沉淀的流程图 | 待补充 | 证明闭环设计能力 | 区分已实现、方案设计、MVP 验证 |
| 页面截图 | 告警、资产、工单、3D 场景、AI 辅助相关截图 | 待补充 | 提供视觉证据 | 必须脱敏 |
| PRD/规划 | 工单、模板库、资产、3D 场景等 PRD 或规划材料 | 已有参考资料，待筛选 | 证明产品设计过程 | 不把规划包装成已上线成果 |
| AI 能力 | RAG、知识检索、诊断建议、人机确认流 | 待补充 | 证明 AI 产品化判断 | AI 是能力层，不写成自动派单/自动维修 |
| 个人贡献 | 你负责的模块、判断、协调、文档、原型 | 待补充 | 证明不是泛泛团队成果 | 只写自己确实参与的部分 |
| 验证材料 | POC、评审、客户反馈、验收、复盘 | 待补充 | 替代虚构数据 | 没有数据就写“阶段性验证/方案沉淀” |
| AI 工具过程 | Cursor / Codex / Kimi 辅助梳理、原型或文档过程 | 待补充 | 体现 AI-Native 工作方式 | 写成工作方法，不写成生产交付成果 |

## 支撑案例候选

| 候选案例 | 可证明能力 | 当前判断 | 待补充材料 |
|---|---|---|---|
| Designer / 数字孪生编辑器 | 空间语义、3D/场景配置、数字孪生底座理解 | 适合做产品底座支撑 | 关键页面、你参与的产品判断、与 Inspector 的关系 |
| DFS / Data Fusion Services | 数据接入、数据治理、分析画布、数字孪生数据底座 | 适合做数据能力支撑 | 功能边界、典型场景、个人贡献、可展示材料 |
| Singapore FDS / 英文文档支持 | 国际化支持、英文产品文档、方案协同 | 适合低权重支撑，不适合主案例 | 英文材料、支持范围、交付对象 |

## 明确禁止或谨慎表达

| 表达 | 处理方式 |
|---|---|
| 提升 OEE X% / 降低停机 X% | 没有真实证据不写 |
| AI 自动派单 / 自动维修 | 不写，改为“诊断建议 + 人工确认 + 流转辅助” |
| 算法训练负责人 | 没有证据不写 |
| 完整商业化上线成果 | 如果只是 Demo/MVP/方案，必须标注阶段 |
| Vibe Coding 生产交付能力 | 改为“产品验证和原型迭代方法” |
