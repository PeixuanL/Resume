# FactVerse Designer MVP：孪生场景架构师 PRD

## 一、 MVP 目标
1.  **降门槛**：用户无需深研手册，即可通过对话完成基础的 3D 布局。
2.  **增效率**：将繁琐的行为树（BT）节点拖拽和参数配置，简化为自然语言描述。
3.  **验证架构**：打通 OpenClaw 的 Workspace/Memory 机制与 Designer 的 API 闭环。

---

## 二、 核心架构定义

### 2.1 工作空间 (Workspace) - AI 的“视网膜”
AI 在对话时实时读取的场景快照（JSON 格式），包含：
*   **Scene Snapshot**：
    *   所有物体的 `EntityID`、`TemplateName`、`Position(x,y,z)`、`Rotation`。
    *   物体间的层级关系（Parent-Child）。
    *   物体的出入口端口（Ports）定义。
*   **BT Snapshot**：
    *   当前画布上所有节点的 ID、类型、连接状态及参数配置。
*   **Template Dictionary**：
    *   当前租户下已加载的所有“孪生体模板”及其包含的属性、动画列表。

### 2.2 记忆 (Memory) - AI 的“大脑”
*   **短期记忆 (Session)**：记录本次会话中用户提及的“它”、“这个”、“刚才那个物体”。
*   **长期记忆 (Knowledge)**：
    *   **手册记忆**：用户手册第 4、5、6 章（快速上手、功能说明、行为树）。
    *   **逻辑模版记忆**：公司标准的“生成-传输-存储”经典行为树范例。
    *   **操作偏好**：例如用户习惯将传送带高度设为 0.8 米。

## 三、 MVP 核心功能模块描述

### 3.1 模块一：对话式空间编排 (Layout Agent)
**核心逻辑**：将 NLP 转化为 `Create` 或 `Transform` 系列指令。

*   **功能点 3.1.1：模糊模板检索与生成**
    *   **用户**：“给我一个 AGV 小车。”
    *   **行为**：AI 在资源库检索最匹配的模板 ID（如 `AGV_Standard_V1`），调用 `CreateEntity` 置于场景原点。
*   **功能点 3.1.2：规则化阵列布局**
    *   **用户**：“把选中的货架生成 2x5 的阵列，间距 3 米。”
    *   **行为**：AI 识别当前选中物体，计算 10 个坐标偏移量，批量执行 API。
*   **功能点 3.1.3：基于语义的对齐与吸附**
    *   **用户**：“把这个箱子放在传送带的末端。”
    *   **行为**：AI 检索传送带出口坐标，自动修正箱子的 `Position`。

### 3.2 模块二：行为树逻辑助手 (BT Copilot)
**核心逻辑**：将 NLP 转化为行为树的 DSL（领域特定语言），并映射至编辑器。

*   **功能点 3.2.1：单任务逻辑骨架生成**
    *   **用户**：“创建一个简单的循环：等待 5 秒，然后播放‘抓取’动画。”
    *   **行为**：在 BT 画布生成：`Root` -> `Sequence` -> `Wait(5s)` -> `PlayAnimation(Grab)`。
*   **功能点 3.2.2：自动参数匹配**
    *   **用户**：“如果距离小于 2 米，就停下来。”
    *   **行为**：AI 自动创建 `Compare Distance` 节点，并从当前选中的 Twin 中自动提取其位置属性填入参数槽位。
*   **功能点 3.2.3：逻辑一键解释**
    *   **用户**：“这段逻辑在干嘛？”
    *   **行为**：AI 读取 BT Snapshot，输出语义化描述：“这是一个检测满载并自动前往充电站的逻辑”。

## 四、 用户交互流程描述 (MVP Example)

### 场景：快速搭建一条原料生产线
1.  **唤起**：用户在 Designer 右侧打开 AI 对话框。
2.  **交互 1（布局）**：
    *   **用户**：“帮我搭建一个原料生产工段，要一个生成器连接一个 5 米的传送带。”
    *   **AI**：*(执行指令)* 在场景中放置两个物体，并自动将传送带的 `Length` 属性设为 5。
    *   **系统反馈**：3D 视图中实时出现模型。
3.  **交互 2（连接）**：
    *   **用户**：“把它们的逻辑连上。”
    *   **AI**：*(查阅手册 4.3.2)* 自动调用 API 将生成器的 `Exit` 连至传送带的 `Entry`。
4.  **交互 3（逻辑）**：
    *   **用户**：“给生成器写逻辑：每隔 3 秒生成一个‘黄色方块’。”
    *   **AI**：*(识别主功能属性，参考手册 6.4.2.1)* 自动在行为树中配置 `Wait(3s)` 和 `CreateTwin(Yellow_Cube)`。
5.  **预览与微调**：
    *   **用户**：“方块生成太快了，改成长一点。”
    *   **AI**：“好的，已将 Wait 节点的间隔时间从 3s 改为 5s。”

## 五、 关键 API 开放清单 (给开发团队)

为实现上述 MVP 场景，Designer 必须向 AI 助手开放以下原子 API：

| 分类 | API 名称 (示意) | 说明 |
| :--- | :--- | :--- |
| **资源库** | `GetTemplateLibrary()` | 让 AI 知道有哪些模型可用 |
| **空间操作** | `CreateEntity(tmplID, pos, rot)` | 放置物体 |
| | `BatchTransform(entities[])` | 阵列、对齐、批量移动 |
| **属性操作** | `SetAttribute(entityID, attr, val)` | 修改速度、长度、颜色等 |
| **逻辑操作** | `AddBTNode(nodeType, parentID)` | 在行为树中加节点 |
| | `UpdateBTNodeParam(nodeID, data)` | 填写节点里的数值 |
| | `ConnectBTNodes(fromID, toID)` | 连线行为树 |
| **感知** | `GetCurrentSelection()` | 让 AI 知道用户在点哪儿 |

为了进一步细化每个对话的“正确率”，我需要确认：
1.  **行为树的底层结构**：它是完全图形化的，还是可以由脚本（如 Python/Lua）驱动？（如果支持脚本，AI 的效率会更高）。
2.  **资源库的元数据**：每个模板是否有清晰的描述？（例如：AI 需要知道“Conveyor”就是“传送带”）。
3.  **UI 反馈机制**：当 AI 在后台修改 API 时，我们是否需要给用户一个“确认执行”的按钮，还是直接实时修改？
