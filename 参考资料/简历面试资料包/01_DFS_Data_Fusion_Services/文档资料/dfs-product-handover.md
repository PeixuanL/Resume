## 1. 交接结论

本次交接不是单一项目或单一版本交接，而是 DFS 产品线整体交接。接收人需要接住 DFS 的产品边界、模块资产、当前问题、后续迭代路线，以及与 FactVerse、Designer、AI Agent、客户交付材料之间的协作关系。

DFS 当前应按“三层产品责任”理解：

| 层次 | 定位 | 接手后的核心任务 |
|---|---|---|
| DFS Lite | 快速数据接入、连接器配置、基础映射、基础质量、基础同步 | 从 demo 态推进到可交付态，优先补真实接入闭环、运行记录、错误态 |
| DFS Pro | 数据资产治理、方法资产、融合任务、治理编排、人工复核、审计权限 | 从高级 CRUD / 页面骨架推进到可执行、可追踪、可审核的治理闭环 |
| DFS + 平台融合 | 与 FactVerse / Designer / AI Agent 打通资源、孪生体、数据资产和智能分析 | 统一资源标识、沉淀 Agent Tool API、支撑自然语言找表和 AI 解释数据 |

接收人后续最重要的判断标准是：DFS 不能只看页面是否存在，而要看能力是否真实可运行、结果是否可观察、失败是否可解释、操作是否可审计、输出是否能沉淀为资产。

## 2. 产品边界

### 2.1 DFS Lite 边界

DFS Lite 面向轻量数据接入和最小真实闭环，主要解决“快速接入、快速试通、快速跑通一条数据链路”的问题。

Lite 负责：

| 模块 | 说明 |
|---|---|
| Connector onboarding | 创建连接器，配置数据源连接信息 |
| Test connection | 验证连接是否可用 |
| Schema discovery | 探测数据结构、字段、表结构或测点结构 |
| Field mapping | 字段映射、映射建议、映射校验 |
| Basic pipeline run | 创建并启动基础同步链路 |
| Basic quality dashboard | 基础质量规则、质量概览 |
| Sync history | 同步状态、同步历史、失败原因、重试记录 |

Lite 的成功标准：

| 标准 | 说明 |
|---|---|
| 快速接入 | 能低成本接入常见协议或数据源 |
| 快速验证 | 测试连接、Schema 探测、映射校验真实可用 |
| 快速运行 | 能真实启动同步任务，而不是只完成配置 |
| 可稳定交付 | 失败态、空态、运行态、重试、日志清晰 |

### 2.2 DFS Pro 边界

DFS Pro 面向组织级数据治理、融合、方法资产和长期运营，主要解决“数据资产如何被管理、复用、执行、审核、审计”的问题。

Pro 负责：

| 模块 | 说明 |
|---|---|
| Dataset Center | 数据资产中心，管理数据集、画像、预览、版本、血缘、责任人 |
| Method Library | 方法资产库，管理预处理、特征提取、计算方法、算法方法 |
| Fusion Task | 融合任务，负责数据融合、匹配、冲突处理、结果沉淀 |
| Governance Canvas | 治理画布 / Pipeline，负责治理流程编排、发布、运行、调试 |
| Review Queue | 人工复核、冲突审核、低置信度结果处理 |
| Audit Trail | 操作审计、执行审计、决策留痕 |
| RBAC / Sensitive Control | 数据集、方法、执行、发布、审计的权限与敏感字段控制 |
| Governance Metrics | 资产数量、方法复用率、任务成功率、冲突率、审核率等运营指标 |

Pro 的成功标准：

| 标准 | 说明 |
|---|---|
| 能执行 | Fusion Task / Governance Pipeline 有真实运行记录 |
| 能追踪 | 每次执行有 runId、状态、耗时、错误、输出 |
| 能复核 | 高风险、冲突、低置信度结果能进入人工处理 |
| 能审计 | 关键配置、发布、执行、处理动作有留痕 |
| 能资产化 | 数据集、方法、治理流程、结果可以长期沉淀和复用 |

### 2.3 不建议混淆的边界

| 易混淆点 | 建议边界 |
|---|---|
| Lite 和 Pro 都有数据集概念 | Lite 更偏接入和同步过程中的基础数据对象，Pro 更偏资产治理和运营 |
| 数据可视化和 BI | DFS 可提供基础看板和报表，但不建议把 DFS 定位成通用 BI 替代品 |
| 数据接入和数据治理 | 接入是入口，治理需要版本、血缘、权限、审核、审计、责任人 |
| AI Agent 和 DFS | Agent 是调用和解释层，DFS 需要先提供稳定的工具化 API 和运行态 |
| Demo 能力和交付能力 | Demo 可展示流程，交付必须有真实执行、错误处理、日志和验收标准 |

### 4.2 主要问题

| 问题 | 影响 | 建议动作 |
|---|---|---|
| Lite 前端存在 mock / fallback / 假成功路径 | 用户和研发容易误判真实能力 | P0 去掉关键 mock，失败时展示真实错误态 |
| 向导流程未完全打通真实执行链路 | 接入流程可能停留在配置层 | 打通 test → schema → mapping → create → start |
| 运行状态机和执行记录不足 | 无法定位失败和评估交付质量 | 建立 runId、状态、耗时、错误、数据量、重试 |
| Pro 偏高级 CRUD / 页面骨架 | 治理价值无法落地 | Fusion Task / Governance 必须真实运行 |
| 方法库尚未完全资产化 | 方法复用和治理难以沉淀 | 版本、样例、测试、性能、使用追踪 |
| 审核、审计、RBAC 不完整 | 企业级客户难以信任 | 建立 Review Queue、Audit Trail、权限边界 |
| Lite / Pro / 旧版本材料口径混用 | 对外表达和内部规划容易混乱 | 接手后需统一版本口径和售卖边界 |

## 5. 后续迭代路线

### 5.1 Phase 1：DFS Lite 最小真实闭环

目标：把 Lite 从 demo 态推进到可交付态。

| 优先级 | 工作包 | 内容 | 验收标准 |
|---|---|---|---|
| P0 | 去掉假闭环 | 移除连接器列表、SyncHistory、QualityDashboard 等关键 mock / fallback | API 失败展示明确错误态，不展示伪数据 |
| P0 | 真实向导闭环 | 打通 test connection、schema discovery、mapping、create pipeline、trigger sync | 可真实创建并启动一条同步链路 |
| P0 | 运行状态机 | connector / pipeline / sync 有状态流转 | 状态包含 draft、testing、ready、syncing、paused、failed |
| P0 | 同步历史真实化 | 同步记录来自真实后端 | 可查看耗时、数据量、失败原因、重试记录 |
| P1 | 后端聚合接口 | dashboard summary、sync history aggregation、connector health summary | 降低前端拼装逻辑，统一分页和筛选 |
| P1 | 连接器能力模型 | supportsTest、supportsSchemaDiscovery、supportsPreview 等能力字段 | 不同连接器能力边界清晰 |
| P2 | 模板市场和告警 | connector template marketplace、health alerts、pipeline preset | 产品化体验增强 |

### 5.2 Phase 2：DFS Pro 执行闭环

目标：把 Pro 从高级 CRUD 推进到可执行、可追踪、可治理。

| 优先级 | 工作包 | 内容 | 验收标准 |
|---|---|---|---|
| P0 | Fusion Task 执行系统 | run history、状态机、重试、冲突和失败记录 | 每个 Fusion Task 有真实运行记录 |
| P0 | Governance Canvas 执行化 | pipeline run、step records、node results、validate / publish / run | Pipeline 可以发布、运行、调试 |
| P0 | Method Library 资产化 | 方法版本、状态、样例输入、预期输出、测试 harness | 方法可测试、可追踪、可复用 |
| P0 | Pro UI 去假数据 | Dataset Center / Fusion Task / Governance 不依赖演示 fallback | 失败、空态、运行态可见 |
| P1 | Dataset Center 资产运营 | schema version、sample preview、profile、quality、ownership、lineage | 数据集不只是列表，而是资产中心 |
| P1 | Review / Approval | low-confidence queue、conflict queue、approval workflow | 冲突和高风险结果可人工处理 |
| P1 | Audit / RBAC | 数据集、方法、执行、发布、审计权限；敏感字段脱敏 | 企业级治理可信 |
| P2 | 差异化治理 | 方法 DSL、智能推荐、跨项目模板、策略调优 | 形成长期竞争力 |

### 5.3 Phase 3：DFS 与 FactVerse / AI Agent 融合

目标：让 DFS 成为平台级数据资产与智能执行能力，而不是孤立数据模块。

| 优先级 | 工作包 | 内容 | 验收标准 |
|---|---|---|---|
| P0 | 资源主数据同步 | tenant、organization、product、module、space、device、point 同步 | DFS 对象能关联平台资源 |
| P0 | 统一资源标识 | tenantId、productKey、moduleKey、resourceId、externalId 规则 | 跨系统对象不混乱 |
| P1 | DFS Tool API | 数据集、方法、任务、审核、审计能力开放给 Agent | Agent 可检索 DFS 资产 |
| P1 | 运行态解释 | Agent 读取并解释 run、review、audit 状态 | 用户能理解任务为何成功或失败 |
| P2 | 智能治理建议 | 基于失败率、冲突率、质量问题生成建议 | 从工具型产品升级为智能治理助手 |

## 6. 风险与待确认事项

### 6.1 产品风险

| 风险 | 等级 | 说明 | 建议处理 |
|---|---|---|---|
| 页面存在但执行不真实 | P0 | mock、fallback、假成功会影响交付判断 | 作为接手后第一轮排查重点 |

## 9. Definition of Done

### 9.1 DFS Lite P0 完成标准

| 标准 | 说明 |
|---|---|
| 无关键 fake success | 核心流程不再用假数据伪装成功 |
| 向导真实闭环 | test、schema、mapping、create、start 全链路真实执行 |
| 有运行记录 | 每次同步有状态、耗时、数据量、失败原因 |
| 有失败态 | API 失败、连接失败、同步失败都有明确提示和重试路径 |
| 可验收 | 测试人员可用真实数据源完成验收 |

### 9.2 DFS Pro P0 完成标准

| 标准 | 说明 |
|---|---|
| Fusion Task 有真实执行 | 有 run history、状态流转、失败分类 |
| Method Library 可资产化 | 方法有版本、样例、测试和使用追踪 |
| Governance Pipeline 可运行 | 支持 validate、publish、run、debug |
| Review / Audit 有最小闭环 | 冲突和高风险结果可复核，关键动作有审计 |
| UI 反映后端真实状态 | 不用演示 fallback 掩盖失败 |
