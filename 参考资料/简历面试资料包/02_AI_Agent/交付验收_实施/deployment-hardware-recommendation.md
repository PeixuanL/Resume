# 平台本地部署硬件建议（POC 优化版）

## 1. 本次建议的适用边界

本建议仅针对本次客户 `POC` 范围，目标是：

- 能完整部署并演示整套平台：`frontend + backend + postgres/timescaledb + redis + ai-engine + 本地小模型`
- 能支撑 `22 台设备`、`2 年历史数据` 的导入、查询、分析与演示
- 支撑本地小模型问答、解释生成、轻量 Agent 调用
- 以 `单机一体部署` 为主，优先保证“能跑通、能演示、成本可控”

本建议**不以正式生产环境**为目标，因此默认不包含以下诉求：

- 高并发、多团队同时访问
- 双机/多机高可用
- 多模型并发推理
- 32B 及以上大模型本地部署
- 大规模知识库、长上下文、多租户正式运营

结论上，本次 POC **没有必要**按照正式生产口径采购 `48GB 显存 GPU + 256GB/512GB RAM + 双机/三机分层架构`。

## 2. 为什么这次可以明显降配

### 2.1 数据量级并不大，数据库不是主要瓶颈

当前已知范围为：

- 设备数：`22 台`
- 历史范围：`2 年`
- 采样频率：待最终确认

按“每台设备每个采样点产生 1 条记录”的粗略估算：

- 若为 `1 分钟 1 条`：约 `2312 万条`
- 若为 `10 秒 1 条`：约 `1.39 亿条`

对于本次 POC，这个量级使用单机 `Postgres/Timescale + NVMe SSD` 完全可承载。  
真正决定成本的主要不是数据库，而是**本地模型推理所需的 GPU 显存**。

### 2.2 本次成本核心在 GPU，不在 CPU

从当前部署架构看：

- `frontend / backend / redis` 对算力要求并不高
- `postgres/timescaledb` 更依赖内存和 SSD
- `ai-engine + 本地 LLM` 才是最吃资源的部分

因此，POC 阶段应优先把预算放在：

- `1 张合适的 GPU`
- `足够的内存`
- `稳定的 NVMe SSD`

而不是过早采购：

- 双 GPU
- 48GB 显存专业卡
- 256GB 以上内存
- 10GbE/25GbE 网络
- 独立应用机 + 独立 GPU 机

## 3. 优化后的推荐结论

### 3.1 最推荐方案：单机一体式 POC 工作站

建议作为这次客户 POC 的默认方案：

| 维度 | 建议配置 |
|---|---|
| 部署形态 | `1 台单机工作站 / 塔式服务器` |
| CPU | `16 核` 左右即可，建议不低于 `12 核` |
| 内存 | `64GB` 起步，建议 `128GB` 更稳妥 |
| GPU | `1 张 24GB 显存 GPU` |
| 系统盘 | `1TB NVMe SSD` |
| 数据盘 | `2TB NVMe SSD` 起步，高频采样建议 `4TB NVMe SSD` |
| 模型盘 | 可与数据盘合并；更稳妥可单独预留 `1TB NVMe SSD` |
| 网络 | `1GbE` 即可 |
| 操作系统 | `Ubuntu 22.04 LTS` |
| 部署方式 | `Docker Compose` |

这套配置已经足以覆盖：

- 平台基础服务运行
- 22 台设备、2 年数据导入与查询
- 本地 `7B - 14B` 量化模型推理
- 问答、解释生成、轻量 Agent 演示
- 1 到 3 人配合演示的低并发场景

### 3.2 GPU 选型建议

本次 POC 的**关键门槛建议设在 24GB 显存**，原因是：

- 24GB 显存对本地 `7B - 14B` 小模型更稳妥
- 更容易兼顾问答、解释、RAG 和简单 Agent 流程
- 根据 Ollama 官方文档，`24GiB - 48GiB` 显存默认可支持更大的上下文窗口，而 `24GiB` 以下默认更保守

推荐优先级如下：

| 选项 | 推荐度 | 说明 |
|---|---|---|
| `RTX 4500 Ada 24GB` | **推荐** | 专业卡，交付口径更稳，适合客户侧本地部署 |
| `L4 24GB` | 推荐 | 低功耗数据中心卡，适合机架式服务器 |
| `RTX 4090 24GB` | 可选 | 性价比高，但属于消费级显卡，仅在客户接受消费卡时推荐 |
| `RTX 4000 Ada 20GB` | 不建议作为默认方案 | 能跑更小模型，但对上下文和模型选择余量偏小 |
| `RTX 6000 Ada 48GB / L40S 48GB` | 本次 POC 不必默认采购 | 更适合后续正式扩展，不是这次最低成本达标解 |

### 3.3 推荐给客户的两档配置

#### 方案 A：最低可行配置

适用于：

- 本次只要求演示跑通
- 并发很低
- 本地模型以 `7B` 为主，`14B` 为辅

建议配置：

- CPU：`12-16 核`
- 内存：`64GB`
- GPU：`1 x 24GB`
- 系统盘：`1TB NVMe SSD`
- 数据盘：`1TB - 2TB NVMe SSD`
- 模型盘：可与数据盘共用；若预算允许，单独增加 `1TB SSD/NVMe`
- 网络：`1GbE`

建议表述：

> 该配置可满足本次 POC 的平台演示、本地小模型问答和 22 台设备两年数据验证，属于成本优先的最低可行配置。

#### 方案 B：稳妥推荐配置

适用于：

- 希望演示更顺畅
- 可能同时进行数据导入、查询、问答与解释生成
- 需要给客户留一点余量，避免现场表现受限

建议配置：

- CPU：`16 核`
- 内存：`128GB`
- GPU：`1 x 24GB`
- 系统盘：`1TB NVMe SSD`
- 数据盘：`2TB NVMe SSD`
- 模型盘：建议单独 `1TB NVMe SSD`，或与数据盘合并为 `1 块 4TB NVMe SSD`
- 网络：`1GbE`

建议表述：

> 该配置是本次 POC 的推荐方案，在不明显抬高预算的前提下，兼顾部署稳定性、数据导入效率和本地模型演示效果。

#### 方案 C：高频采样场景推荐配置

适用于：

- 采样频率较高，例如接近 `1 秒级` 或更高
- 每台设备关联测点较多，历史数据增长较快
- 客户明确希望一次采购就预留一定数据空间

建议配置：

- CPU：`16 核`
- 内存：`128GB`
- GPU：`1 x 24GB`
- 系统盘：`2 x 960GB SSD RAID1` 或 `1TB NVMe SSD`
- 数据盘：`2 x 3.84TB NVMe RAID1` 或 `1 x 4TB NVMe SSD`
- 模型盘：建议单独 `1TB - 2TB NVMe SSD`
- 网络：`1GbE`

建议表述：

> 若客户担心后续数据采样频率偏高，建议在保持单机一体和单张 24GB GPU 不变的前提下，优先增加数据盘容量，并将系统盘、数据盘、模型盘分离，以提高数据导入和历史查询的稳定性。

### 3.4 数据盘与硬盘配置建议

推荐给客户时，建议明确区分三类存储：

- 系统盘：安装操作系统、Docker、平台服务，建议 `1TB` 起
- 数据盘：存放 `Postgres/Timescale` 历史数据，建议 `2TB` 起；高频采样建议 `4TB` 起
- 模型盘：存放本地模型、向量索引、缓存文件，建议 `1TB` 左右

如果客户担心后续采样频率较高，建议优先采用以下方式，而不是先升级 GPU：

- 将系统盘和数据盘分离
- 数据盘优先使用 `NVMe SSD`
- 若客户对磁盘可靠性有要求，数据盘可采用 `2 盘 RAID1`
- 若只是 POC 且更关注成本，数据盘可先用 `单块 4TB NVMe SSD`

对客户的建议口径可简化为：

- 常规 POC：`1TB 系统盘 + 2TB 数据盘`
- 高频采样 POC：`1TB 系统盘 + 4TB 数据盘 + 1TB 模型盘`

## 4. 明确不建议本次 POC 默认采购的配置

以下配置更偏正式生产或后续扩展，本次 POC 不建议默认要求客户采购：

- `2 张 GPU`
- `48GB` 显存专业卡作为起配
- `256GB / 512GB RAM`
- 独立 `应用机 + GPU 机 + 数据库机`
- `10GbE / 25GbE` 网络作为前置条件
- 大容量 `RAID10` 数据盘作为起配

原因很简单：

- 本次数据规模和并发规模还不足以支撑这些成本
- 这些配置更适合正式生产、长期运行和未来扩展
- 作为 POC 门槛会显著拉高客户预算，影响项目推进

## 5. 什么时候需要把配置再往上提

若后续确认出现以下情况，再建议升配：

- 数据频率非常高，例如接近 `1 秒级`
- 每台设备关联大量传感器，导致总记录数显著上升
- 现场需要多人同时使用
- 本地模型希望从 `7B/14B` 升到 `32B`
- 希望引入更大的离线知识库、长上下文、多轮 Agent

对应升配方向：

- 内存从 `64GB` 升到 `128GB`
- 数据盘从 `2TB` 升到 `4TB` 或 `2 x 3.84TB RAID1`
- 模型盘单独增加 `1TB - 2TB SSD`
- GPU 从 `24GB` 升到 `48GB`

## 6. 建议给客户的最终说法

如果客户关注成本，建议直接采用下面这段结论：

> 结合本次 POC 的实际范围，平台只需支撑 22 台设备、2 年历史数据、本地小模型推理和现场演示，不建议按正式生产口径采购高配 AI 服务器。  
> 本次建议采用单机一体部署，配置 `16 核 CPU + 64GB/128GB 内存 + 1 张 24GB GPU + 1TB 系统盘 + 2TB 数据盘` 即可满足演示和跑通要求。  
> 若客户担心采样频率较高，建议调整为 `1TB 系统盘 + 4TB 数据盘 + 1TB 模型盘`，优先通过增加存储容量和分盘来提升数据承载能力。  
> 若客户更关注交付稳定性，推荐选择 `128GB 内存`；若更关注预算控制，`64GB 内存` 也可作为最低可行方案。
