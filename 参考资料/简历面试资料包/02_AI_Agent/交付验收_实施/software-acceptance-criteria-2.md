# 软件验收条款（PdM 一期对客协议版）

## 1. 文档说明

本文档依据 [software-development-plan-delivery.md](C:/work/factverse-ai-agent/docs/customer/pdm-phase1/software-development-plan-delivery.md:1) 整理，用于明确 PdM 一期软件系统的验收范围、验收前提、验收点、验收标准、验收证据及验收结论判定原则。

本文档面向甲乙双方项目管理、业务确认、实施交付和阶段验收场景使用，可作为阶段验收会议、书面确认和遗留问题闭环的依据性材料。

## 2. 验收目标

PdM 一期软件验收的目标是确认系统已按照约定范围完成交付，并具备以下能力：

1. 能够完成现场数据接入、设备对象关联、分析处理、结果展示和成果输出的完整业务闭环。
2. 能够支撑甲方开展设备状态查看、异常识别、趋势分析、报表导出和运行维护工作。
3. 能够满足一期试运行和正式交付阶段的基本可用性、可管理性和可追溯性要求。

## 3. 验收范围

本次验收范围限于 PdM 一期软件开发计划中明确约定的交付内容，包含以下八个业务模块：

1. 统一门户与运行总览
2. 设备资产管理
3. 实时监测与数据采集
4. 告警预警管理
5. 数据接入与系统集成
6. 统计分析与辅助决策
7. 报表报告与成果输出
8. 系统配置与运维管理

本次验收不包括以下内容：

1. 故障诊断
2. 专家建议
3. 工单联动
4. 其他未在 PdM 一期开发计划中明确列示的扩展能力

## 4. 验收前提

验收工作开展前，应满足以下前提条件：

1. 系统已完成部署，前端服务、后端服务、AI Engine、数据库及关键接口处于可访问状态。
2. 甲方已提供用于验收的设备、传感器接入环境，或双方共同确认可替代的模拟数据源与测试数据集。
3. 双方已确认验收账号、权限范围、测试设备清单、测试时间范围及演示路径。
4. 验收所需的关键页面、接口、日志、报表和导出能力已准备完毕。
5. 双方已明确本次验收以“一期范围内功能可用、链路可跑通、结果可验证、资料可留存”为主要判定依据。
6. 对依赖现场硬件、网络、网关、MQTT 链路、第三方接口或甲方基础环境的验收项，应以验收当日实际具备的环境条件为前提。
7. 如现场真实环境暂不具备完整联调条件，经双方确认后，可采用模拟数据源、样例数据集或预置演示数据完成对应验收。
8. 因甲方现场网络策略、设备状态、接口权限、硬件故障、第三方链路异常或测试数据不足导致无法完成的验证，不单独视为乙方软件交付未完成。

## 5. 验收原则

1. 范围一致原则  
   验收内容严格以一期开发计划中已定义的功能范围和交付边界为准。

2. 结果导向原则  
   验收以系统可实际演示、可操作、可查询、可导出、可留痕的结果为准，不以底层代码实现方式作为验收依据。

3. 闭环验证原则  
   验收不仅验证单个页面或单项功能，还验证“接入、分析、展示、输出、运维”之间的完整承接关系。

4. 证据留存原则  
   验收过程中的关键结果应形成截图、导出文件、日志记录、会议纪要或测试记录，作为双方确认依据。

5. 边界控制原则  
   对于未在一期范围内明确约定的内容，不纳入本次通过与否的判定标准。

6. 环境依赖原则  
   对依赖现场硬件、网络、第三方链路和历史数据积累的功能，应在约定验收环境和约定测试数据条件下进行验证，不延伸解释为所有现场条件下的持续效果承诺。

7. 展示能力与效果承诺区分原则  
   对趋势预测、异常识别、综合评分、降噪统计、文字结论等分析类能力，本次验收重点验证系统具备结果生成、展示、查询和导出能力；除非双方另有书面约定，不将识别准确率、预测准确度或长期运行效果作为默认验收依据。

## 6. 功能验收点与验收标准

本章所列各项验收标准，均以双方确认的验收环境、测试数据、账号权限、参数配置和演示路径为前提；如涉及现场链路、第三方系统或历史数据积累，则以验收时实际可获得的条件为准。

### 6.1 统一门户与运行总览模块

| 验收点 | 验收标准 |
|------|------|
| 门户首页展示 | 用户登录后可正常进入统一门户首页，首页能够展示约定范围内的总览信息，并可作为其他模块的统一入口。 |
| 设备健康等级总览 | 系统具备基于 ISO 10816 规则展示设备健康等级分布的能力，并能区分正常设备、关注设备和高风险设备。 |
| 异常概况展示 | 系统能够展示当前统计周期内的异常数量、等级分布及风险概况；在相同统计口径和时间范围下，与异常明细数据保持一致。 |
| 高频关注设备展示 | 系统能够按当前规则配置列示异常频次较高、综合评分偏高或趋势持续退化的重点设备。 |
| 趋势摘要展示 | 系统具备基于已有历史数据生成并展示设备健康变化趋势、等级变化方向或预测退化信息的能力；该项验收不默认包含预测准确度承诺。 |
| 异常抑制统计展示 | 系统能够按当前已启用的抑制规则展示高置信度异常数量、低置信度抑制数量或降噪比例等统计信息。 |
| 快捷导航与联动跳转 | 用户可由首页跳转至设备、异常、分析、报表和配置等页面，且在相同筛选条件下保持对象范围基本一致。 |
| 条件筛选 | 系统支持按等级、设备、状态或时间范围筛选总览内容，筛选结果在同一统计口径下应与相应明细页面保持一致。 |

### 6.2 设备资产管理模块

| 验收点 | 验收标准 |
|------|------|
| 设备台账管理 | 系统能够建立并维护一期范围内设备台账，支持设备名称、编号、类型、安装位置、区域归属等信息的新增、查看和修改。 |
| 设备分类与编码 | 系统能够按设备类型进行分类，并支持统一编码或标识管理。 |
| 设备基础属性维护 | 系统可维护设备额定参数、运行状态、品牌型号、关键部件或安装信息等基础属性。 |
| 设备区域归属管理 | 系统可将设备与区域、产线或安装位置关联，并支持按区域维度查看设备信息。 |
| 设备与传感器绑定 | 系统可建立设备、传感器、测点和通道之间的映射关系；在映射配置正确的前提下，后续采集和分析结果可关联到对应设备对象。 |
| 单设备查询 | 系统支持按单设备查看当前健康等级、异常状态、近期趋势和已有历史分析结果。 |
| 列表检索与筛选 | 系统支持按名称、类型、状态、风险等级和异常情况检索与筛选设备。 |
| 重点设备标识 | 系统支持对关键设备、高风险设备或重点观测设备进行标识并在相关页面突出展示。 |

### 6.3 实时监测与数据采集模块

| 验收点 | 验收标准 |
|------|------|
| 监测对象管理 | 系统能够以设备为核心建立监测对象，并明确其对应的传感器、测点和采集周期。 |
| 多通道数据管理 | 系统支持管理振动、温度等多通道数据，并能区分不同指标和通道类型。 |
| 实时数据采集 | 在接入链路正常、数据源可用的前提下，系统能够接收现场上传或模拟输入的数据，并更新设备当前状态。 |
| 实时监测展示 | 系统能够展示单设备当前振动值、温度值、状态摘要和最近读数。 |
| 历史数据查询 | 系统能够按设备、时间范围和通道类型查询历史数据。 |
| 趋势曲线展示 | 系统能够以图表方式展示关键指标随时间的变化过程。 |
| 采集状态监控 | 在已启用状态监控和有对应输入信号的前提下，系统能够识别传感器中断、最近数据超时或采集异常等状态，并提供可查看提示。 |
| 数据质量校验 | 系统能够对缺失值、异常值、格式错误和时序不连续等问题进行识别和提示；相关异常不应直接导致核心页面整体不可访问。 |

### 6.4 告警预警管理模块

| 验收点 | 验收标准 |
|------|------|
| 预警触发规则 | 系统能够按双方确认并已启用的一期规则配置生成异常事件，并区分正式异常、观察对象和抑制结果。 |
| ISO 等级映射展示 | 系统能够展示设备速度 RMS 与 ISO 10816 A/B/C/D 等级的对应关系。 |
| 综合异常分展示 | 系统能够展示由多种分析结果汇总形成的综合异常评分或置信度结果；该项验收以结果可生成、可展示、可查询为准。 |
| 异常事件列表 | 系统能够以列表形式展示异常事件，并包含设备名称、异常类型、等级、时间、状态等基础字段。 |
| 观察对象管理 | 如已启用观察区间规则，系统能够对处于观察区间的设备或结果进行单独展示和跟踪。 |
| 低置信度抑制统计 | 如已启用低置信度抑制规则，系统能够对被抑制的低置信度事件进行统计展示。 |
| 筛选与检索 | 系统支持按设备、等级、状态、异常类型和时间范围筛选异常事件。 |
| 异常详情查看 | 系统能够在异常详情中展示等级、趋势、特征、综合评分和说明信息中的主要内容，便于用户理解异常识别依据。 |

### 6.5 数据接入与系统集成模块

| 验收点 | 验收标准 |
|------|------|
| MQTT 或等效数据接入 | 在约定接入环境或双方确认的模拟环境下，系统能够接收来自 LoRaWAN、MQTT 或模拟数据源的数据。 |
| 设备与通道映射配置 | 系统能够配置设备、传感器、安装点位和通道类型之间的映射关系。 |
| 数据格式转换与入库 | 系统能够将接入数据转换为平台统一结构并完成入库，入库结果可通过页面、接口或数据库记录进行验证。 |
| AI 分析接口调用 | 系统能够将整理后的数据传递给 AI Engine 或等效分析服务，并返回可用于页面展示、查询或存储的结构化分析结果。 |
| 接入结果校验 | 系统能够校验设备编号、时间戳、通道名称、数值格式和入库结果，确保接入数据在约定口径下完整、可用、可追溯。 |
| 接入任务调度 | 系统能够按设定周期执行采集、转换、分析和结果回写流程；调度执行结果可通过任务状态、日志或记录进行查看。 |
| 接入异常监控 | 对接入失败、消息中断、映射错误、分析调用失败等情况，系统能够保留状态记录或日志。 |
| 联调支撑 | 在实施或验收阶段，乙方提供设备接入、通道预览、数据源配置和结果验证等联调支撑；涉及甲方现场网络、硬件、第三方接口或权限开通的内容，由甲方配合提供条件。 |

### 6.6 统计分析与辅助决策模块

| 验收点 | 验收标准 |
|------|------|
| ISO 10816 分级分析 | 系统能够基于速度 RMS 数据展示设备 A/B/C/D 分级结果。 |
| 轴间关联分析 | 系统能够展示 X/Y/Z 轴之间的特征差异、比值关系或相关异常分析结果。 |
| 温度与振动关联分析 | 在存在对应历史数据和分析结果的前提下，系统能够展示温度变化与振动变化之间的关联结果或辅助解释信息。 |
| 趋势异常检测 | 基于已有历史序列和当前规则配置，系统能够识别趋势持续升高、均值漂移、短期偏离等异常现象，并在页面中展示。 |
| 综合异常评分 | 系统能够输出综合异常分值、等级或置信度结果，作为统一比较口径；该项验收不默认包含评分准确性承诺。 |
| 单设备健康趋势分析 | 系统能够展示单设备一段时期内的健康变化曲线、异常变化点或预计等级变化信息；预计等级变化用于结果展示和辅助判断，不单独作为预测准确度验收依据。 |
| 设备对比分析 | 在已配置设备分组、具备可比数据的前提下，系统能够支持同类设备或设备组之间的关键指标或分析结果对比。 |
| 结果复核查看 | 系统能够查看原始数据、分析结果、评分构成或特征指标中的主要信息，以支持人工复核。 |

### 6.7 报表报告与成果输出模块

| 验收点 | 验收标准 |
|------|------|
| 设备健康概览报表 | 系统能够输出整体健康等级分布、重点设备清单和阶段状态概览。 |
| 异常事件统计报表 | 系统能够输出异常事件数量、等级分布、设备分布和时间分布等统计结果。 |
| 单设备分析报告 | 系统能够输出单设备关键指标、ISO 等级、异常得分和趋势判断结果等内容。 |
| 趋势结果输出 | 系统能够输出设备健康趋势、变化区间和预计等级变化类结果；相关结果用于展示和汇报，不默认构成预测准确度承诺。 |
| 图表汇总输出 | 系统能够将趋势图、分布图或统计图作为成果的一部分进行输出。 |
| 文字结论输出 | 系统能够基于预设模板输出标准化文字说明，用于汇报、归档或阶段说明；该项不等同于专家诊断意见。 |
| 多格式导出 | 系统能够按双方确认的报表类型导出结构化结果或成果文件。 |
| 成果归档管理 | 已生成报表和分析结果可按时间或类型进行归档，或通过约定的文件留存方式保存，并支持再次查询或调取。 |

### 6.8 系统配置与运维管理模块

| 验收点 | 验收标准 |
|------|------|
| 系统参数配置 | 系统能够维护系统基础参数、展示参数和设备级运行模式。 |
| 阈值与分析参数配置 | 系统能够维护 ISO 分级口径、权重参数、预警阈值和相关分析参数。 |
| 接入配置管理 | 系统能够维护数据源类型、采集配置、映射关系和接入参数。 |
| 任务调度管理 | 系统能够查看或维护数据接入、异常识别、结果更新和报表输出任务的执行状态。 |
| 运行监控与健康检查 | 系统具备对前端、后端、AI Engine、数据库及关键接口进行运行监控或健康检查的基础能力，可通过页面、接口或运维工具进行验证。 |
| 日志管理与问题追踪 | 系统能够保留接入、分析、查询和配置相关日志，并支持通过日志文件、日志平台或数据库记录进行问题追踪。 |
| 备份与恢复支撑 | 系统具备关键配置和业务数据的备份与恢复支撑能力；可通过方案说明、脚本、执行记录或恢复演示中的一种或多种方式作为验收依据。 |
| 运维巡检支撑 | 系统能够支撑日常巡检、异常排查、配置复核和运行验证工作；该项验收以具备基础运维支撑能力为准，不默认延伸为完整运维平台承诺。 |

## 7. 端到端链路验收

为避免仅对页面功能进行局部验收，双方应至少完成以下端到端链路验证：

如现场真实链路暂不具备完整验证条件，经双方确认后，可采用模拟数据、样例数据或预置结果完成相应环节验证，并在验收记录中注明验证方式。

1. 数据上送链路验证  
   传感器或模拟数据源发送有效数据后，系统能够完成接收、转换、入库和设备映射；如涉及现场链路异常，应结合日志和替代数据验证结果综合判定。

2. 分析处理链路验证  
   对符合分析条件的数据，系统能够调用分析服务并生成结构化分析结果；该项验收以结果生成和链路可验证为准，不默认包含识别效果指标承诺。

3. 异常识别链路验证  
   当输入数据满足已启用的一期异常触发条件时，系统能够生成对应异常事件、观察对象或抑制结果。

4. 结果展示链路验证  
   分析和异常结果能够在总览页、设备详情页、异常页和相关报表中被查询和展示。

5. 输出归档链路验证  
   相关结果能够形成报表、导出文件或归档成果，并保留查询依据。

6. 运维追踪链路验证  
   对于关键接入、分析和异常处理步骤，系统能够保留日志、状态记录或追踪标识中的一种或多种，支持事后排查。

## 8. 非功能验收要求

### 8.1 登录认证与权限控制

1. 系统应支持统一登录认证机制或本系统内置认证机制。
2. 不同角色或岗位的用户应在授权范围内访问页面、数据和配置项；如需对接甲方统一身份体系或外部权限系统，应以双方另行确认的集成范围为准。
3. 对未授权访问行为，系统应给出明确提示。

### 8.2 错误提示与异常处理

1. 系统应对参数错误、认证错误、权限错误、资源不存在、接入失败、分析失败和服务异常给出明确提示。
2. 对普通用户展示的提示信息应避免直接暴露底层堆栈、内部路径和敏感实现细节。
3. 对运维人员应保留必要的追踪编号、日志信息或诊断线索。

### 8.3 日志与审计追踪

1. 系统应对关键访问、配置变更、接入任务、分析任务和异常处理进行日志记录。
2. 关键请求应具备唯一追踪标识或等效追溯能力。
3. 日志应支持按时间、设备、模块或异常类型进行查询和定位；查询方式可通过日志文件、数据库记录或日志平台实现。

### 8.4 健康检查与运行监控

1. 系统应具备前端、后端、AI Engine、数据库及关键接口的健康检查能力。
2. 系统应具备对接入任务、分析任务和关键接口状态的监控能力。
3. 当关键服务异常时，运维人员应能够识别异常状态并进行排查。

### 8.5 备份与恢复

1. 系统应支持业务数据库定期备份和关键配置留存。
2. 系统应支持在数据损坏、误操作或服务异常情况下开展恢复操作，或具备经验证的恢复方案与执行记录。
3. 双方可通过恢复演示、恢复记录、恢复脚本或恢复方案说明作为验收依据。

### 8.6 发布与升级维护

1. 系统应采用可重复执行的标准化发布方式。
2. 发布前应完成配置检查和服务检查。
3. 发布后应完成健康验证和核心功能验证。

## 9. 验收证据

以下材料可作为验收依据和归档材料：

1. 验收会议纪要或功能演示记录
2. 系统页面截图
3. 测试数据及对应结果截图
4. 报表导出样例
5. 健康检查结果
6. 日志追踪记录
7. 配置维护记录
8. 备份恢复说明、演练记录或恢复结果
9. 遗留问题清单及整改计划
10. 验收环境说明、数据来源说明或甲方配合条件记录

## 10. 验收结论判定

验收结论判定时，应优先以核心业务闭环、主要功能可用性和主要资料留存情况作为依据。与界面样式、非关键字段展示、非核心统计口径微调、现场外部条件不足相关的问题，可根据影响程度纳入遗留问题清单，不直接作为否决整体验收的唯一依据。

1. 通过验收  
   验收范围内主要功能项已完成演示，满足本文件约定标准，且关键业务闭环可跑通，视为通过验收。

2. 有条件通过验收  
   如存在少量不影响一期整体交付目标和核心链路运行的轻微问题，经双方确认形成遗留问题清单，并明确整改责任人与完成时限后，可视为有条件通过验收。

3. 不通过验收  
   如存在核心模块不可用、关键链路无法跑通、主要功能与一期约定范围严重不符，或系统无法支撑基本运行使用的情况，则视为不通过验收。

## 11. 遗留问题处理原则

1. 验收过程中发现的问题应形成书面遗留问题清单。
2. 每项遗留问题应明确问题描述、影响范围、整改措施、责任人和整改时限。
3. 遗留问题整改完成后，可通过补充验证、补充演示或复验方式确认关闭。
4. 遗留问题关闭结果应形成书面记录，并纳入交付归档材料。
