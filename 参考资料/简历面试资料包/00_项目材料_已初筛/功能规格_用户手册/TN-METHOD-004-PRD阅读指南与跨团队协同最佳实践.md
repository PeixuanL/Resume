# PRD阅读指南：从需求到代码的AI原生实践

> [!abstract] 核心摘要
> 本指南面向开发团队（前端Vue/后端Java/Unity），解决AI原生产品团队与传统研发团队协作的摩擦。核心目标：让研发团队掌握"读PRD→理解场景→设计技术→指导AI编码"的完整链路，减少对"更详细PRD"的依赖，提升与产品经理的协作效率。包含Codex提示词模板、Human-in-Loop规范、不同角色的分工侧重。

---

## 一、为什么需要这份指南

### 1.1 当前的协作摩擦

你们可能已经感受到这种张力：

| 产品团队 | 研发团队 | 结果 |
|----------|----------|------|
| 用AI快速产出PRD+原型 | 要求"更详细的PRD和功能清单" | 效率瓶颈 |
| 基于用户场景思考 | 基于功能点实现 | 需求理解偏差 |
| AI-native工作流 | 仅将AI用于代码补全 | 能力代差 |
| 关注"为什么做" | 关注"怎么做" | 目标不一致 |

**根本问题**：研发团队尚未完成向AI-native的转型，仍在用传统瀑布思维（详细需求→开发→交付）应对AI时代的敏捷协作。

### 1.2 AI-native研发团队的标准

一个真正的AI-native研发团队应该：

1. **理解用户场景**，而非仅理解功能清单
2. **用AI辅助技术设计**，而非仅用于代码生成
3. **主动补全信息缺口**，而非等待详细文档
4. **与产品共创**，而非被动接收需求

### 1.3 本指南的目标

- 教会团队如何**高效阅读AI产出的PRD**
- 提供**从PRD到技术设计**的转换方法
- 给出**指导Codex完成开发**的提示词模板
- 建立**Human-in-Loop**的协作规范
- 针对不同角色（Leader/普通开发）提供差异化指导

## 二、PRD阅读方法论

### 2.1 PRD结构速览

以典型的B端产品PRD为例，现代AI产出的PRD通常包含：

```
PRD结构
├── 产品概述（定位、价值主张、差异化）
├── 用户画像与痛点（Persona、场景、痛点矩阵）
├── 功能设计（架构、核心循环、模块详情）
├── 原型与交互（流程图、界面说明）
└── 数据与指标（埋点、成功指标）
```

> 详见附录中的"PRD模板与示例说明"章节。

### 2.2 研发视角的阅读顺序

**不要从第一页开始读！** 推荐阅读顺序：

#### 第一轮：5分钟速读（建立上下文）

1. **产品概述** → 理解"这是什么产品、解决什么问题"
2. **用户画像** → 理解"为谁做、他们的痛点是什么"
3. **功能架构图** → 理解"有哪些模块、模块间关系"

**目标**：能向他人用一句话描述产品，识别核心用户场景。

#### 第二轮：15分钟精读（提取技术要素）

4. **核心循环** → 识别数据流、状态流转
5. **功能模块详情** → 提取接口需求、业务规则
6. **原型/交互** → 理解用户操作流程

**目标**：能画出系统边界，识别关键数据实体。

#### 第三轮：针对性研读（澄清疑问）

7. **不确定的功能点** → 记录疑问
8. **跨模块交互** → 确认接口契约
9. **非功能需求** → 性能、安全、兼容性

**目标**：形成技术设计初稿，列出待确认问题。

### 2.3 不同角色的阅读重点

| 角色               | 第一轮重点     | 第二轮重点     | 第三轮重点      |
| ---------------- | --------- | --------- | ---------- |
| **前端Leader**     | 用户画像、价值主张 | 交互流程、状态管理 | 组件边界、API契约 |
| **前端开发**         | 用户场景、痛点   | 页面结构、交互细节 | 具体实现疑问     |
| **后端Leader**     | 核心循环、数据流  | 数据实体、业务规则 | 架构设计、接口设计  |
| **后端开发**         | 数据实体      | 业务逻辑      | 具体接口实现     |
| **Unity Leader** | 用户场景、交互流程 | 3D/2D场景需求 | 资源管理、性能要求  |
| **Unity开发**      | 场景描述      | 交互逻辑      | 具体功能实现     |

## 三、从PRD到技术设计的转换

### 3.1 技术设计的核心问题

拿到PRD后，技术设计需要回答：

1. **系统边界**：哪些在系统内，哪些在外部？
2. **数据模型**：核心实体有哪些，关系如何？
3. **接口契约**：输入输出是什么，错误如何处理？
4. **状态流转**：业务流程的状态机如何设计？
5. **技术选型**：用什么技术栈，为什么？
6. **风险点**：哪里容易出问题，如何防范？

### 3.2 技术设计文档模板

#### 模板结构

```markdown
## 技术设计文档

### 1. 背景与目标
- 对应PRD章节：[链接]
- 本次设计目标：[一句话描述]
- 成功标准：[如何验证设计成功]

### 2. 系统边界
[系统边界图 - 可用Mermaid或文字描述]
- 内部系统：
- 外部依赖：
- 用户交互：

### 3. 数据模型
[ER图或实体关系描述]

核心实体：
| 实体 | 字段 | 类型 | 说明 |
|------|------|------|------|

实体关系：
- A 1:N B
- C 1:1 D

### 4. 接口设计
#### API列表
| 接口 | 方法 | 路径 | 描述 |
|------|------|------|------|

#### 详细契约
[每个接口的详细输入输出、错误码]

### 5. 状态机
[核心业务流程的状态流转]

### 6. 技术选型
| 决策点 | 选项A | 选项B | 选择 | 理由 |
|--------|-------|-------|------|------|

### 7. 风险与应对
| 风险 | 影响 | 可能性 | 应对措施 |
|------|------|--------|----------|

### 8. 待确认问题
- [ ] 问题1
- [ ] 问题2
```

### 3.3 提取技术要素的清单

从PRD中提取以下要素填入模板：

#### 数据实体提取

```
PRD中的线索：
- 名词（用户、任务、订单）→ 可能是实体
- 属性描述（用户有姓名、邮箱）→ 实体字段
- 关系描述（用户有多个任务）→ 实体关系
```

#### 接口提取

```
PRD中的线索：
- "用户可以..." → 可能对应一个操作/接口
- "系统需要..." → 可能是内部服务调用
- "展示..." → 可能是查询接口
```

#### 状态流转提取

```
PRD中的线索：
- "当...时" → 状态变化触发条件
- "变为..." → 目标状态
- "如果...则..." → 分支逻辑
```

### 3.4 示例：B端审批流程模块的实体提取

以典型的B端审批流程功能为例，从PRD中提取：

| PRD原文                               | 技术解读                                                                                                                        |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| "用户提交报销申请，填写金额、类别、事由"               | 实体：ExpenseRequest（applicant_id, amount, category, reason, created_at）                                                       |
| "系统根据金额自动选择审批人，小额由部门经理审批，大额需财务总监审批" | 实体：ApprovalRule（amount_threshold, approver_role, department）；实体：ApprovalFlow（request_id, current_step, total_steps, status） |
| "审批人可批准、驳回或转交，审批历史需记录"              | 实体：ApprovalRecord（flow_id, approver_id, action, comment, created_at）；状态流转：待审批→审批中→已通过/已驳回                                   |
| "申请人可查看审批进度和历史意见"                   | 关系：ExpenseRequest 1:N ApprovalFlow，ApprovalFlow 1:N ApprovalRecord                                                          |

## 四、用AI辅助技术设计

### 4.1 为什么需要AI辅助

AI-native团队的特点是：让AI处理信息整合和初稿生成，人类专注于判断和决策。

**AI擅长的**：
- 从PRD中提取结构化信息
- 生成数据模型初稿
- 编写接口契约模板
- 检查设计完整性

**人类必须做的**：
- 业务理解判断
- 架构决策
- 风险评估
- 最终确认

### 4.2 提示词模板

#### 模板1：数据模型提取

```
角色：资深后端架构师
任务：从PRD中提取数据模型

输入：
[粘贴PRD相关章节]

请完成以下工作：
1. 识别所有名词实体
2. 为每个实体列出关键字段（含数据类型建议）
3. 识别实体间关系（1:1, 1:N, N:M）
4. 标记需要产品确认的模糊点

输出格式：
## 实体列表
### 实体名
- 字段名: 类型, 说明
- ...

## 实体关系
- A 1:N B（关系说明）

## 待确认问题
- [ ] 问题描述
```

#### 模板2：接口设计生成

```
角色：API设计专家
任务：基于PRD功能描述生成RESTful API设计

输入：
功能描述：[粘贴PRD功能点]
已有实体：[粘贴数据模型]

请完成以下工作：
1. 列出实现该功能需要的所有API端点
2. 为每个API定义：
   - 方法和路径
   - 请求参数（含必填/选填）
   - 响应结构（成功/失败）
   - 错误码设计
3. 标注需要与前端协商的接口

输出格式：
## API清单
| 序号 | 方法 | 路径 | 描述 |

## API详情
### 1. API名称
- 路径: METHOD /path
- 请求: {参数结构}
- 响应: {结构}
- 错误码: 401-未授权, 404-不存在...

## 待协商点
- [ ] 协商点描述
```

#### 模板3：状态机设计

```
角色：业务逻辑设计师
任务：从PRD业务流程中提取状态机

输入：
业务流程：[粘贴PRD中的流程描述]

请完成以下工作：
1. 识别所有可能的状态
2. 识别状态间的转移条件和触发事件
3. 绘制状态流转表
4. 标注需要确认的分支逻辑

输出格式：
## 状态列表
- 状态A: 说明
- 状态B: 说明

## 状态流转
| 当前状态 | 事件 | 条件 | 下一状态 |
|----------|------|------|----------|

## 待确认
- [ ] 问题
```

#### 模板4：技术方案生成（用于Codex）

```
角色：全栈开发专家
任务：基于技术设计文档生成可执行的实现方案

输入：
技术设计：
[粘贴技术设计文档]

技术栈：
- 前端: Vue 3 + TypeScript
- 后端: Java Spring Boot
- 数据库: PostgreSQL

请完成以下工作：
1. 生成项目目录结构建议
2. 为每个实体生成：
   - 数据库表结构（SQL）
   - 后端Entity/Repository代码
   - Service层接口设计
   - Controller层API实现
3. 为关键业务逻辑生成伪代码/流程图
4. 标注需要人工Review的关键决策点

输出要求：
- 代码需符合Spring Boot最佳实践
- 包含基本的错误处理和日志记录
- 标注Performance敏感点
```

### 4.3 Human-in-Loop规范

使用AI辅助设计时，必须遵循以下Human-in-Loop原则：

#### 原则1：AI生成，人类验证

```
❌ 错误：AI生成后直接采用
✅ 正确：AI生成 → 人类Review → 确认或修正 → 采用
```

#### 原则2：关键决策必须人工

以下决策**不能**交给AI：
- 架构选型（微服务 vs 单体）
- 技术栈变更
- 数据库选型
- 安全策略
- 性能指标设定

#### 原则3：标记不确定性

所有AI生成的内容必须标注置信度：

```markdown
## AI生成内容（置信度：高/中/低）

高：PRD中有明确对应
中：需要合理推断
低：需要产品确认

[内容]

---
人工Review意见：
- [ ] 确认无误
- [ ] 需要修改：[说明]
```

#### 原则4：迭代式 refinement

```
第一轮：AI生成初稿
第二轮：人工Review，提出修改意见
第三轮：AI根据意见修订
第四轮：人工最终确认
```

## 五、指导Codex完成开发

### 5.1 Codex最佳实践

根据OpenAI官方指南，Codex最有效的使用方式：

#### 实践1：上下文先行

向Codex提供完整的上下文：

```
任务：实现用户注册功能

上下文：
1. 技术栈：Vue 3 + TypeScript + Pinia（前端），Java Spring Boot + JPA（后端）
2. 数据库表结构：
   [粘贴SQL]
3. 已有代码：
   [粘贴相关Entity/Service代码]
4. 接口契约：
   POST /api/users/register
   请求：{username, email, password}
   响应：{userId, token} 或错误信息

要求：
- 实现完整的Controller + Service + Repository层
- 包含密码加密（BCrypt）
- 包含基础验证（邮箱格式、密码强度）
- 返回统一格式的API响应
```

#### 实践2：测试驱动

让Codex先生成测试，再生成实现：

```
第一步：生成测试用例
"为以下功能生成JUnit测试用例：..."

第二步：人工Review测试
确认测试覆盖了正常流程和异常流程

第三步：生成实现代码
"基于以上测试用例，实现功能代码"

第四步：运行测试验证
```

#### 实践3：小步快跑

不要给Codex过大的任务，推荐任务粒度：

| 粒度 | 示例 | 预估时间 |
|------|------|----------|
| 过大 | "实现整个订单模块" | ❌ 不推荐 |
| 合适 | "实现订单创建API" | 10-20分钟 |
| 过小 | "实现订单创建的参数校验" | 5分钟 |

### 5.3 代码Review清单

Codex生成的代码必须经过以下检查：

#### 功能完整性
- [ ] 是否实现了PRD要求的所有功能点
- [ ] 边界情况是否处理（空值、异常输入）
- [ ] 错误处理是否完善

#### 代码质量
- [ ] 是否符合团队代码规范
- [ ] 命名是否清晰语义化
- [ ] 是否有冗余代码

#### 性能考量
- [ ] 是否有明显的性能问题（N+1查询、重复计算）

#### 安全性
- [ ] 输入是否验证
- [ ] 敏感操作是否有权限检查
- [ ] 是否有SQL注入/XSS风险

## 六、跨团队协作规范

### 6.1 与产品经理的协作流程

```
产品产出PRD+原型
       ↓
研发速读PRD，标记疑问点
       ↓
产品-研发对齐会（30分钟）
- 确认用户场景理解
- 澄清疑问点
- 识别技术风险
       ↓
研发产出技术设计（用AI辅助）
       ↓
技术设计Review（Leader审核）
       ↓
产品确认技术方案
       ↓
进入开发（用Codex辅助）
       ↓
每日站会同步进度
       ↓
测试验收
```

### B.2 示例：审批流程PRD

以下以"B端审批流程"为例，填充上述模板：

---

#### 一、产品概述

**核心价值主张**: 让企业的报销、请假、采购等审批流程自动化、透明化，减少人工干预，提升审批效率，同时为管理者提供数据洞察。

**产品差异化**:
| 传统方式 | 本方案 |
|----------|--------|
| 纸质审批，流程不透明 | **全流程数字化，实时查看进度** |
| 审批人固定，无法灵活调整 | **智能派单，支持多级审批和转交** |
| 审批历史难追溯 | **完整审批记录，支持审计** |
| 无法分析审批效率 | **数据报表，识别流程瓶颈** |

#### 二、用户画像

**画像A: 普通员工 小王（28岁，销售）**
- **场景**: 经常需要报销差旅费用
- **痛点**:
  - 不知道审批进度，多次催促审批人
  - 被驳回后不清楚原因，反复修改
  - 历史报销记录难以查找

**画像B: 部门经理 李总（40岁，管理层）**
- **场景**: 需要审批团队成员的各类申请
- **痛点**:
  - 每天大量审批请求，难以区分轻重缓急
  - 外出时无法及时处理紧急审批
  - 不了解团队的整体费用情况

#### 三、功能设计

**核心循环**:
```
提交申请 → 系统自动派单 → 审批人处理 → 结果通知 → 进度跟踪
```

**模块1: 申请提交**
- **功能描述**: 员工填写申请表单，上传附件，提交审批
- **业务规则**:
  - 金额 < 1000: 部门经理审批
  - 1000 ≤ 金额 < 10000: 部门经理 + 财务审批
  - 金额 ≥ 10000: 部门经理 + 财务总监审批
- **异常处理**:
  - 必填项未填写 → 前端校验提示
  - 附件过大 → 提示压缩或分批上传

**模块2: 智能派单**
- **功能描述**: 根据预设规则自动选择审批人
- **业务规则**:
  - 优先分配给在岗审批人
  - 审批人外出时自动转交代理人
  - 超时未处理自动升级

**模块3: 审批处理**
- **功能描述**: 审批人查看申请详情，选择批准/驳回/转交
- **用户故事**:
  - 作为审批人，我希望一键批量审批同类申请，以便提高效率
  - 作为审批人，我希望在手机上处理紧急审批，以便及时响应

**状态流转**:
```
草稿 → 待审批 → 审批中 → 已通过/已驳回
          ↓
       已撤回
```

#### 四、数据与指标

**埋点事件**:
| 事件 | 属性 |
|------|------|
| 申请提交 | 金额、类别、耗时 |
| 审批操作 | 动作、耗时、是否移动端 |
| 流程完成 | 总耗时、审批人数量 |

**成功指标**:
- 平均审批时长 < 24小时
- 审批人满意度 > 4.0/5.0
- 流程自动化率 > 80%

### B.3 如何从示例PRD提取技术要素

基于上述审批流程PRD：

**数据实体**:
- ExpenseRequest: id, applicant_id, amount, category, reason, status, created_at
- ApprovalFlow: id, request_id, current_step, total_steps, status
- ApprovalRecord: id, flow_id, approver_id, action, comment, created_at
- ApprovalRule: id, amount_threshold, approver_role, priority

**接口设计**:
- POST /api/expense-requests: 创建申请
- GET /api/expense-requests/{id}: 查询申请详情
- GET /api/expense-requests: 查询申请列表（支持筛选）
- POST /api/approval-flows/{id}/approve: 批准
- POST /api/approval-flows/{id}/reject: 驳回

**状态机**:
```
草稿 → 提交 → 待审批 → 审批中 → 已通过
                    ↓
                 已驳回
```

**算法逻辑**:
- 自动派单算法：根据金额和审批规则匹配审批人
- 超时升级算法：检测超时流程并自动升级
