你好！作为产品经理，使用 **Vibe Coding**（通过自然语言驱动AI生成代码或串联现有系统）来落地这个预测性维护的Copilot场景，是一个非常前沿且高效的做法。

既然公司已有基础功能模块（算法包、告警、工单、库存等API），我们在Vibe Coding时的核心任务就是**“写好剧本（Prompts）”和“搭好舞台（前端交互与状态流转）”**。

以下我为你将该场景拆解为可直接用于产品需求文档（PRD）和 Vibe Coding 提示词的落地细化方案：

### 一、 数据的来源 (Data Sources Mapping)

在串联系统前，必须明确各个Agent消费的数据源头在哪里，以便在Vibe Coding时绑定正确的系统库：

| 数据类型 | 具体内容 | 数据来源系统 / 底层模块 | 数据更新频率 |
| :--- | :--- | :--- | :--- |
| **设备物联数据** | 5kHz高频振动加速度、电机电流、环境温度 | IoT物联网平台 / 边缘计算网关 | 实时 / 毫秒级 |
| **算法预测结果** | 特征值(RMS/峭度)、故障分类(置信度)、RUL(72h) | 预测性维护算法包（已内置清洗和推理逻辑） | 准实时 / 轮询 |
| **维修履历数据** | AHU-02历史工单记录、上次更换皮带时间 | EAM (设备资产管理系统) | 事件驱动 (完工时更新) |
| **非结构化知识** | 《精密空调维修手册》、SOP、安全操作规程 | 知识库系统 (需具备RAG检索能力) | 静态 / 定期维护 |
| **资源调度数据** | B-52皮带库存量、技师排班表、机房负载预测 | ERP(库存)、HR(排班)、动环监控系统 | 实时接口查询 |

### 二、 触发方式 (Trigger Mechanisms)

在Vibe Coding中，我们需要定义明确的事件监听（Event Listeners）来触发下一步动作：

1. **阶段一：告警生成（系统级触发）**
   * **触发逻辑**：后端算法定时任务（Cron）或流式计算（Flink），当规则引擎判定 `Confidence > 0.8 && RUL < 72h` 时触发。
   * **动作**：通过 WebSocket / Server-Sent Events (SSE) 向前端 Dashboard 推送 `Alarm_Event`。
2. **阶段二：诊断Agent唤醒（用户交互触发）**
   * **触发逻辑**：用户点击 Dashboard 上的闪烁告警卡片（`onClick`）。
   * **动作**：前端路由跳转至详情页，页面挂载完成（`useEffect/mounted`）时，自动携带 `device_id` 请求诊断 Agent。
3. **阶段三：调度Agent唤醒（用户交互触发）**
   * **触发逻辑**：用户在侧边栏点击【制定方案】按钮（`onClick`）。
   * **动作**：前端提取诊断 Agent 的输出结果（`fault_type: 皮带松弛`），作为入参传递给调度 Agent。
4. **阶段四：生成Agent唤醒（用户交互触发）**
   * **触发逻辑**：用户点击【草拟工单】按钮（`onClick`）。
   * **动作**：前端打包整个 Context（设备+诊断结论+备件+人员+时间），请求工单生成 Agent。
5. **阶段五：审批下发（用户交互触发）**
   * **触发逻辑**：用户点击工单预览弹窗的【批准并派发】（`onClick`）。
   * **动作**：调用核心业务系统写接口，完成闭环。

### 三、 每一环节需要调用的 API (API Contracts)

这里为你整理出标准化的 API 契约，**你可以直接将这部分提供给 Vibe Coding 工具（如 Cursor/Windsurf），让其自动生成前后端对接代码（Axios 请求或 Fetch API）：**

#### 1. 设备诊断阶段 (Agent 1)
*   **读取时序数据**: `GET /api/v1/device/{device_id}/telemetry`
    *   *Params*: `metrics=vibration,current`, `time_range=last_24h`
*   **获取历史工单**: `GET /api/v1/maintenance/history`
    *   *Params*: `device_id=AHU-02`, `limit=5`
*   **知识库RAG检索**: `POST /api/v1/knowledge/rag_search`
    *   *Payload*: `{"query": "AHU精密空调皮带打滑特征及张力校准标准", "top_k": 3}`

#### 2. 工单调度阶段 (Agent 2)
*   **查询备件库存**: `GET /api/v1/inventory/stock`
    *   *Params*: `part_model=B-52`, `warehouse_id=DC_WH_01`
*   **锁定备件**: `POST /api/v1/inventory/reserve` (重要：防并发超卖)
    *   *Payload*: `{"part_id": "P-B52-001", "qty": 2, "reserve_time": "24h"}`
*   **查询技师排班**: `GET /api/v1/staff/schedule`
    *   *Params*: `skill_required=HVAC`, `date=tomorrow`
*   **获取机房负载**: `GET /api/v1/datacenter/load_forecast`
    *   *Params*: `zone_id=Room_02`, `date=tomorrow`

#### 3. 工单生成阶段 (Agent 3)
*   **获取工单模板**: `GET /api/v1/workorder/templates`
    *   *Params*: `type=mechanical_repair`
*   **提取标准SOP**: `POST /api/v1/knowledge/extract_sop`
    *   *Payload*: `{"task": "更换精密空调风机皮带并校准张力"}`
*   **生成工单草稿**: `POST /api/v1/workorder/draft`
    *   *Payload*: `{完整的JSON结构，包含设备、描述、时间、执行人、备件、SOP等}`

#### 4. 审批派发阶段 (最后一步)
*   **发布工单**: `POST /api/v1/workorder/{draft_id}/publish`
    *   *Response*: `{"status": "success", "workorder_id": "WO-20231024-001"}`

### 四、 模块关联方式、Vibe Coding 提示词与最终呈现

不同模块之间通过 **上下文透传 (Context Passing)** 关联。在前端，这通常是一个统一的 JSON State ；在后端，是多个 Agent 之间的信息接力。

#### 1. Vibe Coding 提示词设计 (System Prompts for Agents)
为了让已有的大模型（如 GPT-4o 或你们的私有模型）扮演好这三个角色，你需要将以下 Prompt 注入到产品的后端 LLM 路由层中：

**Agent 1: 设备诊断 Analyst (System Prompt)**
> “你是一个资深的数据中心暖通设备诊断专家。你的任务是分析传入的传感器数据异常、历史维修记录和检索到的技术手册。
> 1. 根据传入的 FFT 频域数据（12Hz/25Hz异常）和时域数据（RMS升高），结合业务规则判断故障根因。
> 2. 查阅历史记录，判断该部件是否达到疲劳寿命。
> 3. 输出格式必须是：【结论】、【依据】、【下一步建议】。语言要专业、简练，直接面向运维经理汇报。”

**Agent 2: 工单调度 Coordinator (System Prompt)**
> “你是一个高效的工单调度员。你需要综合统筹‘物料’、‘人力’和‘环境’。
> 1. 接收上一个环节输出的【需更换备件类型】。
> 2. 分析传入的库存API结果、排班API结果和机房负载API结果。
> 3. 找出一个‘备件充足’、‘有资质技师空闲’且‘机房负载最低（适合停机）’的交集时间窗口。
> 4. 输出格式必须包含：【推荐时间】、【推荐人员】、【备件状态】。”

**Agent 3: 工单生成 Scribe (System Prompt)**
> “你是一个严谨的工单文书助手。你的任务是将非结构化的讨论转化为符合工单系统的标准 JSON 格式。
> 1. 整合之前诊断结论和调度安排。
> 2. 基于传入的 RAG 知识库内容，提炼出不多于 5 条的操作步骤 (SOP) 和 2 条以上的安全注意事项（如：Lockout/Tagout 断电挂牌）。
> 3. 严格按照预设的 API Schema 输出 JSON，不要有任何多余的寒暄。”

#### 2. 产品最终呈现方式 (UI/UX 落地建议)

在 Vibe Coding 生成前端界面时，建议采用 **“主副分屏 (Master-Detail)” + “渐进式披露 (Progressive Disclosure)”** 的设计模式：

*   **左侧 / 背景 (Master View)**：
    *   数据中心机房 3D/2D 拓扑图（Dashboard）。
    *   AHU-02 节点闪烁呼吸灯效果。
    *   点击后弹出右侧抽屉面板。
*   **右侧抽屉 (Copilot Panel - 核心交互区)**：
    *   采用 **对话流/Timeline** 的形式展示 Agent 的工作过程。
    *   *UI 展现*：
        *   [系统告警] 🔴 AHU-02 皮带打滑预警 (72h后失效风险)。
        *   *(加载动画)* AI正在分析根因...
        *   [Agent 1 卡片] 展示诊断结论。底部按钮：👉 `[让AI制定维修方案]`
        *   *(加载动画)* AI正在协调资源...
        *   [Agent 2 卡片] 展示王工+明早10点的推荐。底部按钮：👉 `[让AI草拟工单]`
        *   *(加载动画)* AI正在撰写SOP...
*   **居中弹窗 (Decision Modal)**：
    *   当点击生成草稿后，居中弹出一个类似纸质表单的**“工单预览卡片”**。
    *   高亮显示 AI 自动填写的 SOP 和安全提示（体现 AI 的价值）。
    *   底部常驻两个行动按钮：`[修改草稿] (次要)`，`[批准并派发] (主色/主要)`。

#### 3. 给 PM 的 Vibe Coding 落地实操建议
你可以把这份需求文档分段喂给你的 AI IDE（比如 Cursor），使用如下话术进行 Vibe Coding：
1. *"帮我用 React+Tailwind 写一个数据中心 Dashboard，中间放一个设备节点 AHU-02，处于警告状态。"*
2. *"点击设备后，右侧滑出一个面板。我需要你对接我提供的 `Agent 1` 的 API 契约，用流式输出展示诊断过程。"*
3. *"根据以上交互逻辑，帮我写一个统一的 State Store (Zustand/Redux)，确保 Agent 1 查出来的设备故障类型，能无缝传递给 Agent 2 的 payload 中作为入参。"*

通过这种方式，现有的 API 接口就能被灵活的 UI 和 LLM 的推理能力串联起来，形成一个体验极佳的“人机协同”预测性维护闭环新产品。
