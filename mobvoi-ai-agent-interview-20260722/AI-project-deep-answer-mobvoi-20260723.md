# 出门问问面试：AI 项目详细回答

> 日期：2026-07-23
> 用途：回答“你具体做过什么 AI 项目 / RAG 怎么做 / Agent 怎么拆 / 怎么评估效果 / 没上线怎么证明价值”。
> 口径边界：这是产品侧方案验证、Demo/POC、可交付能力规划，不包装成模型训练或大规模全量上线。

## 1. 一句话定位

我做的不是“给产品加一个 AI 聊天框”，而是把 AI Agent 放进真实业务链路里，围绕上下文、知识源、执行步骤、结构化输出、人工确认和结果回写，形成一个可解释、可验收、可迭代的任务闭环。

面向出门问问这个岗位，可以这样迁移：

| 我的 Inspector 项目 | 出门问问 AI Agent 产品 |
|---|---|
| 告警任务 | 用户创建任务 |
| 设备台账 / 空间 / SOP / 历史工单 | 文档 / 录音 / 项目知识 / 用户偏好 |
| 诊断建议 / 工单草稿 | 总结 / 内容草稿 / action items / 项目知识 |
| 人工确认后派单 | 用户确认后交付 / 分享 / 复用 |
| 工单结果回写知识库 | 任务结果沉淀到项目记忆 / Wiki / 模板 |

## 2. 90 秒推荐回答

我最适合讲 Inspector 设施运维 Agent。这个项目的背景是，客户现场并不是没有系统，而是告警、设备台账、空间位置、SOP、历史工单这些信息分散在不同地方。一线人员看到一个告警以后，仍然要自己判断是哪台设备、影响哪里、过去有没有类似问题、应该先查什么、要不要生成工单。

所以我没有把 AI 定义成一个独立聊天框，而是先拆“告警到工单”的业务闭环：告警触发、设备和空间识别、上下文补全、知识检索、诊断建议、处置方案、工单草稿、人工确认、结果回写。

我个人做得比较多的是产品侧拆解。第一，我把 AI 需要的上下文分层：实时告警和传感器数据说明现在发生了什么；设备台账和空间定位说明对象是谁、在哪里；SOP 和厂商手册是标准依据；历史工单只能作为相似经验参考。第二，我定义了 Agent 的输出结构，不允许只输出一句“可能是某部件异常”，而是要包含可能原因、判断依据、置信度、缺失信息、建议检查项和是否需要人工确认。第三，我和研发把这些拆成字段、接口、状态和验收脚本，比如设备 ID 能不能稳定拿到、SOP 有没有结构化、没有召回依据时是否必须提示依据不足、低置信时是否禁止给确定结论。

在验收上，我不看 AI 回答是否漂亮，而看一个典型告警能不能跑通最小闭环：识别正确设备和空间，召回正确 SOP 或相似工单，输出结构完整，低置信能兜底，高风险必须人工确认，工单草稿能被运维人员少量修改后使用。

所以这段经历的核心不是模型训练，而是我把一个很虚的“大模型诊断”拆成了研发能做、测试能验、客户能理解、现场人员敢用的产品闭环。这个方法迁移到出门问问，就是把海外用户的任务、上下文、Agent 执行、结果确认和协作交付拆清楚。

## 3. 3 分钟完整版

我可以讲一个比较具体的 AI 项目经历：Inspector 设施运维 Agent。

一开始客户不会说“我要一个 Agent”，他们说的是现场告警太多，新人不知道该先看哪里，资深工程师经常被重复问。我们看到的真实问题是，数据并不少，但分散在告警系统、设备台账、空间定位、SOP、历史工单里，没有形成一个处理问题的工作流。

比如一个暖通或设备异常告警，系统可能只提示“温度异常”或者“设备离线”。但现场人员真正要判断的是：是哪台设备、在哪个空间、影响哪些区域、过去有没有类似故障、应该先查哪个部件、要不要生成工单。如果 AI 只返回一句“可能是阀门异常，建议检查”，这个答案是不能真正进入现场流程的。

所以我先做的不是定义 Agent，而是定义业务闭环：

```text
告警触发
-> 设备与空间识别
-> 上下文补全
-> SOP / 历史工单检索
-> 诊断建议
-> 处置方案
-> 工单草稿
-> 人工确认
-> 执行结果回写
-> 知识沉淀
```

拆完以后，我再反过来拆 Agent 职责。告警理解 Agent 负责把原始告警转成结构化问题；RAG / 知识检索模块负责找 SOP、历史工单、设备资料；诊断 Agent 负责输出可能原因、依据和置信度；方案生成 Agent 负责给建议检查项；工单生成 Agent 只生成草稿，不自动派发。

这里我特别强调上下文和知识源优先级。实时告警、时序数据和设备状态用于判断“现在发生了什么”；设备台账和空间关系用于判断“对象是谁、在哪儿、和谁关联”；SOP 和厂商手册是标准依据；历史维修记录只是经验参考。也就是说，如果 SOP 和历史工单冲突，AI 不能直接按历史经验下结论，而要提示冲突，让工程师确认。

RAG 评估我也拆成两层。第一层是召回正确：Top3 或 Top5 里有没有召回正确设备、正确告警类型、正确 SOP 或相似工单。第二层是生成正确：最终回答有没有基于召回材料，有没有乱引用，有没有把历史经验说成标准流程。为了让它能验收，我会整理典型场景测试集，每个问题标注应该命中的资料，再和研发一起看问题属于知识缺失、切片问题、标签问题、检索问题，还是生成表达问题。

我的产品产出不是一句“优化 RAG”，而是一组产品协议：业务流程、业务对象、Agent 职责、输入输出字段、异常分支、人机边界、输出 Schema 和验收脚本。比如诊断输出必须包含可能原因、依据、置信度、缺失信息、建议动作和风险提示；低置信不能给确定结论；消防、电梯、强电等高风险场景必须人工确认或升级专家。

最后结果上我会谨慎表达：这套能力属于产品侧方案验证、Demo/POC 和可交付规划，不是我训练了模型，也不是 AI 已经完全替代人工。但它证明了我能把 AI 从概念变成研发可评审、客户可理解、POC 可验证的产品闭环。对出门问问岗位来说，这个能力可以迁移到海外 AI Agent 工具：用户创建任务后，Agent 需要知道用什么上下文、执行哪几步、结果怎么确认、怎么协作交付、怎么沉淀到项目知识里。

## 4. 面试官追问防守包

### Q1：你个人到底做了什么？

我个人最核心的是产品侧拆解，不是模型训练。具体包括：

- 拆业务流程：告警触发、上下文补全、知识检索、诊断建议、工单草稿、人工确认、结果回写。
- 拆业务对象：设备、空间、告警、SOP、历史工单、知识源、工单字段、状态流。
- 拆 Agent 职责：告警理解、知识检索、诊断、方案生成、工单草稿。
- 定义输出结构：可能原因、依据、置信度、缺失信息、建议动作、风险提示。
- 定义验收脚本：典型告警能否识别对象、召回依据、生成可用草稿、低置信兜底、高风险人审。

### Q2：RAG 怎么判断准确？

我不会只看相似度，也不会只看最后答案顺不顺。我会拆成两层：

| 层级 | 判断标准 |
|---|---|
| 召回正确 | 是否命中正确设备、空间、告警类型、SOP、相似历史工单 |
| 生成正确 | 是否基于召回资料回答，引用是否正确，是否避免过度确定 |

如果召回了相似但不适用的资料，即使回答很流畅，也算不合格。比如空调告警不能因为关键词相似就召回电梯 SOP；历史工单也不能直接被当成标准处理规则。

### Q3：信息源优先级怎么定？

我会按“事实数据 -> 标准依据 -> 经验参考”的顺序：

1. **实时告警 / 传感器 / 设备状态**：判断当前发生了什么。
2. **设备台账 / 空间位置 / 系统关系**：判断对象是谁、在哪里、上下游是谁。
3. **SOP / 厂商手册 / 标准规则**：作为正式处理依据。
4. **历史工单 / 专家经验**：作为相似案例和经验参考。
5. **用户补充信息**：处理现场缺失数据，比如阀门开度、压力、现场照片。

如果来源冲突，产品上要提示冲突和不确定性，而不是让 AI 强行给结论。

### Q4：怎么和研发协作？

我不会跟研发说“做一个智能诊断”，而是把它拆成可实现的协议：

- 输入字段：告警 ID、设备 ID、空间 ID、告警等级、历史曲线、SOP 检索结果、历史工单。
- 输出字段：原因 TopN、依据、置信度、缺失信息、建议动作、风险提示、工单草稿。
- 异常路径：设备不存在、SOP 缺失、多个同名设备、低置信、无依据、告警已恢复。
- 验收用例：典型告警、低置信场景、高风险场景、知识缺失场景、历史工单误召回场景。

这样研发讨论的是字段、接口、状态和测试样例，而不是抽象的“AI 准不准”。

### Q5：如果没有完整线上指标，怎么证明 AI 有价值？

我不会硬讲准确率，而是讲阶段性验收：

- **Demo-ready**：客户能看懂告警到诊断再到工单草稿的故事。
- **POC-ready**：典型场景能用客户数据跑通，输出有依据、置信度、缺失信息和人工确认。
- **Delivery-ready 的前置条件**：真实数据接入、权限、审计、异常兜底、验收脚本和运维反馈回流。

指标上，早期可以看建议打开率、工单草稿采纳率、人工修改率、bad case 类型、客户是否愿意进入试点。等上线稳定后再看平均处置时长、工单一次解决率、重复告警识别率等。

### Q6：这个项目最大的困难是什么？

最大的困难不是“模型不会回答”，而是业务信息分散和信任边界难定义。

信息分散的问题，我用业务对象和上下文清单解决：先明确设备、空间、告警、SOP、历史工单分别从哪里来，哪些是 MVP 必需，哪些可以后补。

信任边界的问题，我用输出结构和人审机制解决：AI 只做辅助诊断和草稿生成，不自动派发高风险工单；输出必须有依据、置信度、缺失信息；低置信时只给检查方向，不给确定结论。

## 5. 如何迁移到出门问问岗位

如果面试官问“这个和我们海外 Agent 产品有什么关系”，可以这样收口：

这段经历的行业是设施运维，但方法可以迁移。出门问问的海外 AI Agent 产品，本质上也不是让 AI 说一段漂亮答案，而是让用户完成一个任务。用户可能上传录音、文档、网页或项目资料，希望 Agent 帮他总结、生成内容、提取 action items、沉淀项目知识，最后由用户确认和复用。

所以我会用同一套方法来做：

```text
用户任务
-> 上下文管理
-> Agent 计划与执行
-> 结构化结果
-> 用户确认
-> 协作交付
-> 项目知识沉淀
```

这正好对应 JD 里的任务创建、上下文管理、Agent 执行、结果确认、协作交付、用户反馈和数据分析。

## 6. 可直接背的 30 秒版本

我做过的 AI 项目不是模型训练，而是 AI Agent 产品化。以 Inspector 为例，客户痛点是告警、设备、SOP、历史工单分散，一线人员看到告警后不知道原因和处理路径。我把它拆成告警理解、上下文补全、知识检索、诊断建议、工单草稿、人工确认和结果回写，并定义输入字段、输出 Schema、低置信兜底和验收脚本。这个经历最能证明我能把 Agent 从概念拆成真实任务闭环。迁移到出门问问，就是把海外用户的任务、文档/录音/项目知识、Agent 执行、结果确认和协作交付拆清楚。
